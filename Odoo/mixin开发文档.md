# External JSON Sync Mixin 开发文档

## 1. 功能概述

`external.json.sync.mixin` 是一个用于 **Odoo 18 外部 JSON 数据同步** 的抽象模型。

它主要解决两个通用问题：

1. **外部 JSON 安全写入**
   - 只写入当前模型允许的字段；
   - 自动忽略未知字段；
   - 自动忽略存在但不允许外部写入的字段；
   - 将异常字段记录到日志模型，便于后续迭代。

2. **Datetime 自动转 UTC**
   - 外部接口返回的时间可能包含时区，也可能不包含时区；
   - mixin 会将 `fields.Datetime` 字段统一转换成 Odoo 推荐的 UTC 时间字符串；
   - `fields.Date` 字段不会做 UTC 偏移，只做日期格式归一化。

该 mixin 适合所有“外部系统数据映射模型”，例如：

- 外部订单；
- 外部客户；
- 外部商品；
- 外部库存；
- 外部支付记录；
- 第三方平台 webhook 数据。

---

## 2. 核心模型

模块中包含两个核心模型。

### 2.1 `external.json.sync.mixin`

抽象模型，不创建业务数据表。

业务模型通过继承它来获得：

- `external_json_create()`
- `external_json_write()`
- 安全字段过滤；
- JSON key 到 Odoo 字段的映射；
- Datetime UTC 转换；
- 异常字段日志记录；
- 自定义字段转换器。

### 2.2 `external.json.issue.log`

日志模型，用于记录外部 JSON 中无法安全写入的内容。

常见日志类型：

| 类型 | 含义 |
|---|---|
| `unknown_keys` | JSON 中存在，但 Odoo 模型中不存在的字段 |
| `ignored_keys` | Odoo 模型中存在，但不允许外部写入的字段 |
| `invalid_keys` | 字段存在且允许写入，但值转换失败的字段 |

---

## 3. 推荐使用方式

业务模型不要直接让外部 JSON 调用原生 `write()`。

推荐统一使用：

```python
record.external_json_write(payload, source_tz="Asia/Shanghai")
```

或：

```python
self.env["x.external.order"].external_json_create(
    payload,
    source_tz="Asia/Shanghai",
)
```

也可以手动通过 context 开启：

```python
record.with_context(
    external_json_safe_write=True,
    external_json_source_tz="Asia/Shanghai",
).write(payload)
```

但为了代码可读性和统一规范，推荐优先使用 `external_json_write()` 和 `external_json_create()`。

## 4. 业务模型示例

```python
# -*- coding: utf-8 -*-

from odoo import fields, models


class ExternalOrder(models.Model):
    _name = "x.external.order"
    _description = "External Order"
    _inherit = ["external.json.sync.mixin"]

    _external_json_allowed_fields = {
        "external_id",
        "name",
        "amount_total",
        "state",
        "external_created_at",
        "external_updated_at",
    }

    _external_json_field_map = {
        "id": "external_id",
        "createdAt": "external_created_at",
        "updatedAt": "external_updated_at",
    }

    _external_json_source_tz = "Asia/Shanghai"

    external_id = fields.Char(index=True)
    name = fields.Char()
    amount_total = fields.Float()

    state = fields.Selection(
        [
            ("draft", "Draft"),
            ("paid", "Paid"),
            ("cancel", "Cancelled"),
        ],
        default="draft",
    )

    external_created_at = fields.Datetime()
    external_updated_at = fields.Datetime()
```

同步代码示例：

```python
payload = {
    "id": "EXT-10001",
    "name": "测试订单",
    "amount_total": 128.5,
    "state": "paid",
    "createdAt": "2026-06-05 12:30:00",
    "updatedAt": "2026-06-05T15:00:00+08:00",
    "api_new_field": "暂未支持的新字段",
}

Order = self.env["x.external.order"]

record = Order.search(
    [
        ("external_id", "=", payload.get("id")),
    ],
    limit=1,
)

if record:
    record.external_json_write(
        payload,
        source_tz="Asia/Shanghai",
        external_json_source="order_api",
        external_json_external_ref=payload.get("id"),
    )
else:
    Order.external_json_create(
        payload,
        source_tz="Asia/Shanghai",
        external_json_source="order_api",
        external_json_external_ref=payload.get("id"),
    )
```

## 5. 字段写入规则

mixin 会把外部 JSON 分成三类。

### 5.1 可写字段

满足以下条件时，字段会进入最终的 `super().create()` 或 `super().write()`：

1. JSON key 能映射到 Odoo 字段；
2. 字段在 `_external_json_allowed_fields` 白名单中；
3. 字段没有被 `_external_json_denied_fields` 禁止；
4. 字段不是不可写的无 inverse 计算字段；
5. 字段值转换成功。

示例：

```python
_external_json_allowed_fields = {
    "external_id",
    "name",
    "external_created_at",
}
```

### 5.2 未知字段

JSON 中存在，但目标 Odoo 模型中不存在。

示例：

```json
{
  "new_api_field": "value"
}
```

如果模型中没有 `new_api_field`，该字段不会写入，会记录到：

```text
external.json.issue.log.unknown_keys
```

### 5.3 被忽略字段

字段在 Odoo 模型中存在，但不允许外部写入。

常见情况：

- 字段不在 `_external_json_allowed_fields`；
- 字段在 `_external_json_denied_fields`；
- 字段是系统字段；
- 字段是无 inverse 的计算字段。

例如：

```json
{
  "create_date": "2026-06-05 00:00:00"
}
```

`create_date` 属于系统字段，不允许外部 JSON 写入，会记录到：

```text
external.json.issue.log.ignored_keys
```

## 6. 配置项说明

### 6.1 `_external_json_allowed_fields`

外部 JSON 允许写入的字段白名单。

强烈建议每个外部映射模型都显式配置。

```python
_external_json_allowed_fields = {
    "external_id",
    "name",
    "external_created_at",
}
```

如果设置为 `None`，mixin 会进入自动模式，仅允许部分简单字段类型。

不推荐在生产系统中过度依赖自动模式。

---

### 6.2 `_external_json_field_map`

用于将外部 JSON key 映射成 Odoo 字段名。

```python
_external_json_field_map = {
    "id": "external_id",
    "createdAt": "external_created_at",
    "updatedAt": "external_updated_at",
}
```

例如外部 JSON：

```json
{
  "id": "EXT-001",
  "createdAt": "2026-06-05T12:00:00+08:00"
}
```

会转换为：

```python
{
    "external_id": "EXT-001",
    "external_created_at": "2026-06-05 04:00:00",
}
```

---

### 6.3 `_external_json_denied_fields`

业务模型可以额外声明禁止外部写入的字段。

```python
_external_json_denied_fields = {
    "company_id",
    "user_id",
    "active",
}
```

mixin 默认已经禁止以下系统字段：

```python
{
    "id",
    "display_name",
    "__last_update",
    "create_uid",
    "create_date",
    "write_uid",
    "write_date",
}
```

---

### 6.4 `_external_json_source_tz`

外部 API 传入的无时区 Datetime 默认属于哪个时区。

```python
_external_json_source_tz = "Asia/Shanghai"
```

例如：

```json
{
  "createdAt": "2026-06-05 12:30:00"
}
```

如果来源时区是 `Asia/Shanghai`，则会写入 UTC：

```text
2026-06-05 04:30:00
```

调用时也可以覆盖：

```python
record.external_json_write(
    payload,
    source_tz="America/New_York",
)
```

---

### 6.5 `_external_json_raise_on_invalid`

控制字段转换失败时的行为。

默认：

```python
_external_json_raise_on_invalid = False
```

宽松模式下：

- 转换失败字段会被跳过；
- 其他字段继续写入；
- 转换失败信息写入日志。

严格模式下：

```python
_external_json_raise_on_invalid = True
```

或调用时：

```python
record.external_json_write(
    payload,
    strict_datetime=True,
)
```

转换失败会直接抛出异常，整个写入中止。

---

### 6.6 `_external_json_epoch_unit`

是否允许数字时间戳。

默认：

```python
_external_json_epoch_unit = None
```

默认不接受数字时间戳。

如果外部接口返回秒级时间戳：

```python
_external_json_epoch_unit = "seconds"
```

如果外部接口返回毫秒级时间戳：

```python
_external_json_epoch_unit = "milliseconds"
```

也可以调用时传入：

```python
record.external_json_write(
    payload,
    epoch_unit="milliseconds",
)
```

---

### 6.7 `_external_json_log_raw_payload`

是否记录原始 payload。

默认：

```python
_external_json_log_raw_payload = True
```

如果 payload 中可能包含大量敏感信息，可以关闭：

```python
_external_json_log_raw_payload = False
```

或者调用时关闭：

```python
record.external_json_write(
    payload,
    external_json_log_raw_payload=False,
)
```

## 7. Datetime 转换规则

### 7.1 `fields.Datetime`

`fields.Datetime` 会被统一转换为 UTC 字符串。

支持以下格式：

```text
2026-06-05 12:30:00
2026-06-05T12:30:00+08:00
2026-06-05T04:30:00Z
```

转换规则：

| 输入类型 | 处理方式 |
|---|---|
| 带时区字符串 | 按字符串自带时区转 UTC |
| `Z` 结尾字符串 | 识别为 UTC |
| 不带时区字符串 | 按 `external_json_source_tz` 解释后转 UTC |
| Python `datetime` 且带时区 | 按对象时区转 UTC |
| Python `datetime` 且无时区 | 按 `external_json_source_tz` 解释后转 UTC |
| 数字时间戳 | 需要显式启用 `epoch_unit` |

示例：

```python
record.external_json_write(
    {
        "createdAt": "2026-06-05 12:30:00",
        "updatedAt": "2026-06-05T15:00:00+08:00",
    },
    source_tz="Asia/Shanghai",
)
```

最终写入：

```python
{
    "external_created_at": "2026-06-05 04:30:00",
    "external_updated_at": "2026-06-05 07:00:00",
}
```

---

### 7.2 `fields.Date`

`fields.Date` 没有时区语义，不会做 UTC 偏移。

例如：

```json
{
  "external_date": "2026-01-01T00:30:00+14:00"
}
```

写入结果仍然是：

```text
2026-01-01
```

不会因为转 UTC 而变成：

```text
2025-12-31
```

---

### 7.3 夏令时歧义处理

对于无时区时间，如果来源时区存在夏令时，可能出现：

- 不存在的本地时间；
- 歧义的本地时间。

默认策略是抛出转换错误，并根据严格/宽松模式决定是否中断写入。

可以通过 context 指定策略：

```python
record.external_json_write(
    payload,
    external_json_dst_strategy="standard",
)
```

或：

```python
record.external_json_write(
    payload,
    external_json_dst_strategy="dst",
)
```

可选值：

| 值 | 含义 |
|---|---|
| `raise` | 默认，遇到歧义时间抛错 |
| `standard` | 按标准时间处理 |
| `dst` | 按夏令时处理 |

## 8. 自定义字段转换器

如果某些字段不能直接写入，可以在业务模型中定义转换器。

命名规则：

```python
_external_json_convert_字段名
```

例如外部 JSON 传入客户外部编号，你需要转换成 Odoo 的 `partner_id`。

```python
class ExternalOrder(models.Model):
    _name = "x.external.order"
    _description = "External Order"
    _inherit = ["external.json.sync.mixin"]

    _external_json_allowed_fields = {
        "external_id",
        "name",
        "partner_id",
    }

    _external_json_field_map = {
        "customerCode": "partner_id",
    }

    external_id = fields.Char(index=True)
    name = fields.Char()
    partner_id = fields.Many2one("res.partner")

    def _external_json_convert_partner_id(self, value):
        if not value:
            return False

        partner = self.env["res.partner"].search(
            [
                ("x_external_customer_code", "=", value),
            ],
            limit=1,
        )

        return partner.id or False
```

外部 payload：

```json
{
  "customerCode": "CUST-001"
}
```

最终会写入：

```python
{
    "partner_id": 123
}
```

## 9. 日志说明

当 mixin 发现不能安全写入的字段时，会创建一条：

```text

```

常用字段：

| 字段 | 说明 |
|---|---|
| `model_name` | 目标业务模型 |
| `res_ids` | 目标记录 ID |
| `operation` | `create` 或 `write` |
| `source` | 外部来源，例如 `order_api` |
| `external_ref` | 外部系统主键 |
| `unknown_keys` | 未知字段名 |
| `ignored_keys` | 被忽略字段名 |
| `invalid_keys` | 转换失败字段名 |
| `unknown_payload` | 未知字段内容 |
| `ignored_payload` | 被忽略字段内容 |
| `invalid_payload` | 转换失败详情 |
| `raw_payload` | 原始 payload，默认会做敏感字段脱敏 |

调用时建议传入来源信息：

```python
record.external_json_write(
    payload,
    source_tz="Asia/Shanghai",
    external_json_source="order_api",
    external_json_external_ref=payload.get("id"),
)
```

这样日志中可以追踪到具体外部系统和外部记录。

_external_json_mask_keywords = {

## 11. create 和 write 的行为差异

### 11.1 `external_json_write()`

如果 payload 中没有任何可写字段：

```python
record.external_json_write(
    {
        "unknown_field": "value",
    }
)
```

默认返回：

```python
True
```

不会修改记录，但会写日志。

---

### 11.2 `external_json_create()`

如果 payload 中没有任何可写字段：

```python
Model.external_json_create(
    {
        "unknown_field": "value",
    }
)
```

默认会抛出 `ValidationError`。

原因是创建记录时，如果完全没有可写值，通常说明同步数据不完整或配置有误。

如果确实允许空创建，可以传：

```python
Model.external_json_create(
    payload,
    external_json_allow_empty_create=True,
)
```

1. 新增 Odoo 字段；
2. 添加到 `_external_json_allowed_fields`；
3. 添加 `_external_json_field_map`；
4. 如果需要，添加自定义转换器。

---

## 14. 注意事项

### 14.1 不要让外部 JSON 直接写系统字段

禁止让外部 JSON 写入：

```python
id
create_uid
create_date
write_uid
write_date
display_name
```

这些字段由 Odoo 自己管理。

---

### 14.2 Many2one 不建议直接接收外部 ID

外部系统的 ID 通常不是 Odoo 数据库 ID。

不推荐：

```json
{
  "partner_id": "CUST-001"
}
```

直接写入 `partner_id`。

推荐使用自定义转换器：

```python
def _external_json_convert_partner_id(self, value):
    partner = self.env["res.partner"].search(
        [
            ("x_external_customer_code", "=", value),
        ],
        limit=1,
    )
    return partner.id or False
```

---

### 14.3 One2many / Many2many 需要谨慎处理

Odoo 的 x2many 字段通常需要 `Command` 格式。

不建议直接让外部 JSON 写入 x2many 字段。

推荐在转换器中显式构造：

```python
from odoo import Command


def _external_json_convert_line_ids(self, value):
    commands = []

    for item in value or []:
        commands.append(
            Command.create(
                {
                    "name": item.get("name"),
                    "quantity": item.get("qty"),
                }
            )
        )

    return commands
```

---

### 14.4 Selection 值需要自行保证合法

如果外部 JSON 传入：

```json
{
  "state": "unknown_state"
}
```

而 `state` 字段的 selection 中没有这个值，最终 `super().write()` 可能会报错。

如果外部状态值不稳定，建议使用转换器：

```python
def _external_json_convert_state(self, value):
    mapping = {
        "new": "draft",
        "paid": "paid",
        "closed": "cancel",
    }

    return mapping.get(value, "draft")
```

---

### 14.5 多记录 write 的行为

如果对多条记录调用：

```python
records.external_json_write(payload)
```

则同一份清洗后的 `payload` 会写入所有记录。

日志中的 `res_ids` 会记录多个 ID。

如果每条记录对应不同 payload，应逐条处理：

```python
for payload in payload_list:
    record = Model.search(
        [
            ("external_id", "=", payload.get("id")),
        ],
        limit=1,
    )

    if record:
        record.external_json_write(payload)
    else:
        Model.external_json_create(payload)
```

## 15. 测试建议

建议至少覆盖以下测试场景：

1. 普通 `write()` 不开启 context 时，未知字段仍然报错；
2. `external_json_write()` 可以过滤未知字段；
3. 存在但不在白名单中的字段会被忽略；
4. `id` 可以通过 `_external_json_field_map` 映射成 `external_id`；
5. 无时区 Datetime 按 `source_tz` 转 UTC；
6. 带时区 Datetime 按自身时区转 UTC；
7. `fields.Date` 不做 UTC 偏移；
8. 非法 Datetime 在宽松模式下写日志并跳过；
9. 非法 Datetime 在严格模式下抛异常；
10. 自定义转换器可以处理 `Many2one`；
11. 敏感字段在日志中被脱敏；
12. `external_json_create()` 无可写字段时抛错。

运行示例：

```bash
./odoo-bin \
  -d odoo18_test \
  -u external_json_sync \
  --test-tags /external_json_sync \
  --stop-after-init
```

## 16. 设计原则

这个 mixin 的核心原则是：

> 外部 JSON 永远不是可信输入。  
> 进入 Odoo ORM 之前，必须先经过过滤、映射、转换和日志记录。

具体原则：

1. **默认安全**
   - 不直接信任外部 JSON；
   - 不允许未知字段进入 `super().write()`；
   - 不允许系统字段被外部覆盖。

2. **显式配置**
   - 推荐每个模型显式声明 `_external_json_allowed_fields`；
   - 外部 key 与 Odoo 字段不一致时，显式配置 `_external_json_field_map`。

3. **可观测**
   - 未知字段、忽略字段、转换失败字段都写日志；
   - 通过日志驱动后续模型迭代。

4. **不影响 Odoo 原生逻辑**
   - 只有开启 `external_json_safe_write` context 或调用 `external_json_write()` / `external_json_create()` 时才启用；
   - 普通后台界面和内部业务代码调用 `write()` 不受影响。

5. **时间统一入库**
   - 所有 `fields.Datetime` 统一转换为 UTC；
   - `fields.Date` 保持日期语义，不做时区偏移。

## 17. 最佳实践

建议所有外部映射模型统一采用如下模板：

```python
class SomeExternalModel(models.Model):
    _name = "x.some.external.model"
    _description = "Some External Model"
    _inherit = ["external.json.sync.mixin"]

    _external_json_allowed_fields = {
        "external_id",
        "name",
        "external_created_at",
        "external_updated_at",
    }

    _external_json_field_map = {
        "id": "external_id",
        "createdAt": "external_created_at",
        "updatedAt": "external_updated_at",
    }

    _external_json_source_tz = "Asia/Shanghai"

    external_id = fields.Char(index=True)
    name = fields.Char()
    external_created_at = fields.Datetime()
    external_updated_at = fields.Datetime()
```

同步代码统一采用：

```python
record.external_json_write(
    payload,
    source_tz="Asia/Shanghai",
    external_json_source="some_api",
    external_json_external_ref=payload.get("id"),
)
```

或：

```python
Model.external_json_create(
    payload,
    source_tz="Asia/Shanghai",
    external_json_source="some_api",
    external_json_external_ref=payload.get("id"),
)
```

## 18. 排查问题清单

### 问题：字段没有写入

检查：

1. 字段是否在 `_external_json_allowed_fields` 中；
2. 外部 key 是否需要配置 `_external_json_field_map`；
3. 字段是否在 `_external_json_denied_fields` 中；
4. 字段是否是无 inverse 的计算字段；
5. 是否有日志记录在 `external.json.issue.log`。

---

### 问题：时间少了 8 小时

这是正常情况。

如果外部时间是：

```text
2026-06-05 12:30:00
```

来源时区是：

```text
Asia/Shanghai
```

写入数据库的 UTC 时间应为：

```text
2026-06-05 04:30:00
```

Odoo 前端会根据用户时区显示成本地时间。

---

### 问题：非法 Datetime 没有报错

默认是宽松模式。

如果希望非法时间直接报错，使用：

```python
record.external_json_write(
    payload,
    strict_datetime=True,
)
```

或在模型中配置：

```python
_external_json_raise_on_invalid = True
```

---

### 问题：日志里看不到原始 payload

检查是否关闭了：

```python
_external_json_log_raw_payload = False
```

或调用时是否传了：

```python
external_json_log_raw_payload=False
```

---

### 问题：敏感字段在日志中被替换为 `***`

这是预期行为。

默认会对包含以下关键词的字段进行脱敏：

```text
password
token
secret
authorization
api_key
refresh_token
```

---

## 19. 总结

`external.json.sync.mixin` 的定位是：

> 为所有外部 JSON 映射模型提供统一、安全、可观测、可扩展的数据写入入口。

使用它之后，业务模型只需要关注：

1. 哪些字段允许外部写入；
2. 外部 key 如何映射到 Odoo 字段；
3. 特殊字段如何转换；
4. 外部时间属于哪个时区。

同步任务则统一调用：

```python
external_json_create()
```

和：

```python
external_json_write()
```

这样可以避免外部 API 字段变化导致 Odoo 写入报错，同时保留日志，方便后续持续迭代。
