## 1. 这个东西是什么？

一个**特殊构造的 dex 文件**，专门用来触发 Android 12 上 dex2oat 的线程死锁。

把它喂给 dex2oat 去编译，它在 8 个线程同时工作时，会因为类之间的交叉引用形成锁竞争风暴，最终编译卡死、看门狗超时、进程自杀。

你可以把它理解为：**给 dex2oat 吃一个它吞不下的东西，让它挂掉，然后你就能抓现场信息来排查问题了。**

## 4. 第 1 步：生成炸弹代码

打开命令行，进入 `D:\Android12` 目录，运行炸弹生成脚本。

```bash
cd D:\Android12
python gen_deadlock_bomb.py
```

**预期输出**：

```
生成了 168 个 smali 文件（160 个类 + 8 个接口）
  每类 30 个交叉引用方法 + 构造 + <clinit>
  总方法: ~5144
  总字段: ~480
  交叉引用对: 80
  命名策略: interleaved（A_n 与 B_n 紧邻 → 线程池并行取到时碰撞）

编译命令（需要 smali.jar 或 d8）：
  java -jar smali.jar assemble smali/ -o deadlock_bomb.dex
```

同时，`D:\Android12\smali\` 文件夹下会生成 168 个 `.smali` 文件。

> **你现在不必理解 smali 是什么。**把它想象成"Java 字节码的文本版"，后面一步会用一个工具把它编译成 dex2oat 能识别的格式。

## 5. 第 2 步：获取 smali.jar 并编译为 dex

这一步把 smali 文本文件编译成 `.dex` 文件——dex2oat 的"食物"。

### 5.1 找 smali.jar

`smali.jar` 是一个把 smali 文本编译成 dex 的工具。你有三种获取方式：

**方式 A（推荐—如果你有 AOSP 源码环境）**：它已经在 AOSP 里

```bash
# 在命令行执行，看看有没有 AOSP 环境变量
echo %ANDROID_BUILD_TOP%

# 如果有输出，smali.jar 大概率在这里：
dir %ANDROID_BUILD_TOP%\prebuilts\build-tools\common\framework\smali.jar
```

**方式 B：自动下载**

```bash
# 在 D:\Android12 目录执行
powershell -Command "Invoke-WebRequest -Uri 'https://github.com/google/smali/releases/download/v3.0.5/smali-3.0.5.jar' -OutFile 'smali.jar'"
```

> 如果下载失败（公司网络限制），用手机开热点或用方式 C。

**方式 C：请同事传给你**

让同事把一个能用的 `smali.jar` 放到 `D:\Android12\smali.jar`。

### 5.2 编译

找到 smali.jar 后，执行编译命令：

```bash
# 把 <你的smali.jar路径> 替换成实际路径，例如：
#   smali.jar                          （如果就在当前目录）
#   %ANDROID_BUILD_TOP%\prebuilts\build-tools\common\framework\smali.jar
#   D:\Android12\smali-3.0.5.jar

java -jar <你的smali.jar路径> assemble smali/ -o deadlock_bomb.dex
```

**预期输出**：没有输出就是成功。命令执行完会生成 `D:\Android12\deadlock_bomb.dex`。

### 5.3 确认 dex 文件生成成功

```bash
dir D:\Android12\deadlock_bomb.dex
```

应该显示一个文件，大小约 **300~600 KB**。

## 6. 第 3 步：推送到设备并引爆

### 6.1 推送 dex 到设备

```bash
adb push D:\Android12\deadlock_bomb.dex /data/local/tmp/
```

**预期输出**：

```
deadlock_bomb.dex: 1 file pushed.  xxx KB/s (xxxxxx bytes in x.xxx秒)
```

### 6.2 开两个命令行窗口

这一步很重要——你需要**两个窗口同时操作**：

| 窗口 | 用途 |
|------|------|
| **窗口 A**（主窗口） | 执行 dex2oat 编译命令 |
| **窗口 B**（观察窗口） | 观察 CPU 状态 + 抓栈 |

### 6.3 窗口 A：引爆

在窗口 A 执行以下命令：

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/deadlock_bomb.dex \
    --oat-file=/data/local/tmp/deadlock_bomb.oat \
    --compiler-filter=speed \
    -j8 \
    --watchdog-timeout=600000
```

**这条命令的含义**：

| 参数 | 含义 |
|------|------|
| `--dex-file=...` | 告诉 dex2oat 要编译哪个 dex 文件 |
| `--oat-file=...` | 编译产物写到哪里 |
| `--compiler-filter=speed` | 编译级别：全量编译（**必须用 speed 或 everything，用 verify 不会触发死锁**） |
| `-j8` | 用 8 个线程并行编译（线程越多越容易死锁） |
| `--watchdog-timeout=600000` | 看门狗超时设为 10 分钟（600000 毫秒） |

执行后，光标停在原地不动，没有任何输出——这**就是正常的**，说明 dex2oat 正在编译。

### 6.4 窗口 B：观察

**在按下回车的同时**，快速切到窗口 B，执行：

```bash
# 找到 dex2oat 的进程 ID
adb shell pidof dex2oat
```

如果输出一个数字（如 `12345`），说明 dex2oat 正在跑。记下这个数字，然后盯着它的 CPU 占用：

```bash
# 每 2 秒刷新一次，看 dex2oat 的 CPU 使用情况
adb shell top -b -n 1 | grep dex2oat
```

> **想看 CPU 占用在哪个核？**
> ```bash
> adb shell cat /proc/<PID>/stat | awk '{print "CPU:"$39}'
> ```
> 把 `<PID>` 换成上一步的数字。

## 7. 第 4 步：观察和取证

### 7.1 正常情况（炸弹没炸）

如果 dex2oat 在 **2-5 分钟**内执行完毕，窗口 A 回到命令行提示符，说明这次没有触发死锁。

**怎么办**：增加线程数重试。

```bash
# 用 16 线程再试一次（线程越多越容易碰撞）
adb shell dex2oat \
    --dex-file=/data/local/tmp/deadlock_bomb.dex \
    --oat-file=/data/local/tmp/deadlock_bomb.oat \
    --compiler-filter=speed \
    -j16 \
    --watchdog-timeout=600000
```

如果多次尝试（3-5 次）仍然没有挂死，在华为设备上不妨**同时触发设备升温**（跑个 3D 游戏或充电），让热控介入压低 CPU 频率——这会增加每个 Visit 的耗时、扩大碰撞窗口。

### 7.2 炸弹成功（挂死）

**典型现象**：

- 窗口 A 长时间无响应（超过 5 分钟）
- 窗口 B 的 `top` 显示 dex2oat 进程 CPU 占用极低（<5%）或为零
- **最终（约 10 分钟时）**窗口 A 出现错误并退出

```
# 窗口 A 可能的输出：
Aborted
# 或没有任何输出但退出了（exit code 非 0）
```

**这时打开第三个窗口，检查看门狗日志**：

```bash
adb logcat -d | grep -i "dex2oat did not finish"
```

如果看到类似这样的输出，**恭喜，炸弹成功了** ✅：

```
dex2oat did not finish after 600000 milliseconds
```

### 7.3 抓取证栈（最重要的一步）

挂死期间，**在 dex2oat 退出之前**，在窗口 B 中抓线程栈：

```bash
# 方法 1：用 debuggerd（推荐）
adb shell debuggerd -b $(adb shell pidof dex2oat)

# 方法 2：用 kill -3（触发 SIGQUIT）
adb shell kill -3 $(adb shell pidof dex2oat)
# 然后查看 logcat 中输出的线程栈
adb logcat -d | grep -A 5 "Cmd line: dex2oat" | head -50
```

**把抓到的栈保存下来**，这是你分析问题的关键证据。

### 7.4 对照实验（证明就是并行导致的）

用**单线程**跑一次，作为对照：

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/deadlock_bomb.dex \
    --oat-file=/data/local/tmp/deadlock_bomb.oat \
    --compiler-filter=speed \
    -j1
```

**预期结果**：单线程正常完成（几十秒到一分钟），不会挂死。

这证明了：**问题出在线程池并行处理上**。

## 8. 如何确认炸弹成功了

| 现象 | 说明 |
|------|------|
| ✅ dex2oat 超过 10 分钟不返回，最终 `Aborted` | 看门狗触发，炸弹成功 |
| ✅ logcat 有 `"dex2oat did not finish after XXX milliseconds"` | 确认是看门狗杀的 |
| ✅ `debuggerd -b` 抓到的栈中，主线程在 `ThreadPool::Wait` | 主线程在等 worker 完成 |
| ✅ worker 线程栈在 `ResolveMethod` / `FindClass` / `LoadClass` | worker 卡在类加载/解析中 |
| ✅ `-j1` 能正常完成，`-j8` 挂死 | 问题确定是并行引入的 |
| ❌ dex2oat 在 2 分钟内正常完成 | 这次没炸，增加 `-j` 线程数重试 |
| ❌ 报错说找不到 dex2oat | 设备上 dex2oat 没在 PATH 里，改用全路径 `/apex/com.android.art/bin/dex2oat` |

## 9. 常见问题（FAQs）

### Q1：smali 编译报错怎么办？

```bash
# 确认 smali 版本 ≥ 2.5（太老的版本不支持某些语法）
java -jar smali.jar --version

# 如果版本太老，用 GitHub 下载最新版
powershell -Command "Invoke-WebRequest -Uri 'https://github.com/google/smali/releases/download/v3.0.5/smali-3.0.5.jar' -OutFile 'smali.jar'"
```

### Q2：提示 "dex2oat: command not found"？

```bash
# 用全路径代替
adb shell /apex/com.android.art/bin/dex2oat --dex-file=... --oat-file=... --compiler-filter=speed -j8
```

### Q3：测试设备是 user 版本，怎么调试？

user 版本通常不能用 adb 直接执行 dex2oat（SELinux 权限限制）。你需要：

1. 找公司编译组出一个 **userdebug** 版本
2. 或者在 debug 版本上复现，然后再用抓到的栈去量产设备上比对

### Q4：一次没炸，要试多少次？

一般来说 1-3 次就能炸。如果连续 5 次都正常完成：

- 增加线程数 `-j16` 甚至 `-j32`
- 在设备上同时充电或跑其他负载（让 CPU 繁忙，增加碰撞窗口）
- 确认 compiler-filter 是 `speed` 而不是 `verify`

### Q5：炸弹会损坏设备吗？

**不会。** 炸弹只在 dex2oat 进程内部造成锁竞争，最多导致 dex2oat 被看门狗 kill。它不会修改系统文件，不会造成数据丢失，重启后无任何残留影响。

### Q6：能用在 Android 13+ 上吗？

**不能。** Android 16 已经从源码中移除了 `ResolveClassFieldsAndMethodsVisitor`，这个死锁路径不存在了。炸弹只在 **Android 12 及更早版本** 上有效。

## 10. 附录：炸弹里面装了什么

### 你不需要读这一节，它只是给好奇的人看的。

炸弹是一个由以下成分构成的 dex 文件：

| 成分 | 数量 | 作用 |
|------|------|------|
| `PairA` / `PairB` 类对 | 80 对 | 相互密集引用对方的方法签名中 |
| 接口 (`I0`~`I7`) | 8 个 | 增加类层级深度和 LoadClass 递归层数 |
| 每个类的方法 | 32 个 | 30 个交叉引用方法 + `<clinit>` + `<init>` |
| 每个类的字段 | 3 个 | `ref`/`ref2` 引用配对类 + `cross` 随机引用 |
| 方法参数 | 1-5 个 | 50% 是配对类，25% 是随机类，25% 是接口/基本类型 |

**为什么 interleaved 排列很重要**：

```
线程 0 取 class_def[0] = PairA0（引用 PairB0）→ 需要 LoadClass(PairB0)
线程 1 取 class_def[1] = PairB0（引用 PairA0）→ 需要 LoadClass(PairA0)
                          ↑
                    两个线程同时需要对方正在加载的类
                    → classlinker_classes_lock_ 排他锁争抢
                    → 活锁直至看门狗超时
```

### 触发死锁的代码路径

```
dex2oat 启动
  → Dex2oat()
    → Compile()
      → CompilerDriver::PreCompile()           [compiler_driver.cc:762]
        → Resolve()
          → ResolveDexFile()                   [compiler_driver.cc:1666]
            → ResolveClassFieldsAndMethodsVisitor  [compiler_driver.cc:1512]
              → ForAll(class_defs, thread_pool)     ← 8 线程并行
                → Visit(class_idx)
                  → ScopedObjectAccess → mutator_lock_(SHARED)
                  → ResolveType → FindClass → LoadClass
                  → ResolveMethod ×30 → LoadClass(交叉类)
                  → ResolveField ×3   → LoadClass(交叉类)
                    ↓ 碰撞 ↓
                  classlinker_classes_lock_ 排他锁 + GC 介入
                    ↓
                  主线程 thread_pool_->Wait() 永远不返回
                    ↓
                  看门狗 9.5 分钟超时 → Fatal exit(1)
```
