# 慢编译炸弹 — 新手操作指南

> 本指南手把手教你如何用慢编译炸弹让 dex2oat 编译超过 15 分钟。
> **和死锁炸弹不同，这个炸弹不会让 dex2oat"卡死"，而是让它"忙死"** —— CPU 持续满载，每个方法都极难编译。
> 适用于：验证看门狗超时机制、在用户看来"像挂死但实际在算"的场景、压力测试。

## 1. 这东西和死锁炸弹有什么不同？

| | 死锁炸弹 v2 | **慢编译炸弹（本文档）** |
|---|---|---|
| 原理 | 线程循环等待类加载 → 死锁 | 每个方法都极重 → 编译器拼命算 |
| CPU 表现 | 降到 ~0%，进程卡住不动 | **持续 100%**，进程在狂算 |
| `top` 输出 | dex2oat 线程几乎不占 CPU | dex2oat 线程 CPU 80~100% |
| 栈特征 | 主线程 `Wait`，worker `LoadClass`/`WaitForClass` | 全部线程在各编译器 Pass 中(`GVN`/`RegisterAllocator`/`LSE`/`LICM`) |
| 本质 | **假死**（卡了但不干活） | **真忙**（干活但活太重） |

**什么时候用慢编译炸弹？**

- 你想验证：**dex2oat 编译一个"吃 CPU 吃到死"的 dex 时，看门狗会在什么时候触发？**
- 你想在设备上制造一个"表面看起来挂死，实际在编译"的现场，测试你的维测日志能否区分这两种情况
- 你想给华为设备做压力测试：热控限频 + 超重编译 → 看门狗多久超时

python gen_slow_compile_bomb.py

## 5. 第 2 步：编译为 dex

和死锁炸弹一样，需要 `smali.jar`。

```bash
# 如果你已经有 smali.jar（在 D:\Android12 目录下）
java -jar smali.jar assemble slow_smali/ -o slow_bomb.dex

# 如果还没有，先下载
powershell -Command "Invoke-WebRequest -Uri 'https://github.com/google/smali/releases/download/v3.0.5/smali-3.0.5.jar' -OutFile 'smali.jar'"
java -jar smali.jar assemble slow_smali/ -o slow_bomb.dex
```

**这次编译会花一些时间**（~700,000 行 smali → dex），耐心等待。预期生成文件：

```bash
dir slow_bomb.dex
# ✅ 应显示一个约 2~4 MB 的文件
```

## 6. 第 3 步：推送到设备并引爆

### 6.1 推送

```bash
adb push slow_bomb.dex /data/local/tmp/
```

### 6.2 开两个窗口

和死锁炸弹一样，你需要 **两个命令行窗口**：

| 窗口 | 用途 |
|------|------|
| **窗口 A** | 执行 dex2oat 命令 |
| **窗口 B** | 观察 CPU 状态、抓栈 |

### 6.3 窗口 A：引爆（基础版）

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/slow_bomb.dex \
    --oat-file=/data/local/tmp/slow_bomb.oat \
    --compiler-filter=speed \
    -j4 \
    --watchdog-timeout=3600000
```

**参数解释**：

| 参数 | 含义 |
|------|------|
| `--compiler-filter=speed` | 全量编译（必须用 speed，`verify` 不编译方法体） |
| `-j4` | 4 个线程并行编译（48 个重方法分给 4 个线程，每线程约 12 个） |
| `--watchdog-timeout=3600000` | 看门狗超时设为 **60 分钟**（3,600,000 毫秒）——给足时间编译完 |

**执行后，窗口 A 光标停在原地**——这是正常的，dex2oat 正在拼命编译。

### 6.4 窗口 B：观察

```bash
# 找 dex2oat 进程 ID
adb shell pidof dex2oat
# 输出一个数字，如 12345

# 每 2 秒刷新看 CPU
adb shell top -b -n 1 -d 2 | grep dex2oat
```

**关键观察**：dex2oat 的 CPU 占用应该在 **80%~400%**（多线程），而不是接近 0%。如果一直接近 0%，那才是有问题（可能是真的卡死了）。

## 7. 第 4 步：三种引爆模式

### 模式 A：串行煎熬（最慢，总耗时最长）

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/slow_bomb.dex \
    --oat-file=/data/local/tmp/slow_bomb.oat \
    --compiler-filter=speed \
    -j1 \
    --watchdog-timeout=5400000
```

| `-j1` | 单线程串行编译 48 个重方法 |
|--------|--------------------------|
| 预估耗时 | **40~90 分钟** |
| CPU 表现 | 单核 100%，其余核空闲 |
| 适用场景 | 最大化单个方法的感知编译时间，测试"一个方法编译超过 1 分钟"的边界 |

### 模式 B：均衡模式（推荐）

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/slow_bomb.dex \
    --oat-file=/data/local/tmp/slow_bomb.oat \
    --compiler-filter=speed \
    -j4 \
    --watchdog-timeout=3600000
```

| `-j4` | 4 线程，每线程 12 个重方法 |
|--------|---------------------------|
| 预估耗时 | **15~40 分钟** |
| CPU 表现 | 4 核接近满载 |
| 适用场景 | 最接近真实 dex2oat 场景，推荐默认用这个 |

### 模式 C：多线程并行（总耗时最短但也可能很久）

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/slow_bomb.dex \
    --oat-file=/data/local/tmp/slow_bomb.oat \
    --compiler-filter=speed \
    -j8 \
    --watchdog-timeout=1800000
```

| `-j8` | 8 线程，每线程 6 个重方法 |
|--------|--------------------------|
| 预估耗时 | **10~25 分钟** |
| CPU 表现 | 8 核全部满载 |
| 适用场景 | 最大化 CPU 负载，测试热控/限频介入后的编译时间变化 |

## 8. 如何观察和取证

### 8.1 实时观察 CPU（窗口 B）

```bash
# 持续刷新 CPU 占用
while true; do
    adb shell top -b -n 1 | grep dex2oat
    sleep 2
done
```

### 8.2 抓取正在编译的线程栈

**如果编译非常慢但 CPU 满载，栈会显示正在哪个编译器 Pass 中**：

```bash
adb shell debuggerd -b $(adb shell pidof dex2oat)
```

**预期看到**（和死锁炸弹完全不同的栈）：

```
# 正常编译栈示例：
"dex2oat" prio=5 tid=... Runnable
  | group="main" ...
  at art::Arm64InstructionSelector::VisitAdd(...)
  at art::HGraphVisitor::VisitBasicBlock(...)
  at art::GraphVisualizer::Dump(...)
  at art::OptimizingCompiler::CompileOptimized(...)
  at art::CompilerDriver::CompileMethod(...)
  ...
```

关键特征：**线程状态是 `Runnable`（在计算），不是 `Blocked`（在等锁）**。

### 8.3 华为热控专项

如果你在华为设备上测试，可以同时监控热控是否在限速：

```bash
# 看 dex2oat 被绑在哪个 cgroup
adb shell cat /proc/$(adb shell pidof dex2oat)/cgroup | grep cpu

# 看当前 CPU 温度
adb shell cat /sys/class/thermal/thermal_zone*/temp

# 看各个 CPU 核的当前频率
adb shell cat /sys/devices/system/cpu/cpu*/cpufreq/scaling_cur_freq
```

**实验**：对比"常温 vs 充电发热"下的编译总时间——如果发热时显著变慢，说明热控限频在起作用。

### 8.4 记录编译耗时

dex2oat 编译完成时会打印 `--dump-timings` 的时间（如果你加了该参数）：

```bash
adb shell dex2oat \
    --dex-file=/data/local/tmp/slow_bomb.dex \
    --oat-file=/data/local/tmp/slow_bomb.oat \
    --compiler-filter=speed \
    -j4 \
    --watchdog-timeout=3600000 \
    --dump-timings
```

logcat 中会有各阶段耗时明细：

```bash
adb logcat | grep -E "dex2oat|Compile|timing"
```

## 9. 预期的实验结果

### 场景 1：编译完成（没超时）

窗口 A 正常返回命令行提示符，logcat 中有 timing 输出。

- 说明：设备性能足够，48 个重方法在 60 分钟内编译完了
- 操作：减少 `-j` 线程数（如 `-j1`）重新测试，或加大 `N_CLASSES`/`METHODS` 重新生成炸弹

### 场景 2：看门狗超时（超时退出）

窗口 A 出现 `Aborted`，logcat 中看到：

```
dex2oat did not finish after 3600000 milliseconds
```

- 说明：编译时间超过了看门狗超时（60 分钟）
- 此时 `debuggerd -b` 抓的栈应该显示：所有线程都在正常的编译器 Pass 中
- **这证明：进程没死锁，只是编译太慢了，被看门狗误杀**

### 场景 3：华为热控加剧

同一设备、同一炸弹、同一参数，在以下条件下编译时间差异很大：

| 条件 | 预估耗时 |
|------|---------|
| 常温 + 充电器供电 + 无其他负载 | 15~20 分钟 |
| 设备发热 + 电池供电 + 后台 3D 游戏 | 40~70 分钟（可能超时） |

这就是你文档里分析的"限速到超时"的典型场景。

## 10. 常见问题

### Q1：smali 编译报 "Unsigned short value out of range"？

dex 格式对方法指令数有 65535 的上限。如果某个方法超过了，smali 会报错。当前炸弹每个方法约 14000 条指令，远低于限制，不会触发此问题。

### Q2：跑了几分钟还没结束，正常吗？

**正常。**这个炸弹的目的就是让编译"非常慢"。CPU 满载说明一切正常。

### Q3：想要更慢/更快的编译？

编辑 `gen_slow_compile_bomb.py` 顶部的参数：

```python
N_CLASSES = 12    # 少一点 = 快一点（如 6）
METHODS   = 4     # 少一点 = 快一点（如 2）
BLOCKS    = 180   # 少一点 = 快很多（如 60），多一点 = 更慢（如 300）
INSNS     = 70    # 少一点 = 快很多（如 20），多一点 = 更慢（如 150）
```

改完后重新运行 `python gen_slow_compile_bomb.py` 生成新炸弹。

### Q4：这和死锁炸弹有什么不同？

| 死锁炸弹 | 慢编译炸弹 |
|---------|-----------|
| 几千个小类，类间交叉引用 | 12 个大类，方法本身巨大 |
| 引爆点：类加载时的锁 | 引爆点：优化编译器 Pass |
| CPU 从高→突然跌落 0% | CPU 持续满载 |
| 线程栈：`LoadClass`/`WaitForClass` | 线程栈：`GVN`/`RegisterAllocator`/`LSE` |

### Q5：能用于 Android 13+ 吗？

**可以。**和死锁炸弹不同（Android 16 已删除那条代码路径），慢编译炸弹依赖的是 ART 优化编译器本身（所有 Android 版本都有），在任何 Android 版本上都能触发长时间编译。

### Q6：炸弹会损坏设备吗？

**不会。**和死锁炸弹一样，最多导致 dex2oat 被看门狗 kill。它不会修改系统文件，重启后无任何残留。

## 附录：炸弹里塞了什么

### 每类结构

```
class C0 extends Object
├── 8 个实例字段 (I/J/F/D/Object)       ← LSE 优化目标
├── 1 个静态字段 (I)                     ← LSE 优化目标
├── <init>: 构造
├── 10 个 helper 方法 (各 1 个小循环)    ← 内联候选
├── 10 个 static helper 方法 (简单运算)  ← 内联候选
└── 4 个超重方法:
    每个: ~14,000 条指令
      ├── 序言: 30 个各类型数组创建       ← BCE 分析目标
      ├── 180 个基本块                    ← SSA/GVN 目标
      ├── 每块 40~100 条随机指令
      │   ├── int/long/float/double 算术
      │   ├── 类型转换 (12 种)
      │   ├── 数组读写 (6 种类型)
      │   ├── 字段读写 (static + instance)
      │   ├── 对象创建 + invoke
      │   ├── 条件分支(if-eq/if-ne/if-lt...) + goto
      │   └── 跨块数据依赖(phi 节点)
      ├── 3 层嵌套循环                    ← LICM 目标
      └── 4 个 try/catch 块               ← CFG 异常边
```

### 为什么 440 个虚拟寄存器是关键

ARM64 设备只有 32 个物理寄存器可用作通用寄存器。440 个虚拟寄存器意味着：

1. 线性扫描寄存器分配器要把 440 个虚寄存器映射到 32 个物理寄存器
2. 多出来的 400+ 个必须"溢出"到栈上(spill/fill)
3. 每次溢出都需要额外指令（`str`/`ldr`）
4. 寄存器分配器的迭代次数与虚寄存器数量正相关

这就是"440 vregs"是优化器杀手的原因。
