这个方案**技术上可行**，但在 Android 12 上要非常谨慎：它本质上是在用“多份相同数据 + 多套权限”绕开不同进程的 DAC / SELinux 访问差异。短期可能能解决兼容性问题，但长期维护、安全一致性、SELinux 审计、数据同步和权限收敛都会有不少坑。

下面分几个方面说。

---

## 1. 方案是否可行？

可行，前提是：

1. `.so` 内部按固定顺序尝试多个路径；
2. 每个路径上的文件内容一致；
3. 各路径的 Linux DAC 权限、owner/group、SELinux label 都正确；
4. 调用方进程至少能读取其中一个路径；
5. 失败时能区分“无权限”和“文件不存在/损坏/内容不一致”等情况；
6. 不违反 Android 分区、SELinux neverallow、Treble/VNDK、APEX 等约束。

伪代码类似：

```cpp
std::vector<std::string> paths = {
    "/path/a/config",
    "/path/b/config",
    "/path/c/config",
};

for (const auto& path : paths) {
    int fd = TEMP_FAILURE_RETRY(open(path.c_str(), O_RDONLY | O_CLOEXEC));
    if (fd < 0) {
        if (errno == EACCES || errno == EPERM) {
            continue;
        }
        // 是否继续，要看你的策略
        continue;
    }

    // fstat 校验文件类型、大小、owner、mode 等
    // read 内容
    close(fd);
    return content;
}

return error;
```

但**能做**不代表**推荐**。

---

## 2. 最大风险：SELinux 不是简单的“文件权限问题”

Android 上访问文件通常同时受：

1. Linux DAC：uid / gid / mode；
2. SELinux：进程 domain 对文件 type 的 allow 规则；
3. 路径所在分区挂载属性；
4. Android 沙箱/命名空间；
5. 可能还有 fs-verity、dm-verity、APEX mount namespace 等限制。

你这个方案的核心风险是：  
**不同路径虽然内容相同，但 SELinux type 不同，访问策略不同，最终行为可能不可预测、难以维护。**

例如：

```text
/path/a/config  u:object_r:foo_config_file:s0
/path/b/config  u:object_r:bar_config_file:s0
```

进程 A 能读 `foo_config_file`，进程 B 能读 `bar_config_file`。

这能工作，但你引入了多个安全入口。以后某个 domain 新增了权限、某个文件 label 变化、某个路径被迁移，都可能导致行为变化。

更麻烦的是，SELinux 策略可能有 `neverallow` 限制。你可能无法随意让某些 domain 读取某些 type，尤其是跨 system/vendor、app/priv_app、hal、system_server、vendor 进程等边界时。

---

## 3. 风险一：数据一致性问题

多个路径放相同内容，最直接的问题就是：**如何保证一致？**

需要考虑：

### 3.1 更新时是否原子

如果你更新多个文件：

```text
/path/a/config
/path/b/config
/path/c/config
```

更新过程中可能出现：

```text
A 文件是新版本
B 文件是旧版本
C 文件写了一半
```

不同进程读到不同版本，会造成非常难查的问题。

建议：

- 每个文件内部带版本号；
- 带 checksum / hash；
- 更新时使用临时文件 + `fsync` + `rename`；
- 多路径更新要有事务策略；
- 读取时校验版本和完整性。

例如文件内容可以设计成：

```json
{
  "version": 12,
  "content_hash": "xxxx",
  "payload": { ... }
}
```

或者二进制格式中加入 magic、version、length、crc。

### 3.2 开机阶段不同步

Android 上某些文件可能由 init、post-fs-data、vendor init、某个 daemon 生成。不同路径的生成时机不同，可能导致：

- 早期启动进程读不到；
- 后期才同步完成；
- 某些进程读到旧文件。

如果涉及 boot 阶段服务，要特别小心。

---

## 4. 风险二：安全边界被放大

你本来可能只想让每个进程读“同一份配置”，但多路径方案可能让攻击面扩大。

### 4.1 权限最宽的那份文件会成为薄弱点

假设有一份路径为了兼容很多进程，权限较宽：

```text
-r--r--r-- system system /some/path/config
```

或者 SELinux type 被多个 domain 允许读取。

那么这份文件实际决定了整体安全强度。

如果内容敏感，多路径中**任意一个路径泄漏**都等于全部泄漏。

### 4.2 可能意外允许不该读的进程读取

你说“不同路径的权限总和覆盖所有进程”，但这句话在安全上等价于：

> 数据对这些路径权限集合中的所有主体可读。

要确认这是否符合安全设计。

如果这个文件包含：

- 设备标识；
- key/material；
- 账号相关信息；
- 策略开关；
- 调试参数；
- 产品配置；
- 风控策略；

就不应该简单扩大可读范围。

---

## 5. 风险三：路径回退顺序可能导致行为不一致

如果多个文件理论上相同，但实际上可能不一致，那么路径顺序会影响结果。

比如：

```cpp
paths = {
    "/vendor/etc/foo.conf",
    "/system/etc/foo.conf",
    "/data/vendor/foo.conf"
};
```

不同进程由于权限不同，会读到不同文件。即使顺序相同，最终实际命中的文件也可能不同。

这会造成：

- 进程 A 读 `/vendor/etc/foo.conf`；
- 进程 B 读 `/system/etc/foo.conf`；
- 进程 C 读 `/data/vendor/foo.conf`；
- 三者行为不一致。

如果这是配置文件，最终可能出现“同一功能在不同进程里表现不同”。

建议至少做到：

1. 每份文件包含同一个 `version`；
2. 每份文件包含同一个 `uuid` 或 `generation`；
3. `.so` 读取后打印命中路径、版本、hash；
4. debug build 下如果有权限读取多份，则比较内容一致性并报警。

---

## 6. 风险四：错误处理不能只看 `EACCES`

打开失败时，不同 errno 含义不同。

常见情况：

```text
EACCES / EPERM   权限不足，包括 DAC 或 SELinux
ENOENT           文件不存在
ENOTDIR          路径中某级不是目录
ELOOP            符号链接循环
EINVAL           flag 不合法
EMFILE           当前进程 fd 用尽
ENFILE           系统 fd 用尽
EIO              I/O 错误
EINTR            被信号中断
```

不建议简单地：

```cpp
if (open(path) < 0) continue;
```

否则真实问题会被吞掉。

建议策略：

- `EACCES` / `EPERM`：尝试下一个；
- `ENOENT`：可尝试下一个，但要记录；
- `EINTR`：重试；
- `EMFILE` / `ENFILE` / `ENOMEM`：不要继续假装没权限；
- `EIO`：应报告错误；
- `ELOOP`：可视为安全异常；
- 所有失败都要有日志，但避免刷屏。

---

## 7. 风险五：TOCTOU 与符号链接问题

如果路径中有可写目录，或者文件可能被非可信方修改，就要防止：

- symlink 攻击；
- hardlink 问题；
- TOCTOU：检查后到读取前文件被替换；
- 读取的是目录、设备节点、pipe 等非普通文件。

建议：

```cpp
int fd = open(path.c_str(), O_RDONLY | O_CLOEXEC | O_NOFOLLOW);
if (fd >= 0) {
    struct stat st;
    if (fstat(fd, &st) != 0) {
        close(fd);
        continue;
    }

    if (!S_ISREG(st.st_mode)) {
        close(fd);
        continue;
    }

    // 检查 size 上限
    // 检查 owner/group/mode
    // 读取内容
}
```

如果是 Android Bionic 支持的场景，可使用：

```cpp
O_CLOEXEC
O_NOFOLLOW
```

并在读取后基于 `fd` 做 `fstat`，不要 `stat(path)` 后再 `open(path)`。

---

## 8. 风险六：SELinux denial 日志和性能

你的 `.so` 在每个进程里都会按顺序尝试路径。如果很多进程都先访问自己没有权限的路径，会产生大量 SELinux denial。

例如进程 B 每次都先试 `/path/a/config`，但它没有权限，然后再试 `/path/b/config`。

这会导致：

1. `dmesg` / logcat 中出现大量 `avc: denied`；
2. 影响问题排查；
3. 可能被安全审计认为策略有问题；
4. 性能上也有损耗；
5. CTS/VTS 或内部安全测试可能不接受这种设计。

所以不要把“尝试无权限路径”作为正常流程大量发生。

如果必须这样做，建议：

- 每个进程内缓存成功路径；
- 第一次探测成功后只访问该路径；
- 使用静态映射减少失败尝试；
- 按进程 domain / uid / process name 选择优先路径；
- 限制日志频率。

例如：

```cpp
static std::once_flag once;
static std::string cached_path;

std::call_once(once, [] {
    cached_path = ProbeReadablePath();
});

return ReadFromPath(cached_path);
```

但注意 `.so` 被多个进程加载时，每个进程都有自己的静态变量，不是全局共享。

---

## 9. 风险七：Android 分区和命名空间差异

Android 12 上要注意这些路径类别：

### 9.1 `/system`

通常只读，适合放系统固定配置。

```text
/system/etc/...
```

### 9.2 `/vendor`

给 vendor 组件使用，system 进程访问 vendor 文件可能有 Treble 限制。

```text
/vendor/etc/...
```

### 9.3 `/product`、`/system_ext`

不同设备、不同构建配置下权限和挂载可能不同。

### 9.4 `/data`

可写，但需要非常小心 owner、mode、SELinux label。

```text
/data/vendor/...
/data/misc/...
/data/system/...
```

`/data` 下的文件更容易出现：

- 创建时 label 不对；
- OTA 后残留旧文件；
- factory reset 后丢失；
- CE/DE 存储时机问题；
- 开机早期不可用；
- 被错误权限创建。

如果这个文件在 early boot 阶段要读，避免放在 credential encrypted storage。

---

## 10. 风险八：app 进程无法随意读系统私有路径

如果调用 `.so` 的进程包括普通 app 进程，那更复杂。

Android app 有：

- per-app UID；
- SELinux domain；
- scoped storage；
- 应用沙箱；
- targetSdk 行为差异。

即便 Linux mode 是 `0644`，SELinux 也可能阻止 app 访问很多系统路径。

如果要给普通 app 提供数据，通常更推荐：

- ContentProvider；
- Binder service；
- system API；
- AIDL；
- 通过 app 自己的 assets / resources；
- public readable 的受控路径，但要慎重。

不建议让普通 app 直接靠 `.so` 读多个系统路径来适配权限。

---

## 11. 风险九：不能把 SELinux 权限问题藏在 native 库里

从架构上看，如果多个进程都需要同一份数据，通常更好的方案是：

### 方案 A：由一个有权限的服务统一读取，通过 Binder 提供接口

这是 Android 上更推荐的方式。

```text
config file
    ↓
privileged service / daemon
    ↓ Binder / AIDL / HIDL / stable AIDL
clients
```

优点：

- 文件只需放一份；
- 权限集中控制；
- SELinux 策略清晰；
- 可做访问控制；
- 可缓存；
- 可统一日志；
- 更新一致性好。

缺点：

- 要新增服务；
- 有 IPC 成本；
- 需要定义接口。

### 方案 B：把文件放到所有目标进程都合法可读的单一路径

如果数据不敏感，最好找一个统一路径和统一 SELinux type，使所有需要读的 domain 都被明确允许。

例如：

```text
/system/etc/your_config.xml
/vendor/etc/your_config.xml
/product/etc/your_config.xml
```

具体放哪里取决于数据归属和调用方归属。

### 方案 C：编译进库或资源

如果文件内容不需要运行时更新，可以考虑：

- 编译进 `.so`；
- 生成头文件；
- 放到 assets；
- 放到 APK / APEX / system image 中。

这样没有运行时文件权限问题。

### 方案 D：使用 Android 系统属性

如果内容很小、简单、非敏感，可以考虑 system property。

例如：

```cpp
__system_property_get("ro.xxx.yyy", buf);
```

但要注意：

- property 长度限制；
- property context；
- `ro.` 只读属性；
- `persist.` 属性时机；
- SELinux property 权限；
- 不适合复杂内容。

### 方案 E：通过 init 创建统一权限的文件

如果文件必须在 `/data`，可以用 `init.rc` 创建目录、设置 owner、mode、restorecon，并由可信进程写入。

## 12. 如果你坚持使用多路径 fallback，建议这样设计

### 12.1 固定文件格式，带完整性校验

文件中至少包含：

- magic；
- format version；
- data version；
- length；
- checksum/hash；
- payload。

读取后先校验，不合法就尝试下一个或返回错误。

### 12.2 区分错误类型

建议记录每个路径失败原因：

```text
/path/a/config: EACCES
/path/b/config: ENOENT
/path/c/config: bad checksum
```

最终全部失败时，把这些信息汇总输出。

### 12.3 命中路径缓存

每个进程首次探测后缓存结果，避免反复触发 denial。

```cpp
class ConfigReader {
public:
    std::string Read();

private:
    std::string ResolvePathOnce();
};
```

注意多线程安全：

```cpp
std::once_flag g_once;
std::string g_path;
std::string g_error;
```

### 12.4 使用 `open` + `fstat`，不要先 `access`

不要这样：

```cpp
if (access(path, R_OK) == 0) {
    fd = open(path, O_RDONLY);
}
```

原因：

- `access()` 与 `open()` 之间有 TOCTOU；
- `access()` 检查的语义和实际 `open()` 可能不完全一致；
- SELinux audit 也可能被触发；
- 没必要多一次系统调用。

直接：

```cpp
fd = open(path, O_RDONLY | O_CLOEXEC | O_NOFOLLOW);
```

### 12.5 限制文件大小

防止异常大文件导致内存问题：

```cpp
constexpr off_t kMaxConfigSize = 1024 * 1024;

if (st.st_size < 0 || st.st_size > kMaxConfigSize) {
    close(fd);
    return error;
}
```

### 12.6 确保路径目录不可被非可信用户写

如果路径在 `/data` 下，目录权限尤其重要：

```text
/data/vendor/xxx
owner: system or vendor-specific uid
mode: 0750 or stricter
label: correct selinux type
```

避免：

```text
0777
world-writable
app-writable
shell-writable in user build
```

### 12.7 使用 `restorecon`

如果由 init 创建或 OTA 后生成文件，要确保 label 正确：

```rc
restorecon_recursive /data/vendor/xxx
```

或使用 `file_contexts` 正确匹配。

### 12.8 保证 OTA 和回滚行为

多路径文件要考虑：

- OTA 后 system/vendor 文件更新；
- `/data` 下旧文件未更新；
- 版本回滚；
- factory reset；
- 首次开机；
- A/B 分区切换；
- vendor/system 版本不匹配。

读取逻辑最好能识别版本不兼容。

## 13. C++ 示例框架

可以参考这种结构：

```cpp
#include <fcntl.h>
#include <sys/stat.h>
#include <unistd.h>
#include <errno.h>
#include <string>
#include <vector>
#include <sstream>

namespace {

constexpr off_t kMaxFileSize = 1024 * 1024;

struct ReadResult {
    bool ok = false;
    std::string data;
    std::string error;
    std::string path;
};

bool ReadAllFromFd(int fd, std::string* out, std::string* err) {
    struct stat st {};
    if (fstat(fd, &st) != 0) {
        *err = "fstat failed";
        return false;
    }

    if (!S_ISREG(st.st_mode)) {
        *err = "not regular file";
        return false;
    }

    if (st.st_size < 0 || st.st_size > kMaxFileSize) {
        *err = "invalid file size";
        return false;
    }

    out->clear();
    out->reserve(static_cast<size_t>(st.st_size));

    char buf[4096];
    while (true) {
        ssize_t n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            out->append(buf, static_cast<size_t>(n));
            if (out->size() > static_cast<size_t>(kMaxFileSize)) {
                *err = "file too large while reading";
                return false;
            }
        } else if (n == 0) {
            break;
        } else {
            if (errno == EINTR) {
                continue;
            }
            *err = "read failed";
            return false;
        }
    }

    return true;
}

ReadResult TryReadOne(const std::string& path) {
    ReadResult result;
    result.path = path;

    int fd;
    do {
        fd = open(path.c_str(), O_RDONLY | O_CLOEXEC | O_NOFOLLOW);
    } while (fd < 0 && errno == EINTR);

    if (fd < 0) {
        result.error = std::string("open failed errno=") + std::to_string(errno);
        return result;
    }

    std::string data;
    std::string err;
    bool ok = ReadAllFromFd(fd, &data, &err);
    close(fd);

    if (!ok) {
        result.error = err;
        return result;
    }

    // TODO: validate magic/version/checksum here

    result.ok = true;
    result.data = std::move(data);
    return result;
}

} // namespace

ReadResult ReadConfigWithFallback() {
    const std::vector<std::string> paths = {
        "/vendor/etc/xxx/config.bin",
        "/system/etc/xxx/config.bin",
        "/data/vendor/xxx/config.bin",
    };

    std::ostringstream errors;

    for (const auto& path : paths) {
        ReadResult r = TryReadOne(path);
        if (r.ok) {
            return r;
        }

        errors << path << ": " << r.error << "\n";

        // 可以根据 errno 决定是否继续。
        // 当前示例为了兼容性，所有失败都继续尝试。
    }

    ReadResult final_result;
    final_result.ok = false;
    final_result.error = errors.str();
    return final_result;
}
```

实际工程里建议把 errno 保存下来，不要在调用其他函数后再读 errno。

## 14. 更推荐的架构判断

可以按下面原则选方案：

### 如果文件内容敏感

不建议多路径 fallback。  
推荐：

```text
单一特权服务读取 + Binder/AIDL 对外提供受控接口
```

### 如果文件内容不敏感、只读、体积小

可以考虑：

```text
统一 public-readable 配置路径
```

或：

```text
编译进 so / system property
```

### 如果调用方跨 system/vendor/app 多种 domain

优先考虑 Binder 服务，不要让所有进程直接读文件。

### 如果只是少数 native daemon 读取

可以通过 SELinux policy 明确授权读取同一个 file type，而不是复制多份文件。

## 15. 你这个方案最需要注意的坑总结

重点如下：

1. **SELinux denial 会很多**：每次先试无权限路径都会触发 AVC。
2. **多份文件一致性很难保证**：尤其 OTA、运行时更新、开机早期。
3. **权限总和扩大了安全暴露面**。
4. **路径命中不同会导致不同进程行为不同**。
5. **不要用 `access()` 判断权限后再 `open()`**。
6. **必须防 symlink、非普通文件、TOCTOU**。
7. **要缓存成功路径，避免反复探测**。
8. **要做版本号、checksum、文件大小限制**。
9. **要考虑 `/system`、`/vendor`、`/product`、`/data` 的 Treble 和挂载差异**。
10. **如果有普通 app 进程参与，风险和限制会更大**。
11. **不要吞掉所有错误，否则调试会非常困难**。
12. **user build 上的 SELinux 行为要实际验证，不要只看 eng/userdebug**。

---

## 结论

你的方案**可以实现**，但更像是一个兼容性 workaround，不是最理想的 Android 架构。

如果数据敏感、需要长期维护、调用方很多且权限差异大，建议改成：

```text
一个有权限的 native/system 服务读取文件
其他进程通过 Binder/AIDL 获取内容
```

如果数据不敏感且静态，建议：

```text
放到一个统一合法可读路径
```

或者：

```text
编译进 so / 使用系统属性 / 使用资源文件
```

只有在短期无法调整 SELinux 策略、无法新增服务、且文件内容不敏感的情况下，才建议采用“多路径 fallback”方案，并务必做好日志、校验、缓存和一致性保护。
