# Android 12 热更新文件记录接口 —— 综合设计方案

> 本方案综合四份独立 AI 方案的核心优势，面向 Android 12 系统级热更新场景，对接口设计、
> 文件格式、并发安全、DAC/SELinux 双层权限管控给出可落地的完整设计。

---

## 一、TSV 还是 SQLite？

### 结论：使用 TSV（Tab-Separated Values）明文文件

| 维度 | TSV 明文 | SQLite |
|---|---|---|
| 人类可读 | ✅ `cat`/`grep`/`sed` 直接操作 | ❌ 二进制，需 sqlite3 工具（user 版本通常缺失） |
| init 依赖 | ✅ 纯 POSIX，零额外依赖 | ❌ 需链接 libsqlite，增大 init 攻击面 |
| 崩溃恢复 | ✅ 临时文件 + fsync + rename，等价原子事务 | ✅ WAL/journal 自带 |
| SELinux 管控 | ✅ 仅需管控 2~3 个文件 | ❌ 还需管控 -wal、-shm、-journal 等辅助文件 |
| 并发写 | ⚠️ 需 flock 配合 | ✅ 内置事务 |
| 数据模型匹配 | ✅ 简单 KV，<100 条记录 | ❌ 索引、SQL、事务能力完全用不上 |

**核心判断逻辑：**

1. **需求硬约束**："文件需要使用人类可识读的明文"——SQLite 是二进制格式，直接出局。
2. **运行阶段约束**：写接口调用方是 `init`（PID 1），它运行在开机极早期，依赖最小化是硬道理。引入 SQLite 意味着 init 需要处理 db 锁、WAL 日志、页缓存损坏恢复——这是明显的可靠性负担。
3. **数据特征**：两列字符串、几十条记录、单写者多读者、查询模式只有"全量读取"和"追加去重"。这正是 TSV 的甜点区。

**最终裁决：TSV 明文文件 + 原子 rename 写入 + flock 并发控制。**

## 二、文件格式设计

### 2.1 `/data/hotupdate/filelist`

```
# hotupdate-filelist-v1
# format: F<TAB>filename<TAB>filepath
F	libfoo.so	/system/lib64/libfoo.so
F	framework.jar	/system/framework/framework.jar
F	services.jar	/system/framework/services.jar
```

### 2.2 `/data/hotupdate/proplist`

```
# hotupdate-proplist-v1
# format: P<TAB>propname<TAB>proppath
P	persist.sys.hotupdate.enable	/data/local.prop
P	ro.vendor.build.fingerprint	/vendor/build.prop
P	persist.vendor.hotupdate.version	/vendor/default.prop
```

### 2.3 格式约定

| 规则 | 说明 |
|---|---|
| 分隔符 | 单个 `\t`（Tab）。选择 Tab 而非空格，因为文件路径中允许空格，而 Tab 可在输入校验中无条件禁止 |
| 行尾 | `\n`（LF） |
| 注释 | `#` 开头的行为注释/表头行，读写均忽略 |
| 类型前缀 | 首字段 `F` 表示文件记录，`P` 表示属性记录——使格式自描述，便于未来扩展 |
| 编码 | UTF-8 |
| 唯一性 | 整行 `F\tfilename\tfilepath` 或 `P\tpropname\tproppath` 组合唯一，重复写入幂等返回 0 |

### 2.4 字段合法性约束

所有字段**禁止**包含：`\0`、`\t`、`\n`、`\r`，以及所有 ASCII 控制字符（`0x00~0x1F`、`0x7F`）。

| 字段 | 长度 | 额外约束 |
|---|---|---|
| `filename` | ≤ 255 | 不能含 `/`，不能含空格 |
| `filepath` | < PATH_MAX | 必须以 `/` 开头（绝对路径），不能含 `..`，限制在白名单路径前缀 |
| `propname` | ≤ 128 | 仅允许 `[a-zA-Z0-9_.-]` |
| `proppath` | < PATH_MAX | 必须以 `/` 开头，不能含 `..`，建议以 `.prop` 结尾 |

**路径白名单**（filepath/proppath 统一适用）：

```
/system/   /system_ext/   /product/   /vendor/   /odm/   /data/
```

## 三、TSV vs DB 深度对比（补充视角）

### 3.1 原子性实现对比

Fable 方案采用了"**写临时文件 → fsync → rename → fsync 目录**"策略，这从根本上保证了崩溃一致性：

- **write 到一半崩溃**：临时文件残留，`filelist` 完好无损，下次 init 调用时 `unlink` 清理 tmp 即可。
- **rename 前崩溃**：同上。
- **rename 后、目录 fsync 前崩溃**：rename 是文件系统元数据操作，ext4/f2fs 保证 rename 本身是原子的——目录项要么指向旧 inode，要么指向新 inode，不存在"指到一半"的中间态。目录 fsync 失败仅意味着 rename 可能未落盘，不会导致文件损坏。

相比之下，SQLite 的事务原子性依赖 `fsync` WAL/journal 文件，在 Android 的 FBE（文件级加密）环境下，init 阶段 `/data` 刚挂载、密钥可能尚未就绪时，SQLite 的 mmap 和文件操作行为更加不可预测。

### 3.2 读写并发

使用 rename 原子替换后，**读接口不需要加锁**——任何时候 `open` 打开的都是一个完整的旧文件或完整的新文件，绝不会读到"写到一半"的行。这与 SQLite 的 WAL 模式（reader 读旧版本、writer 写 WAL）在并发语义上是等价的，但实现简单了一个数量级。

唯一的并发风险是**多个写者同时写**——使用独立的 `.lock` 文件 + `flock(LOCK_EX)` 解决。当前场景只有 init 写，锁主要用于防御未来可能的并发写入者。

### 5.1 推荐目录结构

```
system/core/hotupdate/
├── Android.bp
├── include/hotupdate/
│   ├── hotupdate_writer.h       # 写接口头文件（仅 init 可见）
│   └── hotupdate_client.h       # 读接口头文件（公开）
```

**库拆分原则：**

- `libhotupdate_writer.a`（静态库）：链入 init，提供 `hotupdatefiles()`/`hotupdateprops()`。用 `visibility` 限制链接范围，防止普通应用误链。

### 5.3 公共实现 `hotupdate_common.cpp`（核心逻辑）

```cpp
#define LOG_TAG "hotupdate"

#include "hotupdate_common.h"

#include <errno.h>
#include <fcntl.h>
#include <string.h>
#include <sys/file.h>
#include <sys/stat.h>
#include <unistd.h>

#include <cctype>
#include <set>
#include <string>
#include <vector>

#include <log/log.h>

namespace android {
namespace hotupdate {
namespace internal {

// ---- 字段校验 ----

static bool HasForbiddenChars(const std::string& s) {
    for (unsigned char c : s) {
        if (c == '\0' || c == '\n' || c == '\r' || c == '\t') return true;
        if (c < 0x20 || c == 0x7F) return true;  // 所有控制字符
    }
    return false;
}

static bool HasSpace(const std::string& s) {
    return s.find(' ') != std::string::npos;
}

bool IsValidFileName(const std::string& name) {
    if (name.empty() || name.size() > kMaxFileNameLen) return false;
    if (HasForbiddenChars(name)) return false;
    if (HasSpace(name)) return false;
    if (name.find('/') != std::string::npos) return false;
    if (name[0] == '#') return false;  // 避免与注释冲突
    return true;
}

bool IsValidPropName(const std::string& name) {
    if (name.empty() || name.size() > kMaxPropNameLen) return false;
    if (HasForbiddenChars(name)) return false;
    for (char c : name) {
        if (!(std::isalnum(static_cast<unsigned char>(c)) ||
              c == '_' || c == '.' || c == '-')) {
            return false;
        }
    }
    return true;
}

static bool IsAllowedPathPrefix(const std::string& path) {
    static const char* kPrefixes[] = {
        "/system/", "/system_ext/", "/product/",
        "/vendor/", "/odm/",       "/data/",
    };
    for (const char* p : kPrefixes) {
        if (path.compare(0, strlen(p), p) == 0) return true;
    }
    return false;
}

static bool IsValidAbsolutePath(const std::string& path) {
    if (path.empty() || path.size() >= kMaxPathLen) return false;
    if (path[0] != '/') return false;
    if (HasForbiddenChars(path)) return false;
    if (HasSpace(path)) return false;
    if (path.find("/../") != std::string::npos) return false;
    if (path.size() >= 3 &&
        path.compare(path.size() - 3, 3, "/..") == 0) return false;
    if (!IsAllowedPathPrefix(path)) return false;
    return true;
}

bool IsValidPath(const std::string& path) {
    return IsValidAbsolutePath(path);
}

bool IsValidPropPath(const std::string& path) {
    if (!IsValidAbsolutePath(path)) return false;
    // 必须以 .prop 结尾（兼容 local.prop / build.prop / default.prop）
    if (path.size() < 5) return false;
    if (path.compare(path.size() - 5, 5, ".prop") != 0) return false;
    return true;
}

// ---- TSV 构造与解析 ----

std::string BuildTsvRecord(char type,
                           const std::string& key,
                           const std::string& value) {
    std::string line;
    line.reserve(1 + 1 + key.size() + 1 + value.size() + 1);
    line.push_back(type);
    line.push_back('\t');
    line.append(key);
    line.push_back('\t');
    line.append(value);
    line.push_back('\n');
    return line;
}

bool ParseTsvLine(const std::string& line,
                  char expected_type,
                  std::string* key,
                  std::string* value) {
    if (line.empty() || line[0] == '#') return false;

    size_t tab1 = line.find('\t');
    if (tab1 == std::string::npos || tab1 == 0) return false;

    size_t tab2 = line.find('\t', tab1 + 1);
    if (tab2 == std::string::npos || tab2 == tab1 + 1) return false;

    // 确保只有两列（不能有第三个 tab）
    if (line.find('\t', tab2 + 1) != std::string::npos) return false;

    std::string type_str = line.substr(0, tab1);
    if (type_str.size() != 1 || type_str[0] != expected_type) return false;

    std::string k = line.substr(tab1 + 1, tab2 - tab1 - 1);
    std::string v = line.substr(tab2 + 1);

    if (k.empty() || v.empty()) return false;
    if (HasForbiddenChars(k) || HasForbiddenChars(v)) return false;

    if (key) *key = std::move(k);
    if (value) *value = std::move(v);
    return true;
}

// ---- 文件读写工具 ----

bool ReadFileContent(const char* path, std::string* out) {
    out->clear();
    int fd = TEMP_FAILURE_RETRY(
        open(path, O_RDONLY | O_CLOEXEC | O_NOFOLLOW));
    if (fd < 0) {
        if (errno == ENOENT) return true;  // 不存在即空
        ALOGE("open %s for read: %s", path, strerror(errno));
        return false;
    }
    char buf[4096];
    ssize_t n;
    while ((n = TEMP_FAILURE_RETRY(read(fd, buf, sizeof(buf)))) > 0) {
        out->append(buf, static_cast<size_t>(n));
        if (out->size() > kMaxListFileSize) {
            ALOGE("%s exceeds size limit", path);
            close(fd);
            return false;
        }
    }
    close(fd);
    return n == 0;
}

bool WriteFully(int fd, const char* data, size_t len) {
    while (len > 0) {
        ssize_t n = TEMP_FAILURE_RETRY(write(fd, data, len));
        if (n <= 0) return false;
        data += n;
        len -= static_cast<size_t>(n);
    }
    return true;
}

// === 读取并解析列表文件，返回去重后的 (key, value) 对列表 ===
std::vector<std::pair<std::string, std::string>>
ReadEntries(const char* path, char expected_type) {
    std::vector<std::pair<std::string, std::string>> result;
    std::set<std::string> seen;  // 按 key 去重，保留首次出现顺序

    std::string content;
    if (!ReadFileContent(path, &content)) return result;

    size_t pos = 0;
    while (pos < content.size() && result.size() < kMaxRecords) {
        size_t eol = content.find('\n', pos);
        std::string line = (eol == std::string::npos)
            ? content.substr(pos)
            : content.substr(pos, eol - pos);
        pos = (eol == std::string::npos) ? content.size() : eol + 1;

        // 去掉 \r
        if (!line.empty() && line.back() == '\r') line.pop_back();
        if (line.empty() || line[0] == '#') continue;

        std::string key, value;
        if (!ParseTsvLine(line, expected_type, &key, &value)) {
            ALOGW("skip malformed line in %s", path);
            continue;
        }
        if (seen.insert(key).second) {
            result.emplace_back(std::move(key), std::move(value));
        }
    }
    return result;
}

}  // namespace internal
}  // namespace hotupdate
}  // namespace android
```

### 5.5 写实现 `hotupdate_writer.cpp`（原子 rename 方案）

```cpp
#define LOG_TAG "hotupdate"

#include "hotupdate_writer.h"
#include "hotupdate_common.h"

#include <errno.h>
#include <fcntl.h>
#include <string.h>
#include <sys/file.h>
#include <sys/stat.h>
#include <sys/types.h>
#include <unistd.h>

#include <algorithm>
#include <set>
#include <string>
#include <vector>

#include <log/log.h>
#include <selinux/android.h>   // selinux_android_restorecon

namespace android {
namespace hotupdate {
namespace {

using internal::kHotUpdateDir;
using internal::kFileListPath;
using internal::kPropListPath;
using internal::kLockPath;
using internal::kFileListHeader;
using internal::kPropListHeader;
using internal::kListFileMode;
using internal::kDirMode;
using internal::BuildTsvRecord;
using internal::ReadFileContent;
using internal::WriteFully;
using internal::ParseTsvLine;
using internal::ReadEntries;

// === 确保目录存在 ===
int EnsureDir() {
    struct stat st{};
    if (stat(kHotUpdateDir, &st) == 0) {
        if (!S_ISDIR(st.st_mode)) {
            ALOGE("%s exists but is not a directory", kHotUpdateDir);
            return HOTUPDATE_ERR_MKDIR_FAILED;
        }
        chmod(kHotUpdateDir, kDirMode);
        return HOTUPDATE_OK;
    }
    if (mkdir(kHotUpdateDir, kDirMode) != 0 && errno != EEXIST) {
        ALOGE("mkdir %s failed: %s", kHotUpdateDir, strerror(errno));
        return HOTUPDATE_ERR_MKDIR_FAILED;
    }
    chmod(kHotUpdateDir, kDirMode);
    return HOTUPDATE_OK;
}

// === 原子写入：读全量 → 去重追加 → 写临时文件 → rename ===
int AtomicAppendRecord(const char* path,
                       const char* header,
                       char record_type,
                       const std::string& key,
                       const std::string& value) {
    int rc;

    // 1. 确保目录存在
    rc = EnsureDir();
    if (rc != HOTUPDATE_OK) return rc;

    // 2. 排他锁（防御未来多写者场景）
    int lock_fd = TEMP_FAILURE_RETRY(
        open(kLockPath, O_RDWR | O_CREAT | O_CLOEXEC, 0600));
    if (lock_fd < 0) {
        ALOGE("open lock file failed: %s", strerror(errno));
        return HOTUPDATE_ERR_OPEN_FAILED;
    }
    if (flock(lock_fd, LOCK_EX) != 0) {
        ALOGE("flock failed: %s", strerror(errno));
        close(lock_fd);
        return HOTUPDATE_ERR_LOCK_FAILED;
    }

    // 3. 读取已有记录并去重
    auto entries = ReadEntries(path, record_type);
    std::string new_record = BuildTsvRecord(record_type, key, value);

    // 用 key+value 组合判重
    for (const auto& e : entries) {
        if (e.first == key && e.second == value) {
            // 已存在 → 幂等成功
            flock(lock_fd, LOCK_UN);
            close(lock_fd);
            return HOTUPDATE_OK;
        }
    }
    entries.emplace_back(key, value);

    // 4. 按 key 排序并去重（双重保障）
    std::sort(entries.begin(), entries.end());
    entries.erase(std::unique(entries.begin(), entries.end()), entries.end());

    // 5. 构造新文件内容
    std::string new_content(header);
    for (const auto& e : entries) {
        new_content += BuildTsvRecord(record_type, e.first, e.second);
    }

    // 6. 写临时文件
    std::string tmp_path = std::string(path) + ".tmp";
    unlink(tmp_path.c_str());  // 清理残留

    int tmp_fd = TEMP_FAILURE_RETRY(
        open(tmp_path.c_str(),
             O_WRONLY | O_CREAT | O_EXCL | O_CLOEXEC | O_NOFOLLOW,
             0644));
    if (tmp_fd < 0) {
        ALOGE("open tmp %s failed: %s", tmp_path.c_str(), strerror(errno));
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return HOTUPDATE_ERR_OPEN_FAILED;
    }

    if (!WriteFully(tmp_fd, new_content.data(), new_content.size())) {
        ALOGE("write tmp %s failed", tmp_path.c_str());
        close(tmp_fd);
        unlink(tmp_path.c_str());
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return HOTUPDATE_ERR_WRITE_FAILED;
    }

    // 7. fchmod + fsync 临时文件
    if (fchmod(tmp_fd, kListFileMode) != 0) {
        ALOGE("fchmod tmp failed: %s", strerror(errno));
        close(tmp_fd);
        unlink(tmp_path.c_str());
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return HOTUPDATE_ERR_CHMOD_FAILED;
    }
    if (fsync(tmp_fd) != 0) {
        ALOGE("fsync tmp failed: %s", strerror(errno));
        close(tmp_fd);
        unlink(tmp_path.c_str());
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return HOTUPDATE_ERR_FSYNC_FAILED;
    }
    close(tmp_fd);

    // 8. 原子 rename
    if (rename(tmp_path.c_str(), path) != 0) {
        ALOGE("rename to %s failed: %s", path, strerror(errno));
        unlink(tmp_path.c_str());
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return HOTUPDATE_ERR_RENAME_FAILED;
    }

    // 9. restorecon 修正 SELinux 标签
    selinux_android_restorecon(path, 0);

    // 10. fsync 目录，确保 rename 落盘
    int dir_fd = TEMP_FAILURE_RETRY(
        open(kHotUpdateDir, O_RDONLY | O_DIRECTORY | O_CLOEXEC));
    if (dir_fd >= 0) {
        fsync(dir_fd);
        close(dir_fd);
    }

    flock(lock_fd, LOCK_UN);
    close(lock_fd);
    return HOTUPDATE_OK;
}

}  // namespace

int hotupdatefiles(const std::string& filename, const std::string& filepath) {
    if (!internal::IsValidFileName(filename) ||
        !internal::IsValidPath(filepath)) {
        ALOGE("hotupdatefiles: invalid argument");
        return HOTUPDATE_ERR_INVALID_ARG;
    }
    return AtomicAppendRecord(kFileListPath, kFileListHeader,
                              'F', filename, filepath);
}

int hotupdateprops(const std::string& propname, const std::string& proppath) {
    if (!internal::IsValidPropName(propname) ||
        !internal::IsValidPropPath(proppath)) {
        ALOGE("hotupdateprops: invalid argument");
        return HOTUPDATE_ERR_INVALID_ARG;
    }
    return AtomicAppendRecord(kPropListPath, kPropListHeader,
                              'P', propname, proppath);
}

}  // namespace android
```

### 5.8 关键设计决策说明

**为什么采用"全量重写 + rename"而非"追加写"?**

| 维度 | 追加写 | 全量重写 + rename（本方案） |
|---|---|---|
| 崩溃安全 | 追加一半崩溃 = 文件末尾有半行 | 写 tmp 失败不影响原文件 |
| 读者一致性 | 需读者也 flock(SH) 才能保证不读到半行 | 无需锁，rename 原子替换 |
| 文件增长 | 只增不减 | 每次全量重写（去重后可能缩小） |
| 大文件性能 | O(1) 追加 | O(n) 重写 |
| 适用场景 | 大文件、高频写入 | <100 条记录、低频写入 → 本项目 |

对于热更新场景（几十条记录、低频写入），全量重写的性能开销完全可以忽略，而它带来的崩溃安全性和读端零锁优势是决定性的。

**为什么读接口按 key 去重？**

同一个 `filename` 可能对应不同 `filepath`（如 `foo.so` 同时存在于 `/system/` 和 `/vendor/`）。`gethotupdatefiles()` 只返回文件名列表。本方案按**首次出现**保留去重后的 key，避免调用方收到重复条目。如果未来需要知道完整映射，可扩展接口返回 `vector<pair<string,string>>`。

## 七、权限与 SELinux 管控方案

> **核心目标**：任意进程可读、任何进程（除 init）不能写、不能删除、不能改名。

这需要 **DAC（文件权限）+ SELinux（强制访问控制）+ neverallow（编译期锁死）** 三层防线协同。

### 7.1 第一层：DAC 文件权限

```
/data/hotupdate/              0711 root root    ← 目录：可穿越但不可列出内容
/data/hotupdate/filelist      0444 root root    ← 所有人只读
/data/hotupdate/proplist      0444 root root    ← 所有人只读
/data/hotupdate/.lock         0600 root root    ← 仅 root 可读写
```

**关键认知**：文件设为 `0444` 只能防止"改文件内容"，**不能防止"删文件"**。能否删除一个文件取决于其**父目录的写权限**，而非文件本身权限。

| 权限设计 | 效果 |
|---|---|
| 目录 `0711` | owner(root) 可读写执行；group/other **只有 x（可穿越到已知路径）**，无 r（不能 `ls`）无 w（不能增删文件） |
| 文件 `0444` | 所有人只读 |
| `.lock` `0600` | 仅 root 可读写，普通进程看不见也摸不到 |

### 7.2 第二层：SELinux 强制访问控制

#### (a) 定义 SELinux 类型（`file.te`）

```te
# 目录类型
type hotupdate_dir, file_type, data_file_type;

# 公开文件（filelist / proplist）—— 需跨域可读
type hotupdate_public_file, file_type, data_file_type, mlstrustedobject;

# 私有文件（.lock / .tmp）—— 仅 writer 域可访问
type hotupdate_private_file, file_type, data_file_type;
```

**设计要点**：

- `file_type` + `data_file_type` 是 `/data` 下文件的强制性属性，缺失会触发系统 neverallow 编译失败。
- `mlstrustedobject`：**这是关键属性**。Android 普通应用运行在带 MLS category 的安全上下文（如 `c512,c768`），不加此属性，跨 category 的应用读取会被 MLS 规则拦截。加上后等价于声明"此文件的 MLS 类别检查豁免"。
- 拆分 `public_file` 和 `private_file` 实现**最小权限**——普通读者域只授予到 public_file 的读权限，永远看不到 lock 文件和临时文件。

#### (b) 文件标签（`file_contexts`）

```
/data/hotupdate                  u:object_r:hotupdate_dir:s0
/data/hotupdate/filelist         u:object_r:hotupdate_public_file:s0
/data/hotupdate/proplist         u:object_r:hotupdate_public_file:s0
/data/hotupdate/\.lock           u:object_r:hotupdate_private_file:s0
/data/hotupdate/.*\.tmp          u:object_r:hotupdate_private_file:s0
```

#### (c) 写者权限（Writer Domain）

以 `init` 域为例（若热更新由独立 service 执行，替换为对应域名）：

```te
# init 对目录有完整管理权限
allow init hotupdate_dir:dir {
    create read write open search getattr setattr
    add_name remove_name
};

# init 对公开文件有完整管理权限
allow init hotupdate_public_file:file {
    create open read write append getattr setattr
    lock rename unlink
};

# init 对私有文件有完整管理权限
allow init hotupdate_private_file:file {
    create open read write append getattr setattr
    lock rename unlink
};
```

#### (d) 读者权限（Reader Domain）—— 推荐 Attribute 白名单模式

```te
# 定义 reader 属性
attribute hotupdate_reader_domain;

# 通过属性统一授权
allow hotupdate_reader_domain hotupdate_dir:dir { search getattr };
allow hotupdate_reader_domain hotupdate_public_file:file {
    open read getattr map lock
};

# 将需要读取的域加入白名单
typeattribute system_server hotupdate_reader_domain;
typeattribute platform_app   hotupdate_reader_domain;
typeattribute priv_app       hotupdate_reader_domain;
typeattribute vendor_app     hotupdate_reader_domain;
# ... 按需添加其他 domain
```

**如果确实需要"所有 domain 可读"（理解提示词的"任意进程"）：**

```te
# 注意：必须排除 -isolated_app，否则触发平台 neverallow 编译失败
# 也建议排除 -untrusted_app，通过系统 service 间接暴露给三方应用更安全
allow { domain -isolated_app } hotupdate_dir:dir { search getattr };
allow { domain -isolated_app } hotupdate_public_file:file {
    open read getattr map lock
};
```

但如果希望进一步授予三方普通应用（untrusted_app），需要评估信息泄露风险——filelist/proplist 会暴露：
- 哪些系统文件发生过热更新
- 热更新文件的完整路径
- 哪些系统属性被修改过

如果这些属于敏感信息，建议通过 `system_server` 的 Binder 接口做受控暴露，而非直接给 `appdomain` 文件级读权限。

### 7.3 第三层：neverallow 编译期硬锁

这是"**务必保证代码的可靠与健壮**"的点睛之笔——它不仅限制运行时，更**阻止任何人（包括未来的维护者）在 sepolicy 中误加写权限**，任何违规都会导致 AOSP 编译失败：

```te
# 除 init 外，任何域不得修改/删除/重命名 public 文件
neverallow { domain -init } hotupdate_public_file:file {
    write append create unlink rename setattr relabelfrom
};

# 除 init 外，任何域不得在目录中增删条目（这是防删除的关键）
neverallow { domain -init } hotupdate_dir:dir {
    write add_name remove_name rename reparent rmdir relabelfrom
};

# 除 init 外，任何域不得触碰 private 文件
neverallow { domain -init } hotupdate_private_file:file *;
```

### 7.4 三层防线协同总结

```
攻击面              DAC 层              SELinux 层            neverallow 层
─────────────────────────────────────────────────────────────────────────────
读文件内容     0444 = 所有人可读    allow ... :file read    （放行）
改文件内容     0444 = 非 owner     不授予 write/append     neverallow 锁死
               不可写              → 拒绝                 → 编译不过
删文件         目录 0711 = 非     不授予 dir remove_name  neverallow 锁死
               owner 不可写        → 拒绝                 → 编译不过
创建文件       目录 0711 = 非     不授予 dir add_name     neverallow 锁死
               owner 不可写        → 拒绝                 → 编译不过
```

> **特别提醒**：root 进程持有 `CAP_DAC_OVERRIDE`，DAC 层拦不住它。但 SELinux 对 root 同样强制执行（除非域是 permissive）。neverallow 更是在编译阶段就把"SELinux 放行"这个可能性堵死。三层中，SELinux + neverallow 才是真正的防线。

## 八、init.rc 集成

### 8.1 init.rc 配置

```rc
on post-fs-data
    # 创建目录并恢复 SELinux 标签
    mkdir /data/hotupdate 0711 root root
    restorecon /data/hotupdate

    # 文件首次由写接口自动创建（含表头），此处仅确保标签正确
    # 启动后修正所有文件的上下文
    restorecon_recursive /data/hotupdate
```

### 8.2 文件生命周期

```
首次开机:
  post-fs-data → mkdir /data/hotupdate(0711)
  init 调用 hotupdatefiles() → 发现 filelist 不存在
    → EnsureDir() 目录已存在，跳过
    → ReadEntries() 文件不存在 → 空列表
    → 写 filelist.tmp（含表头 + 新记录）→ fchmod 0444 → fsync → rename
    → restorecon /data/hotupdate/filelist
    → 文件就绪

后续调用（追加记录）:
  init 调用 hotupdatefiles()
    → flock(LOCK_EX)
    → ReadEntries() 解析现有文件
    → 判断 (key,value) 是否已存在
      - 存在 → 解锁返回 0（幂等）
      - 不存在 → 追加到列表 → 全量写 tmp → rename
    → 解锁返回 0

普通进程调用 gethotupdatefiles():
    → open(filelist, O_RDONLY) — 0444 允许
    → read 全量
    → 解析，返回第一列列表
```

## 九、并发模型

| 场景 | 保证机制 |
|---|---|
| 单一写者多次调用 | `flock(LOCK_EX)` 串行化 |
| 多个写者并发（未来） | `flock(LOCK_EX)` 互斥，后者等待 |
| 读者与写者并发 | rename 原子替换：读者要么看到完整的旧文件，要么看到完整的新文件，绝不看到半行 |
| 写过程中崩溃 | tmp 文件残留，下次写入时 `unlink(tmp)` 清理；原文件完好 |
| 掉电 | 原文件 + tmp 文件可能同时存在，rename 在文件系统层面是原子的（ext4/f2fs 均保证），不会出现文件损坏 |

**关键设计优势**：因为使用 rename 原子替换，读接口*完全不需要加锁*。这与 flock 建议性锁不同——如果某个读者不遵守 flock 协议，它仍然安全。这个特性对于"任意进程都可能链接 libhotupdate_client"的场景至关重要。

## 十、健壮性清单

| 措施 | 实现 |
|---|---|
| 输入校验 | 禁止 `\t\n\r\0` 和控制字符；文件名禁止 `/` 和空格；路径必须是绝对路径且在白名单内；propname 仅允许 `[a-zA-Z0-9_.-]` |
| 幂等性 | 重复 (key, value) 组合返回 0，不产生重复行 |
| 崩溃安全 | 临时文件 + fsync + rename + fsync 目录，四步保证原子性 |
| 防缓冲区溢出 | 所有字符串长度硬上限，文件总大小 1MB 上限，记录数 4096 上限 |
| 符号链接防护 | `O_NOFOLLOW` + 创建前 `lstat` 检查 |
| 畸形数据容错 | 读接口跳过非法行（含尾部半行），绝不让调用方崩溃 |
| 目录自动修复 | 每次写调用兜底确保目录存在且权限正确 |
| 临时文件清理 | 写前 `unlink(tmp)` 清理残留；init.rc 启动时 `rm *.tmp` |
| SELinux 标签一致性 | rename 后立即 `selinux_android_restorecon()`，确保标签不因文件来源改变 |
| 依赖最小化 | Writer 只依赖 libc + liblog + libselinux；Reader 只依赖 libc + liblog |

## 十二、方案总结

| 层次 | 决策 |
|---|---|
| **文件格式** | TSV，`<type>\t<key>\t<value>`，`#` 注释，UTF-8 编码 |
| **写接口** | `libhotupdate_writer.a` 静态库链入 init，原子 rename + flock + fsync |
| **读接口** | `libhotupdate_client.so` 动态库，rename 原子性保证零锁读取 |
| **DAC 权限** | 目录 `0711 root root`，文件 `0444 root root`，lock `0600 root root` |
| **SELinux 类型** | `hotupdate_dir` / `hotupdate_public_file`(含 `mlstrustedobject`) / `hotupdate_private_file` |
| **SELinux 授权** | Reader 白名单 attribute；Writer 仅 init |
| **编译期硬锁** | `neverallow { domain -init }` 禁止所有非 init 域的写/删/改名操作 |
| **健壮性** | 全字段校验 + 幂等去重 + 原子写入 + 畸形容错 + 大小/数量硬上限 + `O_NOFOLLOW` |

这套方案综合了四份独立 AI 方案的核心优势，在满足"人类可读明文"需求的前提下，通过 TSV + 原子 rename + DAC 文件权限 + SELinux 标签管控 + neverallow 编译期锁死的五层设计，实现了"任意进程可读、仅 init 可写、任何进程不能删"的安全目标，代码在每个异常路径上都有明确的错误处理和错误码返回，达到了系统关键接口所需的可靠性与健壮性要求。
