# Android 12 热更新文件记录接口 —— 综合设计方案

> 本方案综合四份独立 AI 方案的核心优势，面向 Android 12 系统级热更新场景，对接口设计、
> 文件格式、并发安全、DAC/SELinux 双层权限管控给出可落地的完整设计。

---

## 一、TSV 还是 SQLite？

### 结论：使用 TSV（Tab-Separated Values）明文文件

| 维度 | TSV 明文 | SQLite |
|---|---|---|
| 人类可读 | ✅ `cat`/`grep`/`sed` 直接操作 | ❌ 二进制，需 sqlite3 工具（user 版本通常缺失） |
| init 依赖 | ✅ 纯 POSIX，零额外依赖 | ❌ 需链接 libsqlite，增大 init 攻击面 |
| 崩溃恢复 | ✅ 临时文件 + fsync + rename，等价原子事务 | ✅ WAL/journal 自带 |
| SELinux 管控 | ✅ 仅需管控 2~3 个文件 | ❌ 还需管控 -wal、-shm、-journal 等辅助文件 |
| 并发写 | ⚠️ 需 flock 配合 | ✅ 内置事务 |
| 数据模型匹配 | ✅ 简单 KV，<100 条记录 | ❌ 索引、SQL、事务能力完全用不上 |

**核心判断逻辑：**

1. **需求硬约束**："文件需要使用人类可识读的明文"——SQLite 是二进制格式，直接出局。
2. **运行阶段约束**：写接口调用方是 `init`（PID 1），它运行在开机极早期，依赖最小化是硬道理。引入 SQLite 意味着 init 需要处理 db 锁、WAL 日志、页缓存损坏恢复——这是明显的可靠性负担。
3. **数据特征**：两列字符串、几十条记录、单写者多读者、查询模式只有"全量读取"和"追加去重"。这正是 TSV 的甜点区。

**最终裁决：TSV 明文文件 + 原子 rename 写入 + flock 并发控制。**

## 二、文件格式设计

### 2.1 `/data/hotupdate/filelist`

第一列 `filename` 为**相对路径**（可含 `/`，如 `lib/libfoo.so`、`framework/framework.jar`），
用于在 `/data/hotupdate/para/` 下定位热更新文件的实际存放位置。

```
# hotupdate file list v1
# filename<TAB>filepath
lib/libfoo.so	/system/lib64/libfoo.so
framework/framework.jar	/system/framework/framework.jar
app/MyApp/MyApp.apk	/system/app/MyApp/MyApp.apk
```

### 2.2 `/data/hotupdate/proplist`

```
# hotupdate prop list v1
# propname<TAB>proppath
persist.sys.hotupdate.enable	/data/local.prop
ro.vendor.build.fingerprint	/vendor/build.prop
persist.vendor.hotupdate.version	/vendor/default.prop
```

### 2.3 格式约定

| 规则 | 说明 |
|---|---|
| 分隔符 | 单个 `\t`（Tab）。选择 Tab 而非空格，因为文件路径中允许空格，而 Tab 可在输入校验中无条件禁止 |
| 行尾 | `\n`（LF） |
| 注释 | `#` 开头的行为注释/表头行，读写均忽略 |
| 编码 | UTF-8 |
| 唯一性 | 整行 `filename\tfilepath` 或 `propname\tproppath` 组合唯一，重复写入幂等返回 0 |
| 文件区分 | `filelist` 记录文件热更新，`proplist` 记录属性热更新——两种记录存放在不同文件中，无需类型前缀字段 |

### 2.4 字段合法性约束

所有字段**禁止**包含：`\0`、`\t`、`\n`、`\r`，以及所有 ASCII 控制字符（`0x00~0x1F`、`0x7F`）。

| 字段 | 长度 | 额外约束 |
|---|---|---|
| `filename` | ≤ 255 | **相对路径**，可含 `/`，不能以 `/` 开头，不能含 `..`、空格，不能含 `\` |
| `filepath` | < PATH_MAX | 必须以 `/` 开头（绝对路径），不能含 `..`，限制在白名单路径前缀 |
| `propname` | ≤ 128 | 仅允许 `[a-zA-Z0-9_.-]` |
| `proppath` | < PATH_MAX | 必须以 `/` 开头，不能含 `..`，建议以 `.prop` 结尾 |

**路径白名单**（filepath/proppath 统一适用）：

```
/system/   /system_ext/   /product/   /vendor/   /odm/   /data/
```

## 三、TSV vs DB 深度对比（补充视角）

### 3.1 原子性实现对比

Fable 方案采用了"**写临时文件 → fsync → rename → fsync 目录**"策略，这从根本上保证了崩溃一致性：

- **write 到一半崩溃**：临时文件残留，`filelist` 完好无损，下次 init 调用时 `unlink` 清理 tmp 即可。
- **rename 前崩溃**：同上。
- **rename 后、目录 fsync 前崩溃**：rename 是文件系统元数据操作，ext4/f2fs 保证 rename 本身是原子的——目录项要么指向旧 inode，要么指向新 inode，不存在"指到一半"的中间态。目录 fsync 失败仅意味着 rename 可能未落盘，不会导致文件损坏。

相比之下，SQLite 的事务原子性依赖 `fsync` WAL/journal 文件，在 Android 的 FBE（文件级加密）环境下，init 阶段 `/data` 刚挂载、密钥可能尚未就绪时，SQLite 的 mmap 和文件操作行为更加不可预测。

### 3.2 读写并发

使用 rename 原子替换后，**读接口不需要加锁**——任何时候 `open` 打开的都是一个完整的旧文件或完整的新文件，绝不会读到"写到一半"的行。这与 SQLite 的 WAL 模式（reader 读旧版本、writer 写 WAL）在并发语义上是等价的，但实现简单了一个数量级。

唯一的并发风险是**多个写者同时写**——使用独立的 `.lock` 文件 + `flock(LOCK_EX)` 解决。当前场景只有 init 写，锁主要用于防御未来可能的并发写入者。

### 5.3 公共实现 `hotupdate_common.cpp`（核心逻辑）

```cpp
#define LOG_TAG "hotupdate"

#include "hotupdate_common.h"

#include <errno.h>
#include <fcntl.h>
#include <string.h>
#include <sys/file.h>
#include <sys/stat.h>
#include <unistd.h>

#include <cctype>
#include <set>
#include <string>
#include <vector>

#include <log/log.h>

namespace android {
namespace hotupdate {
namespace internal {

// ---- 字段校验 ----

static bool HasForbiddenChars(const std::string& s) {
    for (unsigned char c : s) {
        if (c == '\0' || c == '\n' || c == '\r' || c == '\t') return true;
        if (c < 0x20 || c == 0x7F) return true;  // 所有控制字符
    }
    return false;
}

static bool HasSpace(const std::string& s) {
    return s.find(' ') != std::string::npos;
}

bool IsValidFileName(const std::string& name) {
    if (name.empty() || name.size() > kMaxFileNameLen) return false;
    if (HasForbiddenChars(name)) return false;
    if (HasSpace(name)) return false;
    if (name[0] == '#') return false;   // 避免与注释冲突
    if (name[0] == '/')  return false;  // 相对路径不能以 / 开头
    if (name.back() == '/') return false; // 不能以 / 结尾
    if (name.find("..") != std::string::npos) return false;  // 禁止路径穿越
    if (name.find('\\') != std::string::npos) return false;  // 禁止反斜杠
    // 禁止连续斜杠
    if (name.find("//") != std::string::npos) return false;
    return true;
}

bool IsValidPropName(const std::string& name) {
    if (name.empty() || name.size() > kMaxPropNameLen) return false;
    if (HasForbiddenChars(name)) return false;
    for (char c : name) {
        if (!(std::isalnum(static_cast<unsigned char>(c)) ||
              c == '_' || c == '.' || c == '-')) {
            return false;
        }
    }
    return true;
}

static bool IsAllowedPathPrefix(const std::string& path) {
    static const char* kPrefixes[] = {
        "/system/", "/system_ext/", "/product/",
        "/vendor/", "/odm/",       "/data/",
    };
    for (const char* p : kPrefixes) {
        if (path.compare(0, strlen(p), p) == 0) return true;
    }
    return false;
}

static bool IsValidAbsolutePath(const std::string& path) {
    if (path.empty() || path.size() >= kMaxPathLen) return false;
    if (path[0] != '/') return false;
    if (HasForbiddenChars(path)) return false;
    if (HasSpace(path)) return false;
    if (path.find("/../") != std::string::npos) return false;
    if (path.size() >= 3 &&
        path.compare(path.size() - 3, 3, "/..") == 0) return false;
    if (!IsAllowedPathPrefix(path)) return false;
    return true;
}

bool IsValidPath(const std::string& path) {
    return IsValidAbsolutePath(path);
}

bool IsValidPropPath(const std::string& path) {
    if (!IsValidAbsolutePath(path)) return false;
    // 必须以 .prop 结尾（兼容 local.prop / build.prop / default.prop）
    if (path.size() < 5) return false;
    if (path.compare(path.size() - 5, 5, ".prop") != 0) return false;
    return true;
}

// ---- TSV 构造与解析 ----

std::string BuildTsvRecord(const std::string& key,
                           const std::string& value) {
    std::string line;
    line.reserve(key.size() + 1 + value.size() + 1);
    line.append(key);
    line.push_back('\t');
    line.append(value);
    line.push_back('\n');
    return line;
}

bool ParseTsvLine(const std::string& line,
                  std::string* key,
                  std::string* value) {
    if (line.empty() || line[0] == '#') return false;

    size_t tab = line.find('\t');
    if (tab == std::string::npos || tab == 0) return false;

    // 确保只有两列（不能有第二个 tab）
    if (line.find('\t', tab + 1) != std::string::npos) return false;

    std::string k = line.substr(0, tab);
    std::string v = line.substr(tab + 1);

    if (k.empty() || v.empty()) return false;
    if (HasForbiddenChars(k) || HasForbiddenChars(v)) return false;

    if (key) *key = std::move(k);
    if (value) *value = std::move(v);
    return true;
}

// ---- 文件读写工具 ----

bool ReadFileContent(const char* path, std::string* out) {
    out->clear();
    int fd = TEMP_FAILURE_RETRY(
        open(path, O_RDONLY | O_CLOEXEC | O_NOFOLLOW));
    if (fd < 0) {
        if (errno == ENOENT) return true;  // 不存在即空
        ALOGE("open %s for read: %s", path, strerror(errno));
        return false;
    }
    char buf[4096];
    ssize_t n;
    while ((n = TEMP_FAILURE_RETRY(read(fd, buf, sizeof(buf)))) > 0) {
        out->append(buf, static_cast<size_t>(n));
        if (out->size() > kMaxListFileSize) {
            ALOGE("%s exceeds size limit", path);
            close(fd);
            return false;
        }
    }
    close(fd);
    return n == 0;
}

bool WriteFully(int fd, const char* data, size_t len) {
    while (len > 0) {
        ssize_t n = TEMP_FAILURE_RETRY(write(fd, data, len));
        if (n <= 0) return false;
        data += n;
        len -= static_cast<size_t>(n);
    }
    return true;
}

// === 读取并解析列表文件，返回去重后的 (key, value) 对列表 ===
std::vector<std::pair<std::string, std::string>>
ReadEntries(const char* path) {
    std::vector<std::pair<std::string, std::string>> result;
    std::set<std::string> seen;  // 按 key 去重，保留首次出现顺序

    std::string content;
    if (!ReadFileContent(path, &content)) return result;

    size_t pos = 0;
    while (pos < content.size() && result.size() < kMaxRecords) {
        size_t eol = content.find('\n', pos);
        std::string line = (eol == std::string::npos)
            ? content.substr(pos)
            : content.substr(pos, eol - pos);
        pos = (eol == std::string::npos) ? content.size() : eol + 1;

        // 去掉 \r
        if (!line.empty() && line.back() == '\r') line.pop_back();
        if (line.empty() || line[0] == '#') continue;

        std::string key, value;
        if (!ParseTsvLine(line, &key, &value)) {
            ALOGW("skip malformed line in %s", path);
            continue;
        }
        if (seen.insert(key).second) {
            result.emplace_back(std::move(key), std::move(value));
        }
    }
    return result;
}

}  // namespace internal
}  // namespace hotupdate
}  // namespace android
```

### 5.5 写实现 `hotupdate_writer.cpp`（累积到 .tmp + NotifyEndUpdate 双目录原子提交）

```cpp

#include "hotupdate_common.h"

#include <string.h>

namespace android {
namespace hotupdate {
namespace {

// tmp 文件的目标 SELinux 上下文（必须与 file.te / file_contexts 一致）。
// 通过 setfscreatecon 预置，使 tmp 一出生即为 public，rename 后无需二次修标签。
constexpr const char* kHotUpdatePublicContext =
    "u:object_r:hotupdate_public_file:s0";

using internal::kHotUpdateDir;
using internal::kFileListPath;
using internal::kPropListPath;
using internal::kLockPath;
using internal::kFileListHeader;
using internal::kPropListHeader;
using internal::kListFileMode;
using internal::kDirMode;
using internal::kVdrHotUpdateDir;
using internal::kVdrFileListPath;
using internal::kVdrPropListPath;
using internal::BuildTsvRecord;
using internal::ReadFileContent;
using internal::WriteFully;
using internal::ParseTsvLine;
using internal::ReadEntries;

// === 确保目录存在（可复用于 /data/hotupdate 与 /data/Vdr/hotupdate）===
int EnsureDir(const char* dir) {
    struct stat st{};
    if (stat(dir, &st) == 0) {
        if (!S_ISDIR(st.st_mode)) {
            ALOGE("%s exists but is not a directory", dir);
            return HOTUPDATE_ERR_MKDIR_FAILED;
        }
        chmod(dir, kDirMode);
        return HOTUPDATE_OK;
    }
    if (mkdir(dir, kDirMode) != 0 && errno != EEXIST) {
        ALOGE("mkdir %s failed: %s", dir, strerror(errno));
        return HOTUPDATE_ERR_MKDIR_FAILED;
    }
    chmod(dir, kDirMode);
    return HOTUPDATE_OK;
}

// 本进程「已重置过」的正式路径集合。init 是单进程跑完整个写入循环：
// 每个列表在本进程首次写入时清空残留 .tmp（丢弃上一轮在 NotifyEndUpdate 之前
// 崩溃遗留的累积内容），之后同一进程内持续累积；NotifyEndUpdate 提交后清空。
// 全程在 kLockPath 排他锁保护下访问，无需额外同步。
std::set<std::string> g_session_tmp_paths;

// === fsync 指定目录，确保其下的 rename/create 落盘 ===
void FsyncDir(const char* dir) {
    int dir_fd = TEMP_FAILURE_RETRY(
        open(dir, O_RDONLY | O_DIRECTORY | O_CLOEXEC));
    if (dir_fd >= 0) {
        fsync(dir_fd);
        close(dir_fd);
    }
}

// === 原子写入 <final_path>.tmp：写 .tmp.new → fchmod → fsync → rename 覆盖 .tmp ===
// 保证累积 tmp 与镜像 tmp 自身永不半写；tmp 一创建即经 setfscreatecon 带 public
// 标签，后续 CommitTmpToFinal 的 rename 使正式文件天然为 public（消除标签窗口）。
// 目录 fsync 与「.tmp → 正式文件」的提交均推迟到 NotifyEndUpdate 统一进行。
int WriteListTmp(const char* final_path, const std::string& content) {
    std::string tmp_path = std::string(final_path) + ".tmp";
    std::string new_path = tmp_path + ".new";
    unlink(new_path.c_str());  // 清理残留

    setfscreatecon(kHotUpdatePublicContext);
    int fd = TEMP_FAILURE_RETRY(
        open(new_path.c_str(),
             O_WRONLY | O_CREAT | O_EXCL | O_CLOEXEC | O_NOFOLLOW,
             0644));
    setfscreatecon(nullptr);
    if (fd < 0) {
        ALOGE("open %s failed: %s", new_path.c_str(), strerror(errno));
        return HOTUPDATE_ERR_OPEN_FAILED;
    }

    if (!WriteFully(fd, content.data(), content.size())) {
        ALOGE("write %s failed", new_path.c_str());
        close(fd);
        unlink(new_path.c_str());
        return HOTUPDATE_ERR_WRITE_FAILED;
    }
    if (fchmod(fd, kListFileMode) != 0) {
        ALOGE("fchmod %s failed: %s", new_path.c_str(), strerror(errno));
        close(fd);
        unlink(new_path.c_str());
        return HOTUPDATE_ERR_CHMOD_FAILED;
    }
    if (fsync(fd) != 0) {
        ALOGE("fsync %s failed: %s", new_path.c_str(), strerror(errno));
        close(fd);
        unlink(new_path.c_str());
        return HOTUPDATE_ERR_FSYNC_FAILED;
    }
    close(fd);

    // 同目录原子替换 .tmp
    if (rename(new_path.c_str(), tmp_path.c_str()) != 0) {
        ALOGE("rename %s -> %s failed: %s",
              new_path.c_str(), tmp_path.c_str(), strerror(errno));
        unlink(new_path.c_str());
        return HOTUPDATE_ERR_RENAME_FAILED;
    }
    return HOTUPDATE_OK;
}

// === 提交：rename <final_path>.tmp → final_path，并防御性二次校正 SELinux 标签 ===
// 目录 fsync 由调用方批量收尾，以尽量缩短主/镜像两目录的不一致窗口。
int CommitTmpToFinal(const char* final_path) {
    std::string tmp_path = std::string(final_path) + ".tmp";
    if (rename(tmp_path.c_str(), final_path) != 0) {
        ALOGE("commit rename %s -> %s failed: %s",
              tmp_path.c_str(), final_path, strerror(errno));
        return HOTUPDATE_ERR_RENAME_FAILED;
    }
    // 因 tmp 已 setfscreatecon 预置 public，正常情况已正确，此步为冗余保险；
    // 仍检查返回值并记录，绝不静默忽略失败。
    if (selinux_android_restorecon(final_path, 0) != 0) {
        ALOGW("restorecon %s failed: %s (label should already be public via "
              "setfscreatecon)", final_path, strerror(errno));
    }
    return HOTUPDATE_OK;
}

// === 累积写入：读当前 .tmp → 按 key 去重合并 → 原子重写 .tmp（不提交正式文件）===
// 与旧版逐条 rename 到正式文件不同：本函数只把记录累积到 <path>.tmp，
// 正式文件的发布统一推迟到 NotifyEndUpdate。init 在开机循环里多次调用本函数，
// 全部写完后再调一次 NotifyEndUpdate 把整批记录一次性对外可见。
int AtomicAppendRecord(const char* path,
                       const char* header,
                       const std::string& key,
                       const std::string& value) {
    int rc;

    // 1. 确保目录存在
    rc = EnsureDir(kHotUpdateDir);
    if (rc != HOTUPDATE_OK) return rc;

    // 2. 排他锁（与其它写入及 NotifyEndUpdate 提交互斥）
    int lock_fd = TEMP_FAILURE_RETRY(
        open(kLockPath, O_RDWR | O_CREAT | O_CLOEXEC, 0600));
    if (lock_fd < 0) {
        ALOGE("open lock file failed: %s", strerror(errno));
        return HOTUPDATE_ERR_OPEN_FAILED;
    }
    if (flock(lock_fd, LOCK_EX) != 0) {
        ALOGE("flock failed: %s", strerror(errno));
        close(lock_fd);
        return HOTUPDATE_ERR_LOCK_FAILED;
    }

    std::string tmp_path = std::string(path) + ".tmp";

    // 3. 本进程首次写入该列表：清空残留 .tmp / .tmp.new，保证本轮从空开始
    //    （丢弃上一轮在 NotifyEndUpdate 之前崩溃残留的累积内容）。
    if (g_session_tmp_paths.find(path) == g_session_tmp_paths.end()) {
        unlink(tmp_path.c_str());
        unlink((tmp_path + ".new").c_str());
        g_session_tmp_paths.insert(path);
    }

    // 4. 读取当前累积记录（累积源为 .tmp；ReadEntries 已按 key 去重）
    auto entries = ReadEntries(tmp_path.c_str());

    // 5. 按 key 判重（key 是唯一主键：filename 相对路径唯一映射到 para/ 下的
    //    实体文件，propname 唯一）：
    //    - 同 key 同 value → 幂等成功，无需重写；
    //    - 同 key 不同 value → 覆盖为最新 value（视为一次“更新”）；
    //    - key 不存在 → 追加。
    //    注意：不能允许同一 key 出现多条不同 value，否则 GetHotUpdateFile 拼出的
    //    /data/hotupdate/para/<key> 会产生歧义，且 reader 与文件内容不一致。
    bool matched = false;
    for (auto& e : entries) {
        if (e.first == key) {
            if (e.second == value) {          // 完全相同 → 幂等成功
                flock(lock_fd, LOCK_UN);
                close(lock_fd);
                return HOTUPDATE_OK;
            }
            e.second = value;                 // 同名不同路径 → 覆盖为最新
            matched = true;
            break;
        }
    }
    if (!matched) entries.emplace_back(key, value);

    // 6. 按 key 排序，保证文件行序稳定（key 已唯一，无需再 unique）
    std::sort(entries.begin(), entries.end());

    // 7. 构造新内容
    std::string new_content(header);
    for (const auto& e : entries) {
        new_content += BuildTsvRecord(e.first, e.second);
    }

    // 8. 原子重写累积 .tmp（不 rename 到正式文件、不做正式文件 restorecon/目录 fsync，
    //    这些统一推迟到 NotifyEndUpdate 提交）
    rc = WriteListTmp(path, new_content);

    flock(lock_fd, LOCK_UN);
    close(lock_fd);
    return rc;
}

}  // namespace

int hotupdatefiles(const std::string& filename, const std::string& filepath) {
    if (!internal::IsValidFileName(filename) ||
        !internal::IsValidPath(filepath)) {
        ALOGE("hotupdatefiles: invalid argument");
        return HOTUPDATE_ERR_INVALID_ARG;
    }
    return AtomicAppendRecord(kFileListPath, kFileListHeader,
                              filename, filepath);
}

int hotupdateprops(const std::string& propname, const std::string& proppath) {
    if (!internal::IsValidPropName(propname) ||
        !internal::IsValidPropPath(proppath)) {
        ALOGE("hotupdateprops: invalid argument");
        return HOTUPDATE_ERR_INVALID_ARG;
    }
    return AtomicAppendRecord(kPropListPath, kPropListHeader,
                              propname, proppath);
}

// === 提交本轮累积：复制到 Vdr 镜像 tmp → 集中 rename 发布主/镜像两处正式文件 ===
void NotifyEndUpdate(void) {
    // 与写入共用同一把排他锁，保证「累积 → 提交」串行、互斥
    int lock_fd = TEMP_FAILURE_RETRY(
        open(kLockPath, O_RDWR | O_CREAT | O_CLOEXEC, 0600));
    if (lock_fd < 0) {
        ALOGE("NotifyEndUpdate: open lock file failed: %s", strerror(errno));
        return;
    }
    if (flock(lock_fd, LOCK_EX) != 0) {
        ALOGE("NotifyEndUpdate: flock failed: %s", strerror(errno));
        close(lock_fd);
        return;
    }

    // 1. 确保主/镜像两个目录存在
    if (EnsureDir(kHotUpdateDir) != HOTUPDATE_OK ||
        EnsureDir(kVdrHotUpdateDir) != HOTUPDATE_OK) {
        ALOGE("NotifyEndUpdate: ensure dir failed, abort commit");
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return;
    }

    struct ListPair {
        const char* src_final;   // /data/hotupdate/xxx
        const char* vdr_final;   // /data/Vdr/hotupdate/xxx
    };
    const ListPair lists[] = {
        {kFileListPath, kVdrFileListPath},
        {kPropListPath, kVdrPropListPath},
    };

    // 2. 逐个列表：把 /data/hotupdate/xxx.tmp 复制成 /data/Vdr/hotupdate/xxx.tmp。
    //    先把「所有」镜像 tmp 准备并落盘就绪，再集中提交，缩短双目录不一致窗口。
    std::vector<const ListPair*> ready;   // 本轮可提交的列表
    for (const ListPair& lp : lists) {
        std::string src_tmp = std::string(lp.src_final) + ".tmp";
        if (access(src_tmp.c_str(), F_OK) != 0) {
            ALOGW("NotifyEndUpdate: %s not present, skip this list",
                  src_tmp.c_str());
            continue;   // 本轮没写该列表，保留其原正式文件不动
        }
        std::string content;
        if (!ReadFileContent(src_tmp.c_str(), &content)) {
            ALOGE("NotifyEndUpdate: read %s failed, skip this list",
                  src_tmp.c_str());
            continue;
        }
        int rc = WriteListTmp(lp.vdr_final, content);
        if (rc != HOTUPDATE_OK) {
            ALOGE("NotifyEndUpdate: prepare vdr tmp for %s failed: %d",
                  lp.vdr_final, rc);
            continue;   // 镜像准备失败则本列表整体不提交，避免半发布
        }
        ready.push_back(&lp);
    }

    // 3. 集中收尾：对每个就绪列表，连续提交主/镜像两处正式文件。
    //    任一 rename 失败仅记 ALOGE 并继续，尽量保证两目录一致。
    for (const ListPair* lp : ready) {
        if (CommitTmpToFinal(lp->src_final) != HOTUPDATE_OK) {
            ALOGE("NotifyEndUpdate: commit %s failed", lp->src_final);
        }
        if (CommitTmpToFinal(lp->vdr_final) != HOTUPDATE_OK) {
            ALOGE("NotifyEndUpdate: commit %s failed", lp->vdr_final);
        }
    }

    // 4. fsync 两个目录，确保 rename 落盘
    FsyncDir(kHotUpdateDir);
    FsyncDir(kVdrHotUpdateDir);

    // 5. 本轮提交完成，清空会话集合，下一轮从空重新累积
    g_session_tmp_paths.clear();

    flock(lock_fd, LOCK_UN);
    close(lock_fd);
}

}  // namespace hotupdate
}  // namespace android
```

#define LOG_TAG "hotupdate"

#include "hotupdate_client.h"
#include "hotupdate_common.h"

#include <cerrno>
#include <cstring>
#include <string>
#include <vector>

#include <log/log.h>

namespace android {
namespace hotupdate {
namespace {

constexpr char kParaPrefix[] = "/data/hotupdate/para/";

std::vector<std::string> ReadFirstColumn(const char* path) {
    std::vector<std::string> result;
    auto entries = internal::ReadEntries(path);
    result.reserve(entries.size());
    for (const auto& e : entries) {
        result.push_back(e.first);
    }
    return result;
}

}  // namespace

std::vector<std::string> gethotupdatefiles(void) {
    return ReadFirstColumn(internal::kFileListPath);
}

std::vector<std::string> gethotupdateprops(void) {
    return ReadFirstColumn(internal::kPropListPath);
}

char* GetHotUpdateFile(const char* pathSuffix, int type, char* buf, int bufLen) {
    // 1. 参数校验
    if (pathSuffix == nullptr || buf == nullptr || bufLen <= 0) {
        ALOGE("GetHotUpdateFile: invalid argument (NULL pointer or bufLen=%d)", bufLen);
        return nullptr;
    }

    // 2. 根据 type 选择查找目标 + 对应的 key 合法性校验规则
    //    （type=0 的 key 是相对路径 filename，type=1 的 key 是 propname，
    //     两者字符集不同，必须分别用对应校验函数）
    std::string suffix(pathSuffix);
    const char* listPath = nullptr;
    if (type == 0) {
        listPath = internal::kFileListPath;
        if (!internal::IsValidFileName(suffix)) {
            ALOGE("GetHotUpdateFile: filename '%s' invalid", pathSuffix);
            return nullptr;
        }
    } else if (type == 1) {
        listPath = internal::kPropListPath;
        if (!internal::IsValidPropName(suffix)) {
            ALOGE("GetHotUpdateFile: propname '%s' invalid", pathSuffix);
            return nullptr;
        }
    } else {
        ALOGE("GetHotUpdateFile: unsupported type=%d (only 0=filelist, 1=proplist)", type);
        return nullptr;
    }

    // 3. 打开并解析 list 文件，逐行匹配
    std::string content;
    if (!internal::ReadFileContent(listPath, &content)) {
        ALOGE("GetHotUpdateFile: failed to read %s", listPath);
        return nullptr;
    }

    std::string matchedKey;
    size_t pos = 0;
    while (pos < content.size()) {
        size_t eol = content.find('\n', pos);
        std::string line = (eol == std::string::npos)
            ? content.substr(pos)
            : content.substr(pos, eol - pos);
        pos = (eol == std::string::npos) ? content.size() : eol + 1;

        if (!line.empty() && line.back() == '\r') line.pop_back();
        if (line.empty() || line[0] == '#') continue;

        std::string key, value;
        if (!internal::ParseTsvLine(line, &key, &value)) {
            continue;  // 畸形行跳过
        }

        if (key == suffix) {
            matchedKey = std::move(key);
            break;  // 找到第一个匹配即停止
        }
    }

    if (matchedKey.empty()) {
        ALOGW("GetHotUpdateFile: pathSuffix '%s' not found in %s", pathSuffix, listPath);
        return nullptr;
    }

    // 4. 拼接绝对路径: "/data/hotupdate/para/" + matchedKey
    std::string fullPath = kParaPrefix + matchedKey;

    // 5. 边界检查：buf 必须能容纳完整路径 + '\0'
    if (static_cast<int>(fullPath.size()) >= bufLen) {
        ALOGE("GetHotUpdateFile: bufLen=%d insufficient for path '%s' (need %zu)",
              bufLen, fullPath.c_str(), fullPath.size() + 1);
        return nullptr;
    }

    // 6. 安全写入 buf
    std::memcpy(buf, fullPath.c_str(), fullPath.size() + 1);  // +1 包含 '\0'
    return buf;
}

}  // namespace android

### 5.8 关键设计决策说明

**为什么采用"全量重写 + rename"而非"追加写"?**

| 维度 | 追加写 | 全量重写 + rename（本方案） |
|---|---|---|
| 崩溃安全 | 追加一半崩溃 = 文件末尾有半行 | 写 tmp 失败不影响原文件 |
| 读者一致性 | 需读者也 flock(SH) 才能保证不读到半行 | 无需锁，rename 原子替换 |
| 文件增长 | 只增不减 | 每次全量重写（去重后可能缩小） |
| 大文件性能 | O(1) 追加 | O(n) 重写 |
| 适用场景 | 大文件、高频写入 | <100 条记录、低频写入 → 本项目 |

对于热更新场景（几十条记录、低频写入），全量重写的性能开销完全可以忽略，而它带来的崩溃安全性和读端零锁优势是决定性的。

**为什么把「逐条发布」改为「累积到 .tmp + NotifyEndUpdate 统一提交」?**

init 在开机时会在一个循环里多次调用 `hotupdatefiles`/`hotupdateprops`。旧版每次调用都 rename 一次正式文件，等价于把一批记录拆成 N 次对外发布——既产生 N 次冗余 rename，reader 还可能在半途读到「只含前几条」的中间态。新版把写入分成两个阶段：

- **累积期**：循环内每次调用只把当前全量（去重+排序后）原子重写到 `<正式路径>.tmp`（经 `.tmp.new` → rename 覆盖 `.tmp`，累积文件自身永不半写），**不碰正式文件**。key 唯一/排序语义与旧版完全一致。
- **提交期**：`NotifyEndUpdate()` 把 `/data/hotupdate/*.tmp` 复制成 `/data/Vdr/hotupdate/*.tmp`，再集中对 4 个 `.tmp` 连续 rename 成正式文件。整批记录**一次性**对外可见；提交前任何崩溃都不会改动正式文件。

**为什么发布到 `/data/hotupdate` 与 `/data/Vdr/hotupdate` 两处？** 二者是**对等镜像**：同一份 `.tmp` 内容分别落到两个目录，供不同消费方按各自约定读取。提交时先把两份镜像 `.tmp` 都准备并 fsync 就绪，最后集中做 4 次 rename，把两目录的不一致窗口降到最小；即便中途崩溃，两目录内容也同源一致，下一轮完整提交会再次对齐。

**会话如何保证从干净状态开始？** 采用「进程内首次写入自动清空」：`AtomicAppendRecord` 用一个进程级 `std::set` 记录本进程已重置过的正式路径，某列表本进程首次写入时先 `unlink` 残留 `.tmp`/`.tmp.new`。init 是单进程跑完整个循环，若上一轮在 `NotifyEndUpdate` 之前崩溃，下次开机是**新进程**，首写时自动丢弃上轮残留——无需新增 `BeginUpdate()` 之类的公开接口，接口面保持最小。

**为什么 key（第一列）是唯一主键？**

因为 `filename` 已经是相对路径，并被 `GetHotUpdateFile` 唯一映射到 `/data/hotupdate/para/<filename>` 下的实体文件——同一个 `filename` 只能对应一个 para 实体文件，因此 **key 必须唯一**。写接口据此保证：

- 同 `key` 同 `value` → 幂等返回 0，不重写；
- 同 `key` 不同 `value` → 覆盖为最新 `value`（等价于“更新记录”），而**不是**追加出第二行；
- `ReadEntries`、`gethotupdatefiles()`、`GetHotUpdateFile` 全部按 key 唯一语义工作，三者与文件实际内容始终一致。

> ⚠️ 早期设计曾允许“同 filename 多 filepath”，但这与 para 唯一定位方案冲突（两行相同 filename 会争用同一个 para 文件），故本方案统一收敛为 key 唯一。

**关于第二列（`filepath` / `proppath`）的可达性：**

现有读接口 `gethotupdatefiles()` 只返回第一列，`GetHotUpdateFile` 返回 `para/ + 第一列`——**第二列（目标系统安装路径）目前没有任何读接口暴露**。若消费方需要“该热更文件应覆盖到哪个系统路径”，当前签名下取不到。在不改动既有签名/格式的前提下，可后续新增 `vector<pair<string,string>> gethotupdatefilepairs()` 之类的接口暴露完整映射；在此之前，请把“安装目标路径解析”视为本接口范围之外。

**`GetHotUpdateFile` 设计要点：**

| 设计点 | 决策 |
|---|---|
| 匹配方式 | **精确匹配**（`key == pathSuffix`），不做子串或后缀匹配，避免意外命中 |
| 首次命中即停 | 遍历找到第一个匹配行即返回，不关心是否有重复——filelist 中 (key,value) 组合唯一保证不会有歧义 |
| `type` 参数 | `0` = 查 filelist，`1` = 查 proplist，便于同一函数覆盖两种查找场景 |
| 返回值 | C 风格 `char*`：成功返回 `buf`（方便链式调用），失败返回 `NULL` |
| 路径拼接 | 固定前缀 `/data/hotupdate/para/` + filename（filename 是相对路径，直接拼接即得绝对路径） |
| 缓冲区安全 | `bufLen` 严格检查，`fullPath.size() >= bufLen` 时拒绝写入并返回 NULL，绝不截断 |
| 无锁读取 | 依靠 writer 的 rename 原子替换保证内容完整性，reader 无需 flock |

> ⚠️ **`type=1`（proplist）的拼接语义须先确认**：`GetHotUpdateFile(propname, 1)` 返回
> `/data/hotupdate/para/<propname>`。这只有在“**每个属性的热更内容以 propname 为文件名单独存放在 para/ 下**”
> 这一约定成立时才有意义。若你的实现里属性其实是整体写在某个 `.prop` 文件（即 `proplist`
> 第二列 `proppath`）中、并不存在按 propname 命名的单文件，那么 `type=1` 拼出的路径将指向一个
> 不存在的文件。此时应改为让 `type=1` 返回 `proppath` 对应的实体（例如 `para/ + basename(proppath)`），
> 或直接不支持 `type=1`。请在落地前明确 para/ 下属性文件的组织方式，避免该分支产出无效路径。

**为什么 filename 改为相对路径？**

`GetHotUpdateFile` 用 `filename` 拼接 `/data/hotupdate/para/` 前缀来定位实际的热更新文件。这要求 `filename` 是相对路径（如 `lib/libfoo.so`），而非裸文件名。对应地：

- `hotupdatefiles()` 的 `filename` 参数也接受相对路径
- `/data/hotupdate/para/` 目录下按相同层级结构存放热更新文件
- `IsValidFileName()` 放宽为允许 `/`，但禁止以 `/` 开头、禁止 `..`、禁止 `//`

**`/data/hotupdate/para/` 目录设计：**

该目录存放实际的热更新文件，需要在 init.rc 中创建：

```rc
on post-fs-data
    # Android 12 FBE：新建 /data 顶层目录需显式声明加密策略
    mkdir /data/hotupdate 0711 root root encryption=Require
    mkdir /data/hotupdate/para 0755 root root
    restorecon_recursive /data/hotupdate
```

注意 `/data/hotupdate/para/` 使用 `0755`（而非其父目录的 `0711`），因为调用方需要通过 `open`/`stat` 访问目录内的文件——如果 `para` 只有 `x` 权限，虽然可以通过已知完整路径打开文件，但某些调用框架可能需要 `r` 权限列出或校验文件。对于热更新文件目录，`0755` 是合理的折中。

## 七、权限与 SELinux 管控方案

> **核心目标**：任意进程可读、任何进程（除 init）不能写、不能删除、不能改名。

这需要 **DAC（文件权限）+ SELinux（强制访问控制）+ neverallow（编译期锁死）** 三层防线协同。

### 7.1 第一层：DAC 文件权限

```
/data/hotupdate/              0711 root root    ← 目录：可穿越但不可列出内容
/data/hotupdate/para/         0755 root root    ← 热更新文件实际存放目录（需可列出）
/data/hotupdate/filelist      0444 root root    ← 所有人只读
/data/hotupdate/proplist      0444 root root    ← 所有人只读
/data/hotupdate/.lock         0600 root root    ← 仅 root 可读写
```

**关键认知**：文件设为 `0444` 只能防止"改文件内容"，**不能防止"删文件"**。能否删除一个文件取决于其**父目录的写权限**，而非文件本身权限。

| 权限设计 | 效果 |
|---|---|
| 目录 `0711` | owner(root) 可读写执行；group/other **只有 x（可穿越到已知路径）**，无 r（不能 `ls`）无 w（不能增删文件） |
| 文件 `0444` | 所有人只读 |
| `.lock` `0600` | 仅 root 可读写，普通进程看不见也摸不到 |

### 7.2 第二层：SELinux 强制访问控制

#### (a) 定义 SELinux 类型（`file.te`）

```te
# 目录类型
type hotupdate_dir, file_type, data_file_type;

# 公开文件（filelist / proplist）—— 需跨域可读
type hotupdate_public_file, file_type, data_file_type, mlstrustedobject;

# 私有文件（.lock / .tmp）—— 仅 writer 域可访问
type hotupdate_private_file, file_type, data_file_type;
```

**设计要点**：

- `file_type` + `data_file_type` 是 `/data` 下文件的强制性属性，缺失会触发系统 neverallow 编译失败。
- `mlstrustedobject`：**这是关键属性**。Android 普通应用运行在带 MLS category 的安全上下文（如 `c512,c768`），不加此属性，跨 category 的应用读取会被 MLS 规则拦截。加上后等价于声明"此文件的 MLS 类别检查豁免"。
- 拆分 `public_file` 和 `private_file` 实现**最小权限**——普通读者域只授予到 public_file 的读权限，永远看不到 lock 文件和临时文件。

> ⚠️ **若 vendor 域进程也要读（本方案 Android.bp 有 `vendor_available: true`、读者白名单含 `vendor_app`）**，必须额外处理 Treble 的 core/vendor 数据隔离：
> 1. **类型定义要放在 public sepolicy**（`system/sepolicy/public` 或等价位置），vendor 策略才可见、才能 `typeattribute`/`allow` 引用它；放在 private 里 vendor 编不过。
> 2. **`core_data_file_type` 属性要谨慎**：给 `/data` 下 core 侧文件加它会受 Treble 隔离 neverallow 约束，可能禁止 vendor 域访问。本方案类型**未加** `core_data_file_type`，正是为放宽 vendor 访问；但不同分支的平台基线约束不同，**务必在你的 sepolicy 基线上实际编译验证**。
> 3. 如果**不需要** vendor 读，请删掉 `libhotupdate_client` 的 `vendor_available: true` 和读者白名单里的 `vendor_app`，避免留下“能链接却读不到 / 或编译失败”的隐患。

#### (b) 文件标签（`file_contexts`）

```
/data/hotupdate                  u:object_r:hotupdate_dir:s0
/data/hotupdate/para(/.*)?       u:object_r:hotupdate_public_file:s0
/data/hotupdate/filelist         u:object_r:hotupdate_public_file:s0
/data/hotupdate/proplist         u:object_r:hotupdate_public_file:s0
/data/hotupdate/\.lock           u:object_r:hotupdate_private_file:s0
/data/hotupdate/.*\.tmp          u:object_r:hotupdate_private_file:s0

# 第二份对等镜像目录（NotifyEndUpdate 同步发布）
/data/Vdr/hotupdate              u:object_r:hotupdate_dir:s0
/data/Vdr/hotupdate/filelist     u:object_r:hotupdate_public_file:s0
/data/Vdr/hotupdate/proplist     u:object_r:hotupdate_public_file:s0
/data/Vdr/hotupdate/.*\.tmp      u:object_r:hotupdate_private_file:s0
```

> 说明：`.tmp` / `.tmp.new` 在 `file_contexts` 中默认落到 `hotupdate_private_file`，但写入时通过
> `setfscreatecon(hotupdate_public_file)` 强制以 public 上下文创建——因 `rename` **不改标签**，
> 提交后的正式文件天然是 public，readers 可读；`.tmp` 的默认 private 标签仅在有人对 tmp 显式
> `restorecon` 时才生效（本方案不对 tmp 执行 restorecon），二者不冲突。

#### (c) 写者权限（Writer Domain）

以 `init` 域为例（若热更新由独立 service 执行，替换为对应域名）：

```te
# init 对目录有完整管理权限
allow init hotupdate_dir:dir {
    create read write open search getattr setattr
    add_name remove_name
};

# init 对公开文件有完整管理权限
allow init hotupdate_public_file:file {
    create open read write append getattr setattr
    lock rename unlink
};

# init 对私有文件有完整管理权限
allow init hotupdate_private_file:file {
    create open read write append getattr setattr
    lock rename unlink
};
```

> 镜像目录 `/data/Vdr/hotupdate` 已标为 `hotupdate_dir`，上面对 `hotupdate_dir:dir` /
> `hotupdate_public_file:file` / `hotupdate_private_file:file` 的授权已覆盖 Vdr 侧的
> 创建、写入、rename。**但** init 还需能穿越并在其父目录 `/data/Vdr` 下增删条目：若
> `/data/Vdr` 属于不同的 dir 类型（如某 `system_data_file` 或 vendor 专用类型），需另加
> `allow init <vdr_parent_dir_type>:dir { search write add_name remove_name };`——
> 具体类型以你的 `file_contexts`/基线为准，务必编译验证。

#### (d) 读者权限（Reader Domain）—— 推荐 Attribute 白名单模式

```te
# 定义 reader 属性
attribute hotupdate_reader_domain;

# 通过属性统一授权
# reader 依赖 writer 的 rename 原子替换实现免锁，不使用 flock，也不 mmap，
# 因此只授最小读权限（不含 lock/map），符合最小权限原则。
allow hotupdate_reader_domain hotupdate_dir:dir { search getattr };
allow hotupdate_reader_domain hotupdate_public_file:file { open read getattr };

# 将需要读取的域加入白名单
typeattribute system_server hotupdate_reader_domain;
typeattribute platform_app   hotupdate_reader_domain;
typeattribute priv_app       hotupdate_reader_domain;
typeattribute vendor_app     hotupdate_reader_domain;
# ... 按需添加其他 domain
```

**如果确实需要"所有 domain 可读"（理解提示词的"任意进程"）：**

```te
# 【编译硬约束，非安全建议】必须同时排除 -all_untrusted_apps 和 -isolated_app：
#   hotupdate_public_file 带 data_file_type 属性，而 AOSP app_neverallows.te 有一条
#   neverallow 禁止 all_untrusted_apps 打开绝大多数 data_file_type 文件（白名单类型除外）。
#   若不排除 all_untrusted_apps，sepolicy 会直接编译失败——这不是“建议”，是必须。
allow { domain -all_untrusted_apps -isolated_app } hotupdate_dir:dir { search getattr };
allow { domain -all_untrusted_apps -isolated_app } hotupdate_public_file:file { open read getattr };
```

> 注意：上面的写法在编译层面**天然就无法**覆盖三方普通应用（`untrusted_app`）——它们被
> 平台 neverallow 挡住了。若业务确实要让三方 App 读到这些信息，**不能**靠硬加文件级
> `allow`（会编译失败），只能：(a) 把新类型加入平台 neverallow 白名单（需改平台策略，
> 不推荐），或 (b) 通过 `system_server` 的 Binder 接口做受控转发（推荐）。

无论是否放开给三方应用，都要评估信息泄露风险——filelist/proplist 会暴露：
- 哪些系统文件发生过热更新
- 热更新文件的完整路径
- 哪些系统属性被修改过

如果这些属于敏感信息，建议通过 `system_server` 的 Binder 接口做受控暴露，而非直接给 `appdomain` 文件级读权限。

### 7.3 第三层：neverallow 编译期硬锁

这是"**务必保证代码的可靠与健壮**"的点睛之笔——它不仅限制运行时，更**阻止任何人（包括未来的维护者）在 sepolicy 中误加写权限**，任何违规都会导致 AOSP 编译失败：

```te
# 除 init 外，任何域不得修改/删除/重命名 public 文件
neverallow { domain -init } hotupdate_public_file:file {
    write append create unlink rename setattr relabelfrom
};

# 除 init 外，任何域不得在目录中增删条目（这是防删除的关键）
neverallow { domain -init } hotupdate_dir:dir {
    write add_name remove_name rename reparent rmdir relabelfrom
};

# 除 init 外，任何域不得触碰 private 文件
neverallow { domain -init } hotupdate_private_file:file *;
```

### 7.4 三层防线协同总结

```
攻击面              DAC 层              SELinux 层            neverallow 层
─────────────────────────────────────────────────────────────────────────────
读文件内容     0444 = 所有人可读    allow ... :file read    （放行）
改文件内容     0444 = 非 owner     不授予 write/append     neverallow 锁死
               不可写              → 拒绝                 → 编译不过
删文件         目录 0711 = 非     不授予 dir remove_name  neverallow 锁死
               owner 不可写        → 拒绝                 → 编译不过
创建文件       目录 0711 = 非     不授予 dir add_name     neverallow 锁死
               owner 不可写        → 拒绝                 → 编译不过
```

> **特别提醒**：root 进程持有 `CAP_DAC_OVERRIDE`，DAC 层拦不住它。但 SELinux 对 root 同样强制执行（除非域是 permissive）。neverallow 更是在编译阶段就把"SELinux 放行"这个可能性堵死。三层中，SELinux + neverallow 才是真正的防线。

> **两条落地必知的运维事实：**
> - **userdebug / eng 版本上 `su` 域是 permissive**：`adb root` 后仍可删改这两个文件，这是调试版本的固有行为，**user 版本不受影响**。定位“文件怎么被改了”时先确认是不是 debug 版。
> - **恢复出厂设置会整体清空 `/data`**：`filelist` / `proplist` 会随之消失。热更新方案必须把“**文件不存在 = 无更新记录**”作为正常初始状态处理——本方案读接口在文件缺失时返回空 vector，正是为此。

## 八、init.rc 集成

### 8.1 init.rc 配置

```rc
on post-fs-data
    # 创建目录并恢复 SELinux 标签
    # Android 12 FBE：新建 /data 顶层目录需显式声明加密策略（encryption=Require），
    # 否则该目录的加密归属在部分基线上不确定
    mkdir /data/hotupdate 0711 root root encryption=Require
    mkdir /data/hotupdate/para 0755 root root
    restorecon /data/hotupdate
    restorecon /data/hotupdate/para

    # 第二份对等镜像目录（NotifyEndUpdate 提交时同步发布）。
    # 若 /data/Vdr 为新建 /data 顶层目录，同样需声明 FBE 加密策略；
    # 若已由其它模块创建则此 mkdir 幂等（仅纠正权限）。
    mkdir /data/Vdr 0711 root root encryption=Require
    mkdir /data/Vdr/hotupdate 0711 root root
    restorecon /data/Vdr/hotupdate

    # 文件首次由写接口自动创建（含表头），此处仅确保标签正确
    # 启动后修正所有文件的上下文
    restorecon_recursive /data/hotupdate
    restorecon_recursive /data/Vdr/hotupdate
```

### 8.2 文件生命周期

```
首次开机:
  post-fs-data → mkdir /data/hotupdate(0711)
  init 调用 hotupdatefiles() → 发现 filelist 不存在
    → EnsureDir() 目录已存在，跳过
    → ReadEntries() 文件不存在 → 空列表
    → 写 filelist.tmp（含表头 + 新记录）→ fchmod 0444 → fsync → rename
    → restorecon /data/hotupdate/filelist
    → 文件就绪

后续调用（追加记录）:
  init 调用 hotupdatefiles()
    → flock(LOCK_EX)
    → ReadEntries() 解析现有文件
    → 判断 (key,value) 是否已存在
      - 存在 → 解锁返回 0（幂等）
      - 不存在 → 追加到列表 → 全量写 tmp → rename
    → 解锁返回 0

普通进程调用 gethotupdatefiles():
    → open(filelist, O_RDONLY) — 0444 允许
    → read 全量
    → 解析，返回第一列列表
```

## 九、并发模型

| 场景 | 保证机制 |
|---|---|
| 单一写者多次调用 | `flock(LOCK_EX)` 串行化 |
| 多个写者并发（未来） | `flock(LOCK_EX)` 互斥，后者等待 |
| 读者与写者并发 | rename 原子替换：读者要么看到完整的旧文件，要么看到完整的新文件，绝不看到半行 |
| 累积期崩溃（NotifyEndUpdate 之前） | 仅 `.tmp`/`.tmp.new` 残留，正式文件从未被改，reader 继续读旧正式文件；下次开机新进程首写时 `unlink` 清理残留、从空重建 |
| 提交期崩溃（NotifyEndUpdate 中途） | 4 次 rename 连续执行、无中间耗时操作，窗口极小；即便主目录已提交而镜像未提交，两者内容一致（源自同一份 `.tmp`），下一轮完整提交会再次对齐 |
| 掉电 | 原文件 + tmp 文件可能同时存在，rename 在文件系统层面是原子的（ext4/f2fs 均保证），不会出现文件损坏 |

**关键设计优势**：因为使用 rename 原子替换，读接口*完全不需要加锁*。这与 flock 建议性锁不同——如果某个读者不遵守 flock 协议，它仍然安全。这个特性对于"任意进程都可能链接 libhotupdate_client"的场景至关重要。

## 十、健壮性清单

| 措施 | 实现 |
|---|---|
| 输入校验 | 禁止 `\t\n\r\0` 和控制字符；文件名禁止 `/` 和空格；路径必须是绝对路径且在白名单内；propname 仅允许 `[a-zA-Z0-9_.-]` |
| 幂等性 | 重复 (key, value) 组合返回 0，不产生重复行 |
| 崩溃安全 | 累积期：记录写入 `.tmp.new` + fsync + rename 覆盖 `.tmp`，累积文件自身永不半写；提交期（`NotifyEndUpdate`）：先备好两份镜像 `.tmp` 再集中 rename + fsync 目录。提交前崩溃则无正式文件被改，reader 继续读旧正式文件 |
| 累积-提交两阶段 + 双目录对等镜像 | 循环内多次 `AtomicAppendRecord` 仅累积到 `/data/hotupdate/*.tmp`；`NotifyEndUpdate` 复制到 `/data/Vdr/hotupdate/*.tmp` 后，集中对 4 个 `.tmp` 做 rename，把主/镜像不一致窗口降到最小；任一 rename 失败记 `ALOGE` 并继续 |
| 会话干净起点 | 进程内每个列表首次写入时 `unlink` 残留 `.tmp`/`.tmp.new`；init 单进程跑完整轮循环，上一轮崩溃遗留的累积在下次开机（新进程）首写时被自动丢弃，无需新增公开接口 |
| 防缓冲区溢出 | 所有字符串长度硬上限，文件总大小 1MB 上限，记录数 4096 上限 |
| 符号链接防护 | 读路径统一 `O_NOFOLLOW`；临时文件用 `O_CREAT\|O_EXCL\|O_NOFOLLOW` 创建（`O_EXCL` 保证不会跟随已存在的同名符号链接），目标文件仅经 `rename` 原子替换、从不直接 `open` 写入 |
| 畸形数据容错 | 读接口跳过非法行（含尾部半行），绝不让调用方崩溃 |
| 目录自动修复 | 每次写调用兜底确保目录存在且权限正确 |
| 临时文件清理 | 本进程首次写某列表时 `unlink(tmp)`/`unlink(tmp.new)` 清理残留；提交成功后 `.tmp` 被 rename 消费掉；init.rc 启动时可再 `rm *.tmp` 兜底 |
| SELinux 标签一致性 | 创建 tmp 前 `setfscreatecon(public)` 使其一出生即为 public 标签，rename 后天然正确（消除标签窗口）；rename 后再 `selinux_android_restorecon()` 作防御性二次校正并检查返回值 |
| 依赖最小化 | Writer 只依赖 libc + liblog + libselinux；Reader 只依赖 libc + liblog |

## 十二、方案总结

| 层次 | 决策 |
|---|---|
| **文件格式** | TSV，`<key>\t<value>`，`#` 注释，UTF-8 编码；filelist 中 `key`(filename) 为相对路径（可含 `/`），proplist 中 `key` 为 propname；两种记录类型由不同文件自然区分，无需类型前缀字段 |
| **写接口** | `libhotupdate_writer.a` 静态库链入 init，原子 rename + flock + fsync |
| **读接口** | `libhotupdate_client.so` 动态库：`gethotupdatefiles()`、`gethotupdateprops()` 返回列表；`GetHotUpdateFile()` 精确查找并拼接绝对路径 |
| **路径拼接** | `GetHotUpdateFile` 将 `/data/hotupdate/para/` + 匹配的相对路径拼接为热更新文件绝对路径 |
| **DAC 权限** | 父目录 `0711 root root`，para 目录 `0755 root root`，文件 `0444 root root`，lock `0600 root root` |
| **SELinux 类型** | `hotupdate_dir` / `hotupdate_public_file`(含 `mlstrustedobject`) / `hotupdate_private_file` |
| **SELinux 授权** | Reader 白名单 attribute；Writer 仅 init |
| **编译期硬锁** | `neverallow { domain -init }` 禁止所有非 init 域的写/删/改名操作 |
| **健壮性** | 全字段校验 + 幂等去重 + 原子写入 + 畸形容错 + 大小/数量硬上限 + `O_NOFOLLOW` + 缓冲区溢出检查 |

这套方案综合了四份独立 AI 方案的核心优势，在满足"人类可读明文"需求的前提下，通过 TSV + 原子 rename + DAC 文件权限 + SELinux 标签管控 + neverallow 编译期锁死的五层设计，实现了"任意进程可读、仅 init 可写、任何进程不能删"的安全目标。新增的 `GetHotUpdateFile` 接口提供精确查找 + 路径拼接能力，配合严格的参数校验和缓冲区边界检查，代码在每个异常路径上都有明确的错误处理和返回值，达到了系统关键接口所需的可靠性与健壮性要求。
