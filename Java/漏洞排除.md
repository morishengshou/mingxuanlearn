要确认自己的 Java 项目是否使用了 CVE 中提到的漏洞模块，可以按以下步骤进行：

1. **准确定义漏洞范围**  
   - 在 CVE 网站或其引用的安全公告中查明：受影响的库（artifactId / groupId）、版本范围、触发条件、修复版本等。  
   - 记录 CVE 编号、漏洞描述、受影响组件的 Maven 坐标（或 Gradle/GAV 坐标）。

2. **清点依赖**  
   - 如果使用 Maven：运行 `mvn dependency:tree -Dincludes=<groupId>:<artifactId>` 查看是否包含受影响模块及其版本。  
   - 如果使用 Gradle：运行 `./gradlew dependencies --configuration <config>` 或 `./gradlew dependencyInsight --dependency <artifact>`。  
   - 若是多模块/层层依赖，务必检查所有子模块和传递依赖。

3. **利用软件成分分析（SCA）工具**  
   - 使用工具如 OWASP Dependency-Check、Snyk、JFrog Xray、GitHub Dependabot、Whitesource 等，可自动扫描并匹配 CVE 数据库。  
   - 将工具接入 CI/CD，确保每次构建都会检测新依赖。

4. **比对版本与补丁**  
   - 若项目确实依赖到受影响库，查看所用版本是否在漏洞范围内。  
   - 若在范围内，优先升级到官方修复版本。若无修复版本，寻找社区补丁、临时缓解措施或替换方案。

5. **确认自研模块是否间接引用**  
   - 如果代码中使用了本地 jar 或自建库，检查其依赖集合（例如 pom.xml、build.gradle、lib 目录中的 jar），确保未包含漏洞组件。  
   - 对于 shaded/重打包的 jar，可用 `jar tf xxx.jar` 查看内部依赖，或用 `jdeps` 进行分析。

6. **代码审计与运行时检查**  
   - 根据 CVE 的利用方式，审查项目中是否调用到受影响 API。  
   - 在运行时通过 classpath 日志（如 Spring Boot 启动日志）确认加载的库版本。

7. **建立依赖清单与变更流程**  
   - 维护 SBOM（Software Bill of Materials），记录所有第三方组件及版本。  
   - 新引入依赖需经过安全审查，更新时同步验证是否修复相关 CVE。

8. **持续关注公告**  
   - 订阅相关库的安全公告、邮件列表或使用安全工具的自动通知，确保第一时间获知新漏洞。

通过以上步骤，可以系统地确认你的代码是否使用了 CVE 涉及的 Java 模块，并在必要时采取升级或缓解措施。
