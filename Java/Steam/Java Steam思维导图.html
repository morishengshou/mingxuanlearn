Java Stream API 思维导图

一句话记忆：Stream = 用“流水线”表达数据处理。
它擅长过滤、转换、统计、分组、排序、收集；但不适合复杂状态控制、强依赖下标和极致性能场景。

记忆口诀
先记思想：声明式 + 流水线 再记结构：Source → 中间操作 → 终止操作 适合：数据加工 不适合：复杂状态控制 刷题：前后处理很好用 核心算法：通常还是 for 更强

1. 设计思想

“写做什么”，而不是“怎么一步步遍历”
声明式 内部迭代 惰性求值 少副作用 一次性消费
  • 声明式：直接描述“过滤 / 转换 / 收集”。
  • 内部迭代：循环细节交给框架处理。
  • 惰性求值：没有终止操作，通常不会真正执行。
  • 少副作用：尽量不要在流里改外部变量。
  • 只能消费一次：同一个 Stream 不能重复用。

2. 核心结构

最重要的主线：数据源 → 中间操作 → 终止操作
Source filter map sorted collect reduce
  • 数据源：List、Set、数组、IntStream.range 等。
  • 中间操作：filter、map、flatMap、distinct、sorted、limit。
  • 终止操作：collect、count、sum、max、findFirst、anyMatch。
  • 基本类型流：IntStream / LongStream / DoubleStream 更高效。

3. 什么时候用最方便

适合“批量数据加工”的链式表达
过滤 转换 统计 排序 分组 字符串处理
  • 过滤 + 转换 + 收集成 List / Set / Map。
  • 求和、计数、最大值、最小值。
  • 去重 + 排序 + 截取前若干项。
  • 频次统计、分组聚合。
  • 字符串清洗、拼接、批量变换。

最核心理解：它更像“数据处理描述器”，不是集合本身，也不是所有场景都要替代 for 循环。

4. 哪些场景不要硬用

复杂状态控制时，传统循环通常更清晰
下标控制 双指针 滑动窗口 DP 回溯 图搜索
  • 强依赖索引、边界和状态转移的题型。
  • 原地修改数组、复杂 break / continue 控制。
  • DFS / BFS / 回溯 / 动态规划等核心过程。
  • 极致性能要求时，普通循环往往更直接高效。

5. LeetCode 常用技巧

适合做“前处理”和“后处理”
distinct + sorted sum / max / min toSet groupingBy flatMap anyMatch
  • 去重排序：Arrays.stream(nums).distinct().sorted().toArray()
  • 快速统计：sum()、max()、min()
  • 转 Set：boxed().collect(Collectors.toSet())
  • 词频：groupingBy(..., counting())
  • 打平：flatMap / flatMapToInt
  • 条件判断：anyMatch / allMatch / noneMatch

6. 常见坑 + 最后结论

会用很方便，乱用会更绕
忘记终止操作 toMap 重复 key 忘记 boxed 滥用 forEach toList 可能不可变
  • 没有终止操作，流通常不会执行。
  • toMap() 遇到重复 key 会报错，要写合并规则。
  • 基本类型流转集合时常常要先 boxed()。
  • forEach 太多，容易又写回命令式风格。
  • 最终原则：Stream 是辅助你把“数据加工”写清楚，不是替代所有循环。
背诵版总结：
Stream 的核心是流水线式数据处理。先有数据源，再接中间操作，最后用终止操作产出结果。
它最适合：过滤、转换、统计、分组、排序、收集。
它不适合：复杂索引控制、滑窗双指针、DP、回溯、图搜索、原地修改。
刷题时把它当作前处理 / 后处理利器最实用，核心算法过程一般仍以 for 循环为主。
