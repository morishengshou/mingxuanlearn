会不会“污染符号表”、会不会让产物变大，取决于**这些函数有没有真的进入最终链接产物**以及**是否被保留**。分两层看：编译（.o）和链接（可执行文件/.so/.a）。

## 1) 只在一个 `.cpp` 里定义了但从未调用：会发生什么？

### 对 `.o`（编译后的目标文件）
- **通常会进入 `.o`**：编译器一般会把这个函数的代码生成出来，放进目标文件的 `.text` 段里。
- **符号表也会有记录**（哪怕没用）：目标文件会包含该函数符号（以及调试信息等）。  
  所以就“编译阶段产物（.o）”来说：**符号表/体积往往会增加**（尤其是开了 `-g` 调试信息时更明显）。

### 对最终链接产物（exe / .so）
- **不一定会进最终产物**：链接器是否把它放进最终文件，取决于是否启用了“未引用代码剔除（dead stripping）”这一类优化，以及该函数的可见性/是否被强制保留。

## 2) 最终产物会不会变大？常见情况

### 情况 A：普通可执行文件/共享库，开启了 section 粒度的剔除
如果你使用了类似：
- GCC/Clang：`-ffunction-sections -fdata-sections`（让每个函数单独一个 section）
- 链接：`-Wl,--gc-sections`（把未引用的 section 回收）

那么**未被引用的函数通常不会进入最终产物**（或者会被回收），最终体积**基本不受影响**。

> 注意：仅有 `-O2/-O3` 不等价于链接层面的剔除；关键是链接器是否能以函数为粒度丢掉它。

### 情况 B：没开上述选项
- 许多平台/构建默认并不会把“同一目标文件里未引用的函数”自动丢掉（尤其当函数和其他用到的东西在同一个 section 里）。
- 结果是：只要这个 `.o` 被链接进来，**这些未用函数的代码也可能跟着进最终产物**，体积会**变大**。

### 情况 C：做成静态库 `.a`
静态库的行为更特殊：
- 链接器通常以“**目标文件（.o）为单位**”从 `.a` 里抽取：如果某个 `.o` 里有任何被需要的符号，整个 `.o` 就会被拉进来。
- 如果你把“常用函数”和“一堆没用函数”放在同一个 `.o`（同一个源文件编译出来的），那么只要常用函数被用到，没用的也可能被带进最终产物。
- 但如果开启了“函数分 section + gc-sections”，就可能把同一个 `.o` 里没用的函数再剔掉（取决于工具链/平台配置）。

## 3) “污染符号表”具体指什么？

- **对链接器/运行时可见的符号**：如果这些函数是**外部链接**（非 `static`、非匿名命名空间、且没有隐藏可见性），它们可能出现在最终二进制的动态符号/导出符号里（取决于平台与链接方式）。这会：
  - 增加符号数量（一定程度上算“污染”）
  - 可能影响链接时间、符号查找、以及 ABI/导出接口管理（尤其是库）
- 如果这些函数是**内部链接**（`static` 或放在匿名命名空间），通常不会对外导出；即使代码被带进最终产物，也不太会“对外污染”。

---

## 4) 实务建议（想确保“不增大、不污染”）

1. **如果只是本 `.cpp` 内部用的工具函数**：  
   用 `static` 或匿名命名空间限制链接性：
   ```cpp
   namespace {
     void helper() {}
   }
   ```
2. **给库控制导出符号**：  
   - GCC/Clang 常用：`-fvisibility=hidden`，需要导出的再显式标注
3. **开启未引用剔除（推荐）**：  
   - 编译：`-ffunction-sections -fdata-sections`
   - 链接：`-Wl,--gc-sections`
4. **拆分文件**（尤其静态库）：把不常用/可选的实现放到单独 `.cpp`，避免一个 `.o` 被拉进来时“夹带私货”。
