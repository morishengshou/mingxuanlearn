## 1. 移植范围界定

### 1.1 移植目标

将以下两个文件对应的能力移植到目标系统：

```
frameworks/config_policy/src/config_policy_utils.c    ← 全部核心逻辑
interfaces/inner_api/include/config_policy_utils.h    ← 公开 C API 头文件
interfaces/inner_api/include/config_policy_impl.h     ← 内部宏定义头文件
```

目标系统特征（已知条件）：
- 鸿蒙内核（非标准 OpenHarmony，有自己的 init 集成方式）
- 支持 C++14 以上标准库（libc 完整）
- 有独有的系统参数读取 API（替代 `init_param.h`）

### 1.2 不移植的内容

| 不移植项 | 原因 |
|----------|------|
| `interfaces/kits/js/` (NAPI) | 依赖 ArkTS 运行时 |
| `interfaces/kits/cj/` (CJ FFI) | 依赖仓颉运行时 |
| `frameworks/dfx/` (HiSysEvent) | 依赖 OpenHarmony 诊断框架 |
| `test/` | 不需要移植测试 |
| `bundle.json` / `BUILD.gn` | 构建系统不同 |

## 2. 依赖全景分析

以下是 `config_policy_utils.c` 中**每一个外部依赖**的完整清单，按可移植性分级：

### 2.1 标准 C/POSIX 头文件 — ✅ 无需修改

| 头文件 | 使用的符号 | 说明 |
|--------|-----------|------|
| `<ctype.h>` | `isspace()` | C89 标准，通用 |
| `<stdbool.h>` | `bool`, `true`, `false` | C99 标准，通用 |
| `<unistd.h>` | `access()`, `F_OK` | POSIX，检查文件是否存在 |
| `<stdlib.h>` | `malloc()`, `calloc()`, `free()`, `atoi()`, `strdup()` | 标准 C，通用 |
| `<string.h>` | `strlen()`, `strchr()`, `strstr()`, `strdup()` | 标准 C，通用 |
| `<errno.h>` | `errno`, `EINVAL` | 标准 C，通用 |

### 2.2 OpenHarmony 专有依赖 — ⚠️ 必须适配

| 头文件 | 来源组件 | 使用的符号 | 适配难度 |
|--------|----------|-----------|----------|
| `<securec.h>` | `bounds_checking_function` | `strcpy_s`, `sprintf_s`, `snprintf_s`, `memmove_s`, `memcpy_s`, `strcat_s`, `memset_s`, `strtok_s`, `EOK` | 中（可用兼容层替代） |
| `"init_param.h"` | `init` | `SystemGetParameter()`, `PARAM_CONST_VALUE_LEN_MAX` | 高（核心集成点，必须实现） |
| `"cmsis_compiler.h"` | CMSIS（仅 LiteOS-M） | `__WEAK` | 低（仅微控制器路径） |

### 2.3 代码中的平台分支

源码通过预处理宏区分三种平台，你的目标系统需要明确归入哪一个分支（或新增一个）：

```
config_policy_utils.c 中的平台分支：

#ifdef __LITEOS_M__          ← 微控制器（LiteOS-M），无参数系统，用静态 buffer
#elif defined(__LITEOS__)    ← 轻量 LiteOS（非 M），使用 DEFAULT_LAYER，无参数读取
#else                        ← 标准系统（Linux 内核），读取 init 系统参数
```

**你的目标系统最可能对应的分支：**
若目标系统有完整 libc 和文件系统，且有自己的 init 参数 API，应对应 `#else`（标准系统路径），只需替换其中的 `init_param.h` 调用。

## 4. 必须适配的代码（逐项详解）

### 4.1 最关键：init 参数读取接口

这是整个移植中**最核心的适配点**。源码通过以下两个来自 `init_param.h` 的符号读取系统参数：

```c
// 来自 init_param.h，用于读取形如 "const.cust.config_dir_layer" 的键值对
int SystemGetParameter(const char *name, char *value, unsigned int *len);
// len 传入时为 buffer 大小，传出时为实际读到的字符串长度（含 '\0'）
// 返回 0 表示成功，非 0 表示失败（key 不存在或 buffer 不足）

// 最大参数值长度（通常为 4096）
#define PARAM_CONST_VALUE_LEN_MAX  4096
```

**在源码中被调用的位置（全部在 `config_policy_utils.c`）：**

| 行号 | 位置 | 读取的参数 Key | 含义 |
|------|------|--------------|------|
| 107-111 | `CustGetSystemParam()` | 任意 key | 通用参数读取封装 |
| 129-133 | `GetOpkeyPath()` | `telephony.sim.opkey0/1` | 读取运营商 PLMN 码（Follow X 使用） |
| 191 | `GetFollowXRule()` | `const.cust.follow_x_rules` | 读取 Follow X 规则 |
| 400 | `GetCfgDirRealPolicyValue()` | `const.cust.config_dir_layer` | **最关键**：读取配置层级路径 |

**适配方式：** 参见第 5 节的完整适配方案。

### 4.2 安全 C 库（`securec.h`）

**`<securec.h>` 的来源：** 华为自研的 `bounds_checking_function` 组件，提供带边界检查的 C 字符串/内存操作函数。

**在源码中使用的函数汇总：**

| securec 函数 | 标准 C/POSIX 等价写法 | 差异说明 |
|-------------|---------------------|---------|
| `strcpy_s(dst, dstMax, src)` | `strncpy(dst, src, dstMax-1); dst[dstMax-1]='\0'` | securec 返回 EOK/错误码 |
| `sprintf_s(buf, bufMax, fmt, ...)` | `snprintf(buf, bufMax, fmt, ...)` 并检查返回值 | securec 截断时返回 -1 |
| `snprintf_s(buf, bufMax, count, fmt, ...)` | `snprintf(buf, bufMax, fmt, ...)` | 见注①  |
| `memmove_s(dst, dstMax, src, count)` | `memmove(dst, src, count)` | 需手动检查 count <= dstMax |
| `memcpy_s(dst, dstMax, src, count)` | `memcpy(dst, src, count)` | 需手动检查 count <= dstMax |
| `strcat_s(dst, dstMax, src)` | `strncat(dst, src, dstMax-strlen(dst)-1)` | 需先确保空间 |
| `memset_s(dst, dstMax, val, count)` | `memset(dst, val, count)` | 需手动检查 count <= dstMax |
| `strtok_s(str, delim, &ctx)` | `strtok_r(str, delim, &ctx)` | POSIX 等价（参数名相同） |
| `EOK` | `0` | 成功返回值常量 |

> **注①**：`snprintf_s(buf, bufLen, bufLen-1, fmt, ...)` 中第三参数是最大写入字符数（不含 `\0`）。源码中一律传 `bufLen-1`，等价于 `snprintf(buf, bufLen, fmt, ...)`，但 snprintf_s 在截断时返回 -1，snprintf 返回"应该写入的字符数"（可能大于 bufLen）。源码用 `> 0` 判断成功，替换时需加额外截断判断，详见第 6 节。

**适配方式：** 参见第 6 节的兼容层方案。

### 4.3 平台宏定义与内核标识

源码中有三个分支，你需要确认目标系统进入哪个：

```c
#ifdef __LITEOS_M__      // ← 是 LiteOS-M (MCU) 内核吗？
    ...
#elif defined(__LITEOS__)  // ← 是其他 LiteOS (不带 _M_) 内核吗？
    ...
#else                    // ← 标准 Linux 或其他具有完整 init 参数的系统
    ...
#endif
```

**对于"有自己 init 参数读取 API 的鸿蒙内核系统"：**

- 若目标系统不是 LiteOS-M 也不是轻量 LiteOS，**直接走 `#else` 分支即可**
- 只需确保编译时 **不定义** `__LITEOS__` 和 `__LITEOS_M__` 宏
- 在构建脚本中加入 `-DTARGET_HARMONY_CUSTOM` 之类的自定义宏用于未来扩展（可选）

**`__WEAK` 宏（仅 LiteOS-M 路径使用）：**

```c
// 原代码：来自 cmsis_compiler.h
__WEAK void TrigSetMiniConfigPolicy() { return; }

// 如果你的编译器是 GCC 或 Clang，直接替换为：
__attribute__((weak)) void TrigSetMiniConfigPolicy() { return; }

// 或者在你的适配头文件中加：
#ifndef __WEAK
#define __WEAK __attribute__((weak))
#endif
```

### 4.4 DEFAULT_LAYER 路径常量

当系统参数 `const.cust.config_dir_layer` 不存在时，使用此默认值：

```c
// 原定义（config_policy_impl.h）
#define DEFAULT_LAYER  "/system:/chipset:/sys_prod:/chip_prod"
```

**需要根据目标系统的实际分区结构修改。** 常见情况：

| 目标系统类型 | 建议的 DEFAULT_LAYER |
|------------|---------------------|
| 标准 OpenHarmony 设备 | `/system:/chipset:/sys_prod:/chip_prod`（不变） |
| 自定义鸿蒙设备（双分区） | `/system:/vendor` |
| 嵌入式设备（单分区） | `/system` |
| 自定义路径设备 | 按实际分区结构填写 |

**修改位置：** `interfaces/inner_api/include/config_policy_impl.h` 第 37 行：

```c
#define DEFAULT_LAYER  "/your_partition_1:/your_partition_2"
```

**也可以完全不修改代码，在运行时通过设置系统参数覆盖：**
在 init 启动脚本中设置 `const.cust.config_dir_layer=/your_partition_1:/your_partition_2`，运行时优先使用参数值。

## 5. 完整适配方案：参数适配层设计

**设计思路：** 创建一个适配层头文件，将"读取系统参数"的需求抽象成一个统一接口，屏蔽底层 API 差异，让 `config_policy_utils.c` 不需要感知目标系统的 init API。

### 5.1 创建适配头文件

在你的项目中新建文件：`port/config_policy_param_adapter.h`

```c
/*
 * config_policy_param_adapter.h
 *
 * 参数读取接口适配层
 * 将目标系统的参数 API 适配为 config_policy 所需的接口
 *
 * 使用方法：
 *   在编译 config_policy_utils.c 时，通过 -I 将此文件所在目录加入头文件搜索路径，
 *   并在编译命令中添加 -DUSE_PARAM_ADAPTER，
 *   同时将 init_param.h 的包含替换为本文件。
 */

#ifndef CONFIG_POLICY_PARAM_ADAPTER_H
#define CONFIG_POLICY_PARAM_ADAPTER_H

#ifdef __cplusplus
extern "C" {
#endif

/* =========================================================
 * 步骤1：包含目标系统的参数头文件
 * 将下方的 #include 替换为目标系统实际提供的头文件
 * ========================================================= */
#include "your_system_param.h"   /* ← 替换为目标系统的头文件名 */

/* =========================================================
 * 步骤2：定义 PARAM_CONST_VALUE_LEN_MAX
 * 如果目标系统头文件中已经定义了等价常量，直接用 #define 映射；
 * 如果没有，使用默认值 4096（OpenHarmony 中的值）
 * ========================================================= */
#ifndef PARAM_CONST_VALUE_LEN_MAX
#ifdef YOUR_SYSTEM_PARAM_MAX_VALUE_LEN   /* ← 替换为目标系统的等价宏名 */
#define PARAM_CONST_VALUE_LEN_MAX  YOUR_SYSTEM_PARAM_MAX_VALUE_LEN
#else
#define PARAM_CONST_VALUE_LEN_MAX  4096
#endif
#endif

/* =========================================================
 * 步骤3：提供 SystemGetParameter 函数
 *
 * 此函数的语义必须与 OpenHarmony 的完全一致：
 *   - name:   参数 key，以 '\0' 结尾的 C 字符串
 *   - value:  输出缓冲区，可以为 NULL（此时只查询长度）
 *   - len:    传入时为 value 缓冲区大小；
 *             传出时为实际字符串长度（含 '\0'）
 *   - 返回值: 0 表示成功，非 0 表示失败（key 不存在或 buffer 不足）
 *
 * 根据目标系统的实际 API 选择以下三种方式之一：
 * ========================================================= */

/* --- 方式 A：目标系统 API 与 OpenHarmony 完全相同，直接复用 ---
 * 如果目标系统直接提供同名同签名的 SystemGetParameter，
 * 则什么都不用写，头文件 include 之后就可以直接用。
 */

/* --- 方式 B：目标系统 API 名称不同，但语义相同，用宏映射 ---
 * 例如目标系统的函数叫 YourSystem_GetParam(name, buf, bufLen)
 * 且返回值语义相同：
 */
/*
#define SystemGetParameter(name, value, len) \
    YourSystem_GetParam((name), (value), *(len))
*/

/* --- 方式 C：目标系统 API 语义有差异，需要封装函数 ---
 * 例如目标系统返回字符串指针而非写入 buffer，或使用不同的 len 语义：
 */
/*
static inline int SystemGetParameter(const char *name, char *value, unsigned int *len)
{
    // value == NULL 时，只查询 key 是否存在和值的长度
    if (value == NULL) {
        const char *val = YourSystem_GetParamValue(name);
        if (val == NULL) return -1;
        *len = (unsigned int)(strlen(val) + 1);
        return 0;
    }
    // value != NULL 时，将值写入 buffer
    const char *val = YourSystem_GetParamValue(name);
    if (val == NULL) return -1;
    unsigned int srcLen = (unsigned int)(strlen(val) + 1);
    if (srcLen > *len) {
        *len = srcLen;   // 告知调用方需要多大 buffer
        return -1;       // buffer 不足，返回失败
    }
    memcpy(value, val, srcLen);
    *len = srcLen;
    return 0;
}
*/

#ifdef __cplusplus
}
#endif

#endif /* CONFIG_POLICY_PARAM_ADAPTER_H */
```

## 6. 完整适配方案：securec 兼容层

**优先方案（推荐）：确认目标系统是否已有 securec**

由于目标系统是鸿蒙内核系统，大概率已包含 `bounds_checking_function` 组件（`libsec_shared` 或 `libsec_static`）。请先检查：

```bash
# 在目标系统 SDK 或工具链中查找
find /path/to/sdk -name "securec.h" 2>/dev/null
find /path/to/sdk -name "libsec*.a" -o -name "libsec*.so" 2>/dev/null
```

**若已有 securec.h**：无需此节，直接链接即可，`config_policy_utils.c` 不需要任何修改。

**若没有 securec.h**：创建兼容层头文件 `port/compat_securec.h`：

```c
/*
 * compat_securec.h
 *
 * 当目标系统没有 securec.h 时的兼容层
 * 提供与 Huawei bounds_checking_function 兼容的宏/函数
 *
 * 设计原则：
 *   - 行为与 securec 函数一致（成功返回 EOK=0，失败返回非 0）
 *   - 对 NULL 指针和越界保持安全（不崩溃，返回错误码）
 *   - 不追求 100% 功能等价，只覆盖 config_policy_utils.c 实际使用的场景
 */

#ifndef COMPAT_SECUREC_H
#define COMPAT_SECUREC_H

#include <string.h>
#include <stddef.h>
#include <errno.h>

/* securec 的成功返回值 */
#ifndef EOK
#define EOK  0
#endif

/* ---- strcpy_s ---- */
static inline int strcpy_s(char *dest, size_t destMax, const char *src)
{
    if (dest == NULL || src == NULL || destMax == 0) return EINVAL;
    size_t srcLen = strlen(src);
    if (srcLen >= destMax) return ERANGE; /* 目标空间不足 */
    memcpy(dest, src, srcLen + 1);
    return EOK;
}

/* ---- strcat_s ---- */
static inline int strcat_s(char *dest, size_t destMax, const char *src)
{
    if (dest == NULL || src == NULL || destMax == 0) return EINVAL;
    size_t destLen = strnlen(dest, destMax);
    if (destLen >= destMax) return EINVAL; /* dest 本身已超界 */
    size_t srcLen = strlen(src);
    if (destLen + srcLen >= destMax) return ERANGE;
    memcpy(dest + destLen, src, srcLen + 1);
    return EOK;
}

/* ---- memcpy_s ---- */
static inline int memcpy_s(void *dest, size_t destMax, const void *src, size_t count)
{
    if (dest == NULL || src == NULL) return EINVAL;
    if (count > destMax) return ERANGE;
    if (count == 0) return EOK;
    memcpy(dest, src, count);
    return EOK;
}

/* ---- memmove_s ---- */
static inline int memmove_s(void *dest, size_t destMax, const void *src, size_t count)
{
    if (dest == NULL || src == NULL) return EINVAL;
    if (count > destMax) return ERANGE;
    if (count == 0) return EOK;
    memmove(dest, src, count);
    return EOK;
}

/* ---- memset_s ---- */
static inline int memset_s(void *dest, size_t destMax, int c, size_t count)
{
    if (dest == NULL) return EINVAL;
    if (count > destMax) return ERANGE;
    if (count == 0) return EOK;
    memset(dest, c, count);
    return EOK;
}

/* ---- snprintf_s ----
 * 注意：securec 的 snprintf_s 在截断时返回 -1，
 *       标准 snprintf 返回"应写入的字符数"（可能大于 bufLen）。
 * 此兼容实现在截断时返回 -1，与 securec 行为一致。
 */
#include <stdio.h>
static inline int snprintf_s(char *str, size_t strMax, size_t count, const char *format, ...)
{
    (void)count; /* strMax-1 已足够，count 在源码中始终传 strMax-1 */
    if (str == NULL || strMax == 0) return -1;
    va_list args;
    va_start(args, format);
    int ret = vsnprintf(str, strMax, format, args);
    va_end(args);
    /* 若截断（ret >= strMax）或出错（ret < 0），返回 -1 */
    if (ret < 0 || (size_t)ret >= strMax) return -1;
    return ret;
}

/* ---- sprintf_s ---- */
static inline int sprintf_s(char *str, size_t strMax, const char *format, ...)
{
    if (str == NULL || strMax == 0) return -1;
    va_list args;
    va_start(args, format);
    int ret = vsnprintf(str, strMax, format, args);
    va_end(args);
    if (ret < 0 || (size_t)ret >= strMax) return -1;
    return ret;
}

/* ---- strtok_s ----
 * securec/Windows 版：strtok_s(str, delim, &context)
 * POSIX 版：strtok_r(str, delim, &saveptr) — 接口完全相同
 * 直接用宏映射即可
 */
#define strtok_s(str, delim, context)  strtok_r((str), (delim), (context))

#endif /* COMPAT_SECUREC_H */
```

改为：

## 10. 常见陷阱与注意事项

### 10.1 ⚠️ `CfgDir->paths[]` 是指向 `realPolicyValue` 内部的指针

```c
/* GetCfgDirList() 的内部实现：
   对 realPolicyValue 字符串（如 "/system:/chipset"）原地切割，
   paths[] 里的指针直接指向这块内存内部，不是独立 strdup 出来的。
   因此：
   - 不要单独 free(dirs->paths[i])
   - FreeCfgDirList 会 free(realPolicyValue)，同时所有 paths 都失效
   - 使用完 dirs 之前，不要 free realPolicyValue
*/
CfgDir *dirs = GetCfgDirList();
// ✅ 正确用法
for (int i = 0; i < MAX_CFG_POLICY_DIRS_CNT; i++) {
    if (dirs->paths[i]) printf("%s\n", dirs->paths[i]);
}
FreeCfgDirList(dirs);  // 只调用这一个

// ❌ 错误用法
// free(dirs->paths[0]);  // 会破坏 realPolicyValue 内存
```

### 10.2 ⚠️ `GetOneCfgFile` 的 `bufLength` 必须 ≥ `MAX_PATH_LEN`

```c
/* bufLength < MAX_PATH_LEN 会直接返回 NULL，这是有意为之的防御 */
char buf[256] = {0};
char *path = GetOneCfgFile("etc/test.xml", buf, 255);  // ❌ 返回 NULL！

char buf2[MAX_PATH_LEN] = {0};
char *path2 = GetOneCfgFile("etc/test.xml", buf2, MAX_PATH_LEN);  // ✅
```

### 10.3 ⚠️ `snprintf_s` 与标准 `snprintf` 的截断行为差异

源码中用 `snprintf_s(buf, len, len-1, fmt, ...) > 0` 来判断成功。
若使用兼容层的 `snprintf_s`（截断返回 -1），行为与 securec 一致。
若直接改成标准 `snprintf`，需注意：

```c
/* 标准 snprintf 截断时返回"应写入的字节数"（可能 > len），而非 -1 */
/* 替换时需要添加截断检测 */
int ret = snprintf(buf, bufLen, fmt, ...);
if (ret > 0 && ret < (int)bufLen) {
    /* 成功且未截断 */
}
```

### 10.4 ⚠️ `strtok_s`（securec）与 C11 标准的 `strtok_s` 签名不同

```c
/* securec / Windows 版（源码使用的） */
char *strtok_s(char *str, const char *delim, char **context);

/* C11 标准版（签名不同！） */
char *strtok_s(char *s1, rsize_t *s1max, const char *s2, char **ptr);

/* POSIX 版（与 securec 版接口完全相同，只是名字不同） */
char *strtok_r(char *str, const char *delim, char **saveptr);

/* ✅ 正确替换：用宏映射到 strtok_r */
#define strtok_s(str, delim, ctx)  strtok_r((str), (delim), (ctx))
```

### 10.5 ⚠️ Follow X 功能依赖运营商参数键

若目标系统没有 `telephony.sim.opkey0/1` 这两个参数（非手机设备），`GetOpkeyPath()` 会返回 NULL，Follow X 的 SIM 模式自动退化为"不跟随"，**不会崩溃，只是找不到运营商子目录**。这是正常的降级行为，无需特别处理。

### 10.6 ⚠️ `access()` 在某些嵌入式文件系统上的行为

部分嵌入式实时系统的 `access()` 实现可能与 POSIX 有差异（如总是返回 0）。如遇异常，可检查目标系统的 `unistd.h` 中 `access()` 的实现。

### 10.7 ✅ strdup() 的可用性

`strdup()` 虽然是 POSIX 函数而非 C 标准，但几乎所有支持 C++14 libc 的系统都提供它。
若不确定，在适配层中加一个保险：

```c
/* 在 config_policy_param_adapter.h 中追加 */
#ifndef HAVE_STRDUP
#include <string.h>
#include <stdlib.h>
static inline char *strdup_compat(const char *s)
{
    if (!s) return NULL;
    size_t len = strlen(s) + 1;
    char *p = (char *)malloc(len);
    if (p) memcpy(p, s, len);
    return p;
}
#define strdup strdup_compat
#endif
```

### 10.8 线程安全说明

`config_policy_utils.c` 中的所有函数：
- **无全局状态**（除 `__LITEOS_M__` 路径的静态 `gConfigPolicy[]`）
- **线程安全**（在非 LiteOS-M 平台上，每次调用都从参数系统读取，没有缓存）
- 调用方负责多线程环境下对返回的 `CfgDir*` / `CfgFiles*` 的同步访问

## 附录 C：适配层接口契约（供系统对接方参考）

以下是目标系统 init 模块需要提供（或通过适配层包装）的接口语义规范：

```c
/*
 * 接口名称：SystemGetParameter（或等价包装）
 *
 * 功能：从系统参数存储中读取键值对
 *
 * 参数：
 *   name  [in]  参数键名，如 "const.cust.config_dir_layer"
 *   value [out] 输出缓冲区，NULL 时只查询是否存在及长度
 *   len   [in/out] 传入：value 的缓冲区大小（字节）
 *                  传出：实际字符串长度（含 '\0'）
 *
 * 返回值：
 *   0    成功
 *   非 0  失败（key 不存在、缓冲区不足、参数非法等）
 *
 * 示例调用序列：
 *   // 第一次调用：value=NULL，查询所需长度
 *   unsigned int len = 0;
 *   if (SystemGetParameter("const.cust.config_dir_layer", NULL, &len) != 0) {
 *       // key 不存在，len 无意义
 *   }
 *   // 第二次调用：分配 buffer 后读取值
 *   char *buf = calloc(len, 1);
 *   SystemGetParameter("const.cust.config_dir_layer", buf, &len);
 *   // buf 现在包含 "/system:/chipset:/sys_prod:/chip_prod"
 *
 * 附加常量：
 *   PARAM_CONST_VALUE_LEN_MAX — 参数值的最大允许长度（字节），推荐 ≥ 4096
 */
```
