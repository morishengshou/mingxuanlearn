## 1. 阅读前的认知准备

### 1.1 这件事有多复杂？

运行 OpenHarmony 系统级组件的测试，**不像运行普通 Python 脚本或 Java 程序那么简单**，它需要：

- 一台 **Linux 电脑**（运行编译工具链，Windows 本机无法直接使用）
- 至少 **400 GB 空闲磁盘**（存放源代码和编译产物）
- 至少 **16 GB 内存**（推荐 32 GB）
- **数小时**的源码下载时间（约 50~100 GB）
- 一次**首次完整编译**可能需要几小时

### 1.2 测试需要什么设备？

有两种选择：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **真实 OpenHarmony 设备**（如润和 Hi3516、RK3568 开发板） | 最真实 | 需要购买硬件，价格百元~千元不等 |
| **QEMU 模拟器**（软件模拟） | 无需硬件，免费 | 配置复杂，部分功能受限 |

### 1.3 本文档的测试路径总览

```
你的 Windows 电脑
    │
    └─ （通过 VMware/WSL2 等运行）Ubuntu 22.04
            │
            ├─ 1. 安装编译工具链
            ├─ 2. 下载 OpenHarmony 源码（包含本仓代码）
            ├─ 3. 编译测试二进制：ConfigPolicyUtilsTest
            │
            ├─ 连接 OpenHarmony 设备 或 启动 QEMU 模拟器
            │       │
            │       ├─ 4. hdc 推送测试文件到设备
            │       ├─ 5. 在设备上创建测试所需的目录和文件
            │       └─ 6. 执行测试，查看结果
            │
            └─ 7. （可选）运行模糊测试
```

## 2. 关键概念解释

### 2.1 Ubuntu / Linux 环境

OpenHarmony 的编译工具链只支持 **Linux 系统**（推荐 Ubuntu 22.04 LTS）。如果你用的是 Windows，有三种方式获得 Linux 环境：

- **WSL2**（Windows Subsystem for Linux 2）：在 Windows 上直接运行 Linux，推荐新手使用
  - 安装指南：https://learn.microsoft.com/zh-cn/windows/wsl/install
- **VMware Workstation**：虚拟机，稳定但需要更多内存
- **双系统**：在电脑上同时安装 Windows 和 Ubuntu

### 2.2 repo 工具

`repo` 是 Google 开发的多仓库管理工具，OpenHarmony 由数百个 Git 仓库组成，`repo` 能一次性同步所有仓库。

```
repo → 管理多个 git 仓库，类似于"批量 git clone"
```

### 2.3 hb 编译工具

`hb`（HarmonyOS Build）是 OpenHarmony 的命令行编译工具，封装了底层的 GN + Ninja 构建过程。

```
你 → hb build → GN 生成构建规则 → Ninja 执行编译 → 得到 .so / 测试二进制文件
```

### 2.4 GN + Ninja 构建系统

- **GN**（Generate Ninja）：读取所有 `BUILD.gn` 文件，生成 Ninja 构建规则
- **Ninja**：根据规则执行实际编译，类似 make 但速度更快

> 更多 GN 介绍：https://gn.googlesource.com/gn/+/main/docs/quick_start.md

### 2.5 hdc（HarmonyOS Device Connector）

`hdc` 是 OpenHarmony 的设备连接工具，**功能等同于 Android 的 adb**。通过 USB 或网络连接设备后，可以：

- 向设备推送/拉取文件
- 在设备上执行 shell 命令
- 安装应用

```bash
hdc file send <本地文件> <设备路径>   # 推送文件（类似 adb push）
hdc shell <命令>                      # 在设备上执行命令（类似 adb shell）
```

> hdc 安装包下载：https://gitee.com/openharmony/developtools_hdc

### 2.6 HWTEST / HWTEST_F 宏

OpenHarmony 的测试框架基于 **Google Test（gtest）** 封装，提供 `HWTEST` 和 `HWTEST_F` 宏来编写测试用例。

```cpp
// HWTEST_F: 带测试套件类的测试（F = Fixture，可共享 setUp/tearDown）
HWTEST_F(TestSuiteName, TestCaseName, TestSize.Level1)
{
    EXPECT_TRUE(someCondition);   // 期望为真
    EXPECT_EQ(expected, actual);  // 期望相等
}
```

`TestSize.Level0` ~ `Level3` 是测试等级，Level0 最基础（最关键），Level3 是压力测试。

> gtest 入门：https://google.github.io/googletest/

### 2.7 xDevice 测试框架

xDevice 是 OpenHarmony 的**自动化测试执行框架**，可以：

- 自动连接设备
- 自动推送测试文件
- 批量执行测试
- 生成测试报告

> xDevice 仓库：https://gitee.com/openharmony/testfwk_xdevice

### 2.8 ohos_test.xml（测试资源配置文件）

位于 `test/unittest/resource/ohos_test.xml`，是测试的"准备和清理脚本"：

- **`<preparer>`**：测试运行**前**自动执行的 shell 命令（创建目录、创建测试文件等）
- **`<cleaner>`**：测试运行**后**自动执行的 shell 命令（删除测试文件、清理环境）

本项目的 `ohos_test.xml` 会在设备上创建如下目录和文件（测试用例依赖它们）：

```
/system/etc/custxmltest/system.xml      ← 用于测试 /system 层级查找
/system/etc/custxmltest/both.xml        ← 用于测试多层级同名文件
/sys_prod/etc/custxmltest/user.xml      ← 用于测试 /sys_prod 层级查找
/sys_prod/etc/custxmltest/both.xml      ← 用于测试多层级同名文件
/sys_prod/etc/carrier/46060/...         ← 用于测试 Follow X（运营商路径）
```

### 2.9 LibFuzzer（模糊测试）

模糊测试（Fuzz Testing）是一种**安全测试手段**：向函数输入大量随机数据，检测是否会崩溃、内存溢出等安全漏洞。OpenHarmony 使用 LLVM 的 **LibFuzzer** 框架。

> LibFuzzer 文档：https://llvm.org/docs/LibFuzzer.html

## 4. 环境要求

### 4.1 开发主机要求

| 项目 | 最低要求 | 推荐配置 |
|------|----------|----------|
| 操作系统 | Ubuntu 20.04 64位 | Ubuntu 22.04 LTS 64位 |
| CPU | 4核 | 8核以上 |
| 内存 | 16 GB | 32 GB |
| 磁盘 | 400 GB | 500 GB SSD |
| 网络 | 能访问 Gitee | 带宽越大越好 |

> **WSL2 用户注意**：需要在 WSL2 中给磁盘分配足够空间，默认 WSL2 磁盘大小可能不足。

### 4.2 目标设备（二选一）

**方案 A：真实开发板（推荐）**
- RK3568 开发板（如润和 HH-SCDAYU200）
- Hi3516DV300 开发板
- 需已烧录 OpenHarmony 系统镜像

**方案 B：QEMU 模拟器（无硬件时使用）**
- QEMU 版本 ≥ 6.0
- 仅支持 ARM/RISC-V 架构模拟
- 标准系统测试推荐使用 qemu_arm_linux_min 产品

## 5. 第一步：搭建编译环境

> 以下所有命令均在 **Ubuntu 22.04** 终端中执行。

### 5.1 安装基础工具包

```bash
sudo apt-get update
sudo apt-get install -y apt-utils binutils bison flex bc build-essential make \
    mtd-utils gcc-arm-linux-gnueabi u-boot-tools python3 python3-pip \
    git git-lfs gnupg zip unzip curl wget libssl-dev libelf-dev \
    libdwarf-dev lib32stdc++6 lib32z1 libncurses5 libncurses5-dev \
    zlib1g-dev m4 ccache liblz4-tool gettext gperf perl ruby \
    genext2fs libc6-dev-i386 lib32ncurses5-dev x11proto-core-dev \
    libx11-dev lib32z-dev libgl1-mesa-dev xsltproc xmlto
```

> **说明**：这一步安装的是 C/C++ 编译器、Python、Git 等基础工具，需要几分钟。

### 5.2 安装 Python 3.9+（如系统默认版本不够）

```bash
# 检查 Python 版本（需要 3.8+）
python3 --version

# 如果版本过低，安装 Python 3.10
sudo apt-get install -y python3.10 python3.10-pip python3.10-venv
sudo update-alternatives --install /usr/bin/python3 python3 /usr/bin/python3.10 1
```

### 5.3 安装 repo 工具

```bash
# 下载 repo 脚本
mkdir -p ~/bin
curl https://gitee.com/oschina/repo/raw/fork_flow/repo-py3 > ~/bin/repo
chmod a+x ~/bin/repo

# 将 ~/bin 加入 PATH（永久生效）
echo 'export PATH=~/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

# 验证安装
repo --version
```

> **repo 是什么**：类似"超级 git clone"工具，一次性下载数百个 git 仓库。

### 5.4 安装 hb 编译工具

hb 工具在下载完源码后安装，现在先跳过，等到下载源码后执行（见第6步）。

### 5.5 配置 Git 用户信息（必须）

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱@example.com"
git config --global credential.helper store
```

## 6. 第二步：获取 OpenHarmony 源代码

> ⚠️ **警告**：源码下载约需 50~100 GB 磁盘空间，下载时间视网速可能需要几小时到一天。请确保网络稳定，建议挂后台下载。

### 6.1 创建源码目录

```bash
mkdir -p ~/openharmony
cd ~/openharmony
```

### 6.2 初始化 repo（选择一个版本）

```bash
# 推荐：OpenHarmony 5.0 Release 版本
repo init -u https://gitee.com/openharmony/manifest \
          -b refs/tags/OpenHarmony-v5.0-Release \
          --no-repo-verify

# 或者：OpenHarmony 4.1 Release（更稳定）
repo init -u https://gitee.com/openharmony/manifest \
          -b refs/tags/OpenHarmony-4.1-Release \
          --no-repo-verify
```

### 6.3 同步源码

```bash
# 开始同步（-c 表示只同步当前分支，减少下载量）
repo sync -c
repo forall -c 'git lfs pull'
```

> **提示**：如果同步中断，重新执行 `repo sync -c` 即可续传。

### 6.4 安装 hb 工具（源码下载完成后执行）

```bash
cd ~/openharmony
pip3 install build/hb
```

验证安装：
```bash
hb --help
```

如果出现帮助信息，说明安装成功。

### 6.5 下载预编译工具链

```bash
cd ~/openharmony
bash build/prebuilts_download.sh
```

> 这一步下载编译器（Clang、GCC 等），约几 GB，需要一定时间。

## 7. 第三步：将本仓代码放入源码树

OpenHarmony 的 `config_policy` 组件**已经包含在源码树**中（位于 `base/customization/config_policy/`）。如果你对本仓做了修改，需要替换源码树中的对应目录。

### 7.1 确认源码中的位置

```bash
ls ~/openharmony/base/customization/config_policy/
```

你应该能看到和本仓相同的文件结构（BUILD.gn、bundle.json 等）。

### 7.2 如果需要替换为本地修改版（可选）

```bash
# 备份原版
cp -r ~/openharmony/base/customization/config_policy \
      ~/openharmony/base/customization/config_policy.bak

# 替换（假设你的本地仓库在 D:/projects/... 可通过 scp 或 WSL 共享路径传入）
# 在 WSL2 中，Windows 的 D: 盘挂载在 /mnt/d/
cp -r /mnt/d/projects/customization_config_policy/* \
      ~/openharmony/base/customization/config_policy/
```

## 8. 第四步：编译测试二进制文件

### 8.1 选择编译产品

先查看支持哪些产品（设备型号）：

```bash
cd ~/openharmony
hb set
```

会显示可选产品列表，常用的有：

| 产品名 | 适用设备 |
|--------|----------|
| `rk3568` | 润和 RK3568 开发板 |
| `hi3516dv300` | Hi3516 开发板 |
| `qemu_arm_linux_min` | QEMU ARM 模拟器（标准系统） |

选择你的目标产品，例如 `rk3568`。

### 8.2 编译单元测试

```bash
cd ~/openharmony

# 方式一：使用 build.sh（推荐，更通用）
# 格式：./build.sh --product-name <产品名> --build-target <测试目标名>
./build.sh --product-name rk3568 --build-target ConfigPolicyUtilsTest

# 方式二：使用 hb 工具
hb set -p rk3568
hb build -t ut ConfigPolicyUtilsTest
```

> **ConfigPolicyUtilsTest** 这个名字来自 `test/unittest/BUILD.gn` 中的 `ohos_unittest("ConfigPolicyUtilsTest")`。

### 8.3 等待编译完成

首次编译时间较长（0.5~3小时），后续增量编译会快很多。编译完成后显示 `BUILD SUCCESS`。

### 8.4 查找编译产物

```bash
# 单元测试二进制文件位置
find ~/openharmony/out -name "ConfigPolicyUtilsTest" -type f

# 典型路径（以 rk3568 为例）：
# ~/openharmony/out/rk3568/tests/unittest/config_policy/config_policy/ConfigPolicyUtilsTest

ls ~/openharmony/out/rk3568/tests/unittest/config_policy/config_policy/
```

你应该能看到 `ConfigPolicyUtilsTest` 可执行文件。

### 8.5 编译模糊测试（可选）

```bash
./build.sh --product-name rk3568 --build-target GetCfgFilesFuzzTest
./build.sh --product-name rk3568 --build-target GetCfgFilesExFuzzTest
./build.sh --product-name rk3568 --build-target GetOneCfgFileFuzzTest
./build.sh --product-name rk3568 --build-target GetOneCfgFileExFuzzTest
```

## 9. 第五步：连接设备（hdc）

### 9.1 安装 hdc 工具（在主机上）

**Ubuntu 上安装：**

```bash
# 下载 hdc 工具
# 访问 https://gitee.com/openharmony/developtools_hdc/releases 下载最新版
# 或者从编译产物中获取
ls ~/openharmony/out/rk3568/developtools/hdc/
```

编译产物中通常包含 `hdc` 或 `hdc_std` 可执行文件，将其复制到 PATH：

```bash
sudo cp ~/openharmony/out/rk3568/developtools/hdc/hdc /usr/local/bin/
sudo chmod +x /usr/local/bin/hdc
```

**Windows 上安装 hdc（如果使用 Windows 连接设备）：**

在 DevEco Studio 安装目录下的 `sdk/default/openharmony/toolchains/` 中可以找到 `hdc.exe`，将其目录加入系统 PATH。

### 9.2 通过 USB 连接设备

1. 用 USB 线将开发板连接到电脑
2. 确保开发板已启动并运行 OpenHarmony
3. 验证连接：

```bash
hdc list targets
```

如果显示设备序列号（如 `192.168.1.100:5555` 或 `XXXXXXXX`），说明连接成功。

### 9.3 通过网络连接设备（如设备与电脑在同一局域网）

```bash
# 连接
hdc tconn 192.168.x.x:5555

# 验证
hdc list targets
```

### 9.4 QEMU 模拟器用户

如果使用 QEMU，参考 OpenHarmony QEMU 文档启动模拟器：

```bash
# 启动 QEMU（以 arm linux 为例，具体命令看对应产品说明）
cd ~/openharmony/device/qemu/arm_virt/
./qemu-run

# QEMU 通常通过本地网络或 hdc over USB 暴力模式连接
```

> QEMU 详细文档：https://gitee.com/openharmony/device_qemu/blob/master/README_zh.md

## 10. 第六步：准备设备测试环境

这一步**非常关键**！单元测试用例依赖特定的目录结构和文件存在于设备上，需要手动创建（使用 xDevice 自动化框架时会自动执行，手动运行时需要手动完成）。

### 10.1 以读写方式重新挂载文件系统

```bash
# 以下命令通过 hdc 在设备上执行
hdc shell mount -o remount,rw /
hdc shell mount -o remount,rw /sys_prod
```

> **为什么需要这步**：OpenHarmony 设备的 `/system` 等分区默认是只读的，需要重新挂载为可读写才能创建测试文件。

### 10.2 创建测试所需目录和文件

```bash
# 创建目录
hdc shell mkdir -p /system/etc/custxmltest/
hdc shell mkdir -p /sys_prod/etc/custxmltest/
hdc shell mkdir -p /system/etc/carrier/46060/etc/custxmltest/
hdc shell mkdir -p /sys_prod/etc/carrier/46060/etc/custxmltest/
hdc shell mkdir -p /sys_prod/etc/carrier/46061/etc/custxmltest/
hdc shell mkdir -p /sys_prod/etc/carrier/46060/etc/custxmltest/custxmltest/

# 创建测试文件（空文件即可）
hdc shell touch /system/etc/custxmltest/system.xml
hdc shell touch /system/etc/custxmltest/both.xml
hdc shell touch /sys_prod/etc/custxmltest/both.xml
hdc shell touch /sys_prod/etc/custxmltest/user.xml
hdc shell touch /system/etc/carrier/46060/etc/custxmltest/both.xml
hdc shell touch /sys_prod/etc/carrier/46060/etc/custxmltest/user.xml
```

## 15. 常见问题与解决方案

### Q1：编译失败提示 "No module named 'xxx'"

**原因**：Python 依赖包缺失。

**解决**：
```bash
pip3 install -r ~/openharmony/build/scripts/requirements.txt
```

### Q2：hdc 提示找不到设备

**检查步骤**：
```bash
# 1. 确认设备已开机并进入系统（不是在 fastboot 模式）
# 2. 检查 USB 连接
hdc list targets

# 3. 如果是 USB，检查设备是否开启 USB 调试
# 在设备上：设置 → 关于本机 → 点击7次版本号开启开发者模式 → 开发者选项 → USB调试

# 4. 重启 hdc 服务
hdc kill
hdc start
```

### Q3：测试用例 002~004 失败（文件不存在）

**原因**：测试依赖的文件没有创建，或者文件系统是只读的。

**解决**：
```bash
# 重新挂载为读写
hdc shell mount -o remount,rw /
hdc shell mount -o remount,rw /sys_prod

# 重新执行第10步的所有创建命令
hdc shell mkdir -p /system/etc/custxmltest/
hdc shell touch /system/etc/custxmltest/system.xml
# ... 参考第10步完整执行
```

### Q4：测试用例 006~011 不执行

**原因**：这些测试被 `#ifndef __LITEOS__` 宏包裹，在 LiteOS 系统上会被跳过，这是**正常现象**。只有在标准 Linux 内核的 OpenHarmony 设备上才会执行。

### Q5：param set 命令失败（权限不足）

**原因**：系统参数有权限控制，非 root 用户不能修改 `const.*` 开头的参数。

**解决**：
```bash
# 尝试以 root 进入 shell
hdc shell
su root

# 或者使用 setenforce 临时关闭 SELinux（开发调试用）
setenforce 0
```

### Q6：磁盘空间不足（编译失败）

**解决**：
```bash
# 清理编译缓存
cd ~/openharmony
rm -rf out/rk3568/    # 只删除特定产品的编译产物

# 清理 ccache
ccache -C
```

### Q7：repo sync 速度太慢

**解决**：
```bash
# 使用国内镜像（如果 gitee 速度慢）
# 或增加并发数
repo sync -c -j8   # -j8 表示8个并发下载
```

### Q8：编译产物找不到 ConfigPolicyUtilsTest

**排查**：
```bash
# 查找所有 unittest 产物
find ~/openharmony/out -name "ConfigPolicyUtilsTest" 2>/dev/null

# 检查 BUILD.gn 是否正确被包含
grep -r "ConfigPolicyUtilsTest" ~/openharmony/out/rk3568/build_configs/ 2>/dev/null | head -5
```
